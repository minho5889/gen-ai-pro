## Section 1: Token Economics & Cost Fundamentals

Every optimization technique in this guide — prompt compression, prompt caching, model cascading, Provisioned Throughput, batch inference — is in service of one variable, and this section makes that variable explicit before any technique is taught. The variable is the token. When you run a foundation model on Amazon Bedrock, you are not renting a server, reserving a GPU, or paying for an instance that sits idle between requests. Bedrock is a fully managed, serverless service: the caller never provisions or sees the underlying compute, so there is no instance type to choose, no cluster to size, and no compute-hour meter ticking in the background on the default on-demand path. What you pay for instead is the volume of text the model reads and writes, counted in tokens — sub-word fragments that the model's tokenizer produces from your input and generates as output. On-demand inference is pay-per-use, billed on the number of input tokens sent plus the number of output tokens generated. This is the single most important reframing in Domain 4: the cost of a generative workload is a function of token volume, not of how big a machine you ran it on, because on the default path there is no machine you can point to.

This is why a "request" is the wrong unit to reason about cost, and why reasoning at the request level leads directly into the canonical exam trap. Two requests to the same model can differ by orders of magnitude in token volume — a one-line classification prompt and a hundred-page document summarization both count as "one request," yet one might consume a few dozen tokens and the other tens of thousands. Counting requests tells you nothing about spend. Counting compute hours tells you nothing either, because you never bought compute hours. The fundamental cost unit has to be the thing that actually varies with the work the model does, and that thing is the token. Internalize this and exam Pattern 4 ("Optimize cost / latency for an FM workload") becomes much harder to get wrong. The trap built into Pattern 4 is to reach for the reflex that traditional infrastructure trains: "the system is expensive or slow, so scale up the compute — pick a bigger instance, add nodes, raise the autoscaling ceiling." For a Bedrock FM workload on the default path there is no instance to scale, so that entire class of answer is a category error. The real levers act on something else entirely: they reduce the token volume (token efficiency, or switching to a smaller model), or they change the billing mode for the tokens you do consume (caching repeated tokens, committing to throughput, or batching). When a scenario describes an FM workload and offers "use a larger instance type" or "enable compute autoscaling" as a fix for cost or latency, recognize it as the Pattern 4 distractor and look instead for the answer that touches tokens or billing mode.

### Input tokens versus output tokens — and why the asymmetry matters

Tokens are not billed as a single undifferentiated pool. The Amazon Bedrock pricing page lists input tokens and output tokens as separate line items for every text-generation model, with their own per-token rates. The structurally important fact — true across every model family, and the part you should carry into the exam — is that the rate is asymmetric: output tokens are priced higher than input tokens. (The exact per-token figures vary by model, provider, modality, and Region and change frequently, so they must be looked up live; see the pricing-model note below. What is durable is the *direction* of the asymmetry, not any specific number.) There is a sound technical reason for the asymmetry. Input tokens are processed in a single parallel forward pass — the model ingests the whole prompt at once — whereas output tokens are produced autoregressively, one token at a time, each requiring its own forward pass that attends to everything generated so far. Generating the 500th output token is more expensive than ingesting the 500th input token, and the price schedule reflects that compute reality.

The asymmetry is not a billing curiosity; it is what makes output-length control a distinct cost lever, separate from input-side techniques. Because each generated token costs more than each ingested token, the cheapest place to shave cost is often the output, and the output is also the most directly controllable dimension: you can bound it. Instructing the model to be concise and setting a maximum-tokens limit on the response puts a hard ceiling on the most expensive part of the bill, which is exactly why Section 2 treats output-length control as its own technique rather than folding it into general prompt trimming. A related but separate mechanism reinforces the point: for some newer Anthropic Claude models, Bedrock applies a higher quota "burndown rate" to output tokens — output tokens count more heavily against your tokens-per-minute service quota than input tokens do. That burndown multiplier is a *quota-accounting* mechanism, not a billing rate (it governs how fast you approach a throttling limit, not what you are charged), but it pushes in the same direction as the price asymmetry: output tokens are the scarcer, costlier resource, so controlling output length pays off twice — once on the bill and once on your throughput headroom. (Which models carry the higher output burndown changes as the model lineup evolves, so treat the specific model list as requiring live verification.)

### What drives token consumption per call

If tokens are the unit of cost, the next question is what makes a single call expensive — and the answer sets up every technique in Sections 2 and 3. Bedrock keeps no server-side memory of a conversation. The model is stateless between invocations, which means the *entire* context the model needs to see has to be re-sent on every single call and is re-billed as input tokens each time. That context is the sum of four things: the system prompt (the standing instructions, persona, and rules), any retrieved context injected by a RAG pipeline (the passages pulled from a knowledge base), the accumulated conversation history (every prior user turn and model reply you choose to carry forward), and the new user message. Each of these is an input-token cost you pay again on every invocation unless a caching mechanism intervenes.

Two consequences follow directly, and each names a target for a later section. First, a long, static system prompt is not paid for once — it is paid for on every call for the life of the workload. A 1,500-token system prompt re-sent on a million calls is 1.5 billion input tokens, none of which changed between calls. That is precisely the waste Amazon Bedrock prompt caching is built to eliminate (Section 3): cache the repeated prefix so it is neither reprocessed nor re-billed at full rate on subsequent calls. Second, multi-turn conversation history grows without bound if you append every turn verbatim — turn 20 re-sends the entire transcript of turns 1 through 19, so per-call token cost climbs steadily as the conversation lengthens. That unbounded growth is the target of history summarization and truncation (Section 2), which cap per-call token growth by compressing or dropping old turns. A useful operational habit that supports all of this: the Bedrock CountTokens API estimates the input-token count of a request *before* you invoke the model, at no charge, so you can measure context size and catch runaway prompts during development rather than discovering them on the bill.

### The Cost Lever Priority

When a workload costs too much, you do not pull all the levers at once — you pull them in order, cheapest win first, escalating to higher-commitment levers only when the cheaper ones are exhausted. This ordering is the mental model that organizes the entire guide and supplies the canonical answer shape for Pattern 4. There are four levers, in this order.

1. **Token efficiency (lever 1).** Reduce the number of tokens each call consumes — trim and compress prompts, prune stale or irrelevant context, summarize conversation history, and bound output length. It is first because it costs nothing to adopt: no infrastructure commitment, no model change, no new service. It lowers cost on *every* invocation immediately, and it compounds with every other lever (fewer tokens make caching, routing, and committed throughput all cheaper too). This is developed in Section 2.
2. **Caching (lever 2).** Avoid re-paying for content that repeats across calls — cache a repeated prompt prefix so it is not reprocessed and re-billed (Bedrock prompt caching), or return a stored answer for a semantically equivalent query so the model is not invoked at all (semantic caching). It sits second because it is still low-commitment and infrastructure-light, but unlike lever 1 it only helps to the extent that content actually repeats — so it is the next cheapest win once prompts are already lean. This is developed in Section 3.
3. **Model tiering / cascading (lever 3).** Change the unit cost of a request by routing it to a cheaper or smaller model — send easy requests to a small model and escalate only the hard ones to a large model (tiering/cascading), or let Amazon Bedrock Intelligent Prompt Routing pick the cost-appropriate model within a family. It is third because it requires real engineering and validation work — you must prove the cheaper model still clears the task's quality bar — and it changes behavior rather than just volume. This is developed in Section 4.
4. **Provisioned Throughput / batching (lever 4).** Change the billing mode itself — commit to dedicated capacity in Model Units for steady high volume (Provisioned Throughput), or accept asynchronous turnaround in exchange for a documented discount (batch inference). It is last because it is the highest-commitment lever: Provisioned Throughput is an hourly commitment that bills whether or not you use it, and batch trades away real-time responsiveness. You reach for it only when volume is high and predictable enough to justify the commitment. This is developed in Section 5.

The ordering is not arbitrary preference; it is risk and commitment ascending. Levers 1 and 2 are reversible, infrastructure-free, and apply on every call. Lever 3 alters model behavior and demands quality validation. Lever 4 locks in spend or latency characteristics. Pull them in this sequence and you exhaust the free wins before taking on commitment — which is exactly the reasoning an exam answer for Pattern 4 should display.

This whole problem sits in Domain 4 (Operational Efficiency and Optimization — 12% of the exam), Task 4.1 (cost optimization and resource efficiency). It is also where this guide takes ownership of depth the neighbors deferred. Guide 01 established model selection and the inference options (Provisioned Throughput, Cross-Region Inference, batch, Intelligent Prompt Routing) and the cost/latency tradeoffs among models; this guide builds the cost reasoning on top of that foundation rather than re-teaching the mechanics. Guide 05 owns prompt caching and token-efficiency techniques at the prompt-authoring level; here they are reframed as cost levers. Most importantly, Guide 06 (Testing, Evaluation & Troubleshooting) repeatedly deferred the standing cost-and-monitoring program to "future Guide 07" — this is that Guide 07, and it owns the continuous token-economics and cost-instrumentation depth (developed fully in Section 7) that Guide 06 pointed toward but did not unpack.

### The Amazon Bedrock pricing model (no dollar figures)

You can reason about Bedrock cost correctly without ever memorizing a price, and on this exam you should — because the prices change and the exam will not ask you to recite them. What is stable is the *shape* of the pricing model, and that is what to learn. Pricing on Bedrock varies along several axes: by modality (text, image, embeddings), by provider and model (each model family sets its own rates), and by Region. On top of that, Bedrock offers different billing modes and service tiers for the same model:

- **On-demand** is the default: serverless, pay-per-use, billed on input tokens plus output tokens, with no commitment. It is bounded by per-model tokens-per-minute and requests-per-minute service quotas, and exceeding them produces throttling — which is one of the pressures that motivates moving to a committed option at high volume.
- **Provisioned Throughput** buys dedicated capacity expressed in Model Units (each Model Unit guarantees a fixed input/output token-per-minute throughput), billed hourly, with no-commitment, one-month, or six-month commitment terms (longer terms carry larger discounts). It is required for serving custom (fine-tuned) models, and billing continues until the provisioned model is deleted.
- **Batch inference** processes a large set of requests asynchronously through Amazon S3 and is documented at a lower price than on-demand for the same model, in exchange for non-real-time turnaround.
- Bedrock also exposes service tiers (such as Standard, Flex, and Priority) and other options that trade latency, throughput, and price against one another for eligible models.

Beyond the model shapes, there is one hard rule for this document: it states no specific dollar amounts. Any exact per-token rate, per-Model-Unit hourly rate, batch discount percentage, or service-tier price is fast-moving and must be verified live against the Amazon Bedrock pricing page at the time you need it — never recited from memory or from a study guide. The exam knows this too, which is why the *pricing trap* (treating a memorized dollar figure as a current fact) is a distractor in its own right: a question that hinges on a precise price is testing whether you know prices must be looked up, not whether you memorized one.

**Pricing-model verification note.** The following specifics require live verification against current AWS documentation and are not asserted here as fixed facts: every exact per-token, per-model, and per-Model-Unit dollar rate (no dollar figures appear in this guide by policy); the exact batch-inference discount (documented at 50% lower than on-demand at the time of research — verify the current rate and which models are eligible); the exact tokens-per-minute / tokens-per-day quotas and the approximate ten-concurrent-batch-jobs-per-model-per-Region limit; which models and Regions support each service tier (Standard, Flex, Priority) and each commitment term, and their GA-versus-preview status; and the set of models (currently newer Anthropic Claude models) that carry the higher output-token quota burndown rate. The pricing *model* (input/output token billing, per-model tiers, on-demand versus committed-throughput versus batch, service tiers) is stable; the *numbers* are not.

### Knowledge Check

**Q1:** A team runs a summarization feature on Amazon Bedrock (on-demand). Cost has tripled month over month. An engineer proposes "moving to a larger, more powerful instance type with autoscaling" to handle the load more efficiently. What is wrong with this, and what should they do instead?
- A) Nothing wrong — a larger instance with autoscaling is the standard cost fix
- B) On-demand Bedrock is serverless with no instance to scale; reduce token volume (concise output + max-tokens, prune context) and consider caching repeated content
- C) Switch to a GPU instance reservation to lock in a lower hourly rate
- D) The cost is fixed per request, so the only fix is to send fewer requests

**A: B** — This is the canonical Pattern 4 trap. On the default on-demand path Bedrock is fully managed and serverless: the caller never provisions instances, so there is no instance type to enlarge and no compute-hour meter to optimize. Cost is driven by token volume (input + output tokens). The correct first moves are token-efficiency levers — bound output length, trim and prune context — followed by caching repeated content. Options A and C invent compute that does not exist on this path; option D is wrong because cost is *not* fixed per request (two requests can differ by orders of magnitude in tokens).

**Q2:** True or False — On Amazon Bedrock, input tokens and output tokens are billed at the same per-token rate, so it does not matter whether you reduce the prompt or the response.

**A: False.** The Bedrock pricing page lists input and output tokens as separate line items, and across every model family output tokens are priced higher than input tokens. Output is generated autoregressively (one forward pass per token), so it costs more compute per token than the single parallel pass that ingests input. This asymmetry is exactly why output-length control (concise instructions plus a max-tokens limit) is a distinct cost lever — the response is the costlier dimension to leave unbounded. (The exact per-token rates are fast-moving and must be verified live.)

**Q3:** Fill in the order — The Cost Lever Priority, from cheapest win to highest commitment, is: (1) ___, (2) ___, (3) ___, (4) ___.

**A:** (1) token efficiency, (2) caching, (3) model tiering/cascading, (4) Provisioned Throughput/batching. The order ascends in commitment and risk: token efficiency is free, reversible, and applies on every call; caching is low-commitment but only helps when content repeats; model tiering/cascading requires routing logic and quality validation; Provisioned Throughput/batching is the highest-commitment lever (an hourly capacity commitment or a tolerance for asynchronous turnaround) and is evaluated last, only when volume is high and predictable.

**Q4:** A multi-turn chatbot on Bedrock sees per-call cost climb steadily as conversations get longer, even though each user message is short. Why does this happen, and which two later levers address it?

**A:** Bedrock is stateless — it keeps no server-side memory of the conversation, so the entire context (system prompt + any retrieved context + the full accumulated history of prior turns + the new message) is re-sent and re-billed as input tokens on every call. As the transcript grows, each new turn re-sends all prior turns, so input-token cost per call rises with conversation length. Two levers address it: token efficiency (Section 2) via conversation-history summarization or truncation to bound per-call token growth, and caching (Section 3) via prompt caching of the stable repeated prefix (such as a long system prompt) so it is not re-billed at full rate every call.

**Q5:** An exam item asks which is the correct unit of cost for a foundation model invocation on Amazon Bedrock's default path. Which choice is correct, and why are the others wrong?
- A) Number of requests
- B) Compute hours consumed
- C) Input tokens plus output tokens
- D) Instance size / instance type

**A: C** — On-demand Bedrock is serverless pay-per-use billed on input tokens plus output tokens generated. A is wrong because a "request" is not a stable unit — two requests can differ by orders of magnitude in token volume. B and D are wrong because the caller never provisions compute on the default path: there are no compute hours purchased and no instance to size, so neither can be the billing dimension. The token is the only quantity that scales with the actual work the model performs.

## Section 2: Token Efficiency & Context Optimization

Section 1 established that the token is the unit of cost and ordered the four levers you pull to control it. This section develops the first lever — token efficiency — and explains why it is always where you start. The reasoning is almost entirely about commitment and reach. Token efficiency is a prompt-engineering and configuration change: you rewrite a prompt, prune some context, cap a response length. There is no infrastructure to stand up, no capacity to commit to, no model to swap, and nothing to undo if you change your mind. Because it acts on the request itself rather than on how the request is served, it lowers cost on *every single invocation* the moment it ships, and it does so for free. The AWS Well-Architected Generative AI Lens makes this the operational heart of cost-aware prompting (best practice GENCOST03): foundation models charge by the token, so reducing the number of tokens a prompt sends directly reduces the processing cost, and "controlling resource consumption parameters" is treated as a distinct operational-level lever, separate from the heavier decision of which model to run. Token efficiency also compounds with every later lever — fewer tokens make a cached prefix cheaper, make a routed request to a smaller model cheaper still, and make committed throughput stretch further — which is the other reason it comes first: it multiplies the value of everything downstream.

There is a hard constraint that bounds this lever, and it is the single most important thing to carry out of this section. Token reduction is only legitimate while it preserves output quality at or above the use case's defined quality bar — the success criteria you set for the task. AWS states the same boundary from both ends: keep prompts "as short as possible while still meeting your performance requirements," and keep responses short "without sacrificing usability," only "so long as it satisfies the use case." The implication is that token efficiency is not "make the prompt as small as possible." It is "make the prompt as small as possible *that still clears the bar*," and it must never be applied past the point where quality falls below that bar. Every technique below has a corresponding failure mode that is exactly this: a token saving taken one step too far, where the bytes you removed turn out to have been load-bearing. The discipline is to reduce tokens *and* measure quality against the bar, treating a drop below the bar as the signal to stop, not as an acceptable cost of saving money.

A concrete scenario frames the rest of the section. Suppose you operate a customer-support assistant on Amazon Bedrock backed by a knowledge base. Each call carries a long system prompt (persona, tone rules, escalation policy, a few worked examples), several retrieved passages from the knowledge base, the running conversation history, and the customer's new message. Cost per call has crept up as the system prompt grew and conversations lengthened. Token efficiency attacks every one of those four inputs — and the rest of this section walks through how, while keeping the answer quality the support team actually requires.

### Prompt compression and context pruning

These two techniques are often spoken of together, but they remove different things and fail in different ways, so the exam rewards keeping them distinct. A terminology flag first: the names "prompt compression" and "context pruning" are this guide's labels for the underlying ideas. AWS does not market a feature called "prompt compression" — its guidance is to "optimize prompt token length" by "trimming as many unnecessary words as possible" — and AWS does not name "context pruning" at all; that term is the guide's synthesis of several AWS recommendations about how much context to send. Use the ideas, but do not expect to see these exact phrases in AWS documentation.

Prompt compression reduces the token count of a prompt while preserving its instructional content. What you remove is wording, not meaning: redundant phrasing, filler, verbose restatements, decorative boilerplate, and over-long examples — the tokens that cost money on every call but do not change what the model is being asked to do. AWS frames this as trimming unnecessary words, and notes you can optionally use a separate LLM pass to shorten a prompt "without reduction in performance," and that Amazon Bedrock Prompt Optimization can rewrite a prompt automatically. The failure mode of over-applying it is loss of instructional fidelity: compress past the point where the words were merely verbose and you start deleting the actual instructions, constraints, and examples the model was relying on, which changes the model's behavior — it stops following a rule it can no longer see, drops a format it was being shown, or loses the persona that kept its tone on-brand. (That over-compression failure mode is the guide's inference, grounded in AWS's own "without reduction in performance" guardrail: the guardrail exists precisely because shortening *can* reduce performance if taken too far.) In the support-assistant scenario, compressing the system prompt from three verbose worked examples to one tight example is a safe token win; deleting the escalation policy because it was "long" is the failure mode — the bot stops escalating angry customers correctly.

Context pruning removes or summarizes low-value tokens before the call — stale conversation turns that no longer bear on the current question, and retrieved passages that are not actually relevant to the query. Where compression rewrites the *instructions*, pruning trims the *context* wrapped around them. AWS's guidance points squarely at this even without naming it: there are four input-token sources (system prompt, user prompt, retrieved context from the vector store, and the question-and-answer conversation history), and you are told to "test accuracy with various numbers of chunks and chunk sizes," to limit how many QnA history pairs you retain (test 2, 3, 4, and 5 pairs and pick the optimal), and to use reranking (the Amazon Bedrock Rerank API) to cut retrieved results down to "fewer but more pertinent results that can reduce cost and latency." The unifying instruction is to "strike the right balance" — send enough context to answer well, but not more. The failure mode of over-pruning is the mirror image of the balance: cut too aggressively and you remove context the model genuinely needed, degrading the answer — the assistant gives a worse response because the passage that held the answer was pruned as "low value," or it loses the thread of a conversation because the turn that established what "it" refers to was dropped. (This degradation failure mode is again the guide's synthesis, grounded directly in AWS's "strike the right balance" wording — the balance has two sides, and over-pruning falls off the far one.)

The mechanics of authoring and automatically optimizing prompts — how to write them well in the first place, and the Amazon Bedrock Prompt Optimization and prompt-management tooling that rewrites and versions them — belong to Guide 05 (Prompt Engineering & Management), which is the prompt-authoring home. This section treats those techniques only as cost levers and points there for the authoring depth rather than restating it.

### Context window optimization — the cost of everything you re-send

Section 1's stateless-re-billing rule is the reason this subsection exists: because Amazon Bedrock keeps no server-side memory, the full context is re-sent and re-billed as input tokens on every call, so the *size* of that context is a recurring cost, not a one-time one. Context window optimization is the practice of keeping that recurring payload as small as the quality bar allows. It helps to know where the tokens actually sit, because the components are very unequal in size. Using the rough four-characters-per-token rule of thumb from the AWS cost-optimization guidance, the new user message is usually tiny (a one-line question is on the order of twenty tokens). The system prompt can be large, especially when it carries few-shot examples — three examples at roughly a hundred tokens each is already three hundred tokens sent on every call. Retrieved context tends to be the largest single component: three chunks of about two thousand characters each is roughly fifteen hundred tokens per call. And conversation history grows with every turn you retain, because each retained turn is re-sent in full on all subsequent calls. The practical takeaway is that the biggest recurring savings usually live in retrieved context and in conversation history, not in the user's message — so that is where pruning effort pays off most.

Conversation history deserves its own treatment because, unlike a fixed system prompt, it grows without bound if left alone. There are two ways to bound it, and the exam wants you to distinguish their tradeoffs cleanly. Truncating keeps a fixed window of the most recent turns and discards the oldest ones — when a new turn would push the request past the window, the oldest messages are trimmed until it fits again. It is cheap (it does no extra work, it just drops messages) and it bounds per-call tokens reliably, but it loses older turns outright, and a naive truncation can sever a tool-use request from its matching response and break the call. Summarizing instead condenses the older turns into a running summary while keeping the most recent turns verbatim — it preserves the important information from the dropped region rather than discarding it, so it retains coherence better than a hard cut. Its cost is that summarizing requires an extra model call to produce the summary (you can point that summarization at a cheaper, faster model to limit the added cost), and any summary can lose detail that a verbatim transcript would have kept. The Strands Agents SDK packages both as conversation managers — a sliding-window manager that truncates to a fixed number of recent messages, and a summarizing manager that summarizes older messages with tunable knobs for how much to summarize and how many recent messages to preserve verbatim — and the same idea appears AWS-native as the "size of trailing history" limit in the generative-AI application patterns, which is the truncation approach. The drivers for choosing how to manage history are token limits, performance, relevance, and coherence: truncation optimizes for the first two cheaply; summarization spends a little to protect the last two.

| Approach | What happens to older turns | Added processing | Main risk |
|---|---|---|---|
| Truncating | Oldest turns dropped until the request fits a fixed recent-message window | None — it only discards messages | Loses older turns outright; a naive cut can break a tool-use request/response pair |
| Summarizing | Older turns condensed into a running summary; recent turns kept verbatim | An extra model call to build the summary (can use a cheaper/faster model) | Possible loss of detail in the summary; more compute and latency per trim |

### Output-length control and the output-token asymmetry

The fourth target of token efficiency is the response itself, and it connects directly back to the pricing asymmetry established in Section 1: output tokens are priced higher than input tokens across every model family, so the generated response is the costliest dimension per token and the one with the most leverage per token saved. Output-length control is the lever, and it has two complementary halves. The first is instructional: tell the model to be concise — ask for a short answer, a bounded list, a one-word verdict where a one-word verdict will do. The Well-Architected Lens (GENCOST03-BP02, "Control model response length") goes further and suggests introducing determinism where the full natural-language text is not needed — instructing the model to answer "True" or "False," or to return a single label, instead of a paragraph. The second half is a hard ceiling: set the max-tokens inference parameter, which caps the maximum number of tokens the model will generate and therefore puts an absolute upper bound on the most expensive part of the bill regardless of how verbose the model would otherwise have been. Instruction shapes the typical response; max-tokens bounds the worst case. Together they make output-length control a distinct cost lever rather than a sub-case of prompt trimming, which is exactly why it is called out separately.

The asymmetry that justifies this is durable, but its magnitude is not. The AWS cost-optimization guidance has described output tokens as priced several times higher than input tokens (on the order of three to five times at the time of that writing), but the exact multiple varies by model and changes over time, so treat the *direction* of the asymmetry — output costs more than input — as the fixed fact and the specific multiple as something to verify live rather than memorize. There is also a secondary, throughput-side reason to bound output that ties back to Section 1's burndown note: max-tokens participates in quota accounting. At the start of a request Bedrock deducts the input tokens plus the full max-tokens value from your quota up front, then reconciles to actual usage afterward (replenishing the unused remainder), so an unnecessarily large max-tokens reserves quota you will not use and can constrain concurrency and throughput even when it does not change the final bill. On some newer Anthropic Claude models the output side also carries a higher quota burndown multiplier, which compounds the effect. (The exact output-to-input price multiple and the specific set of models that carry the higher output burndown are both fast-moving and must be verified against current AWS documentation rather than asserted from memory; no dollar figures are stated here by this guide's policy.) Bounding output length therefore pays off twice — once on cost, because output tokens are the priciest, and once on throughput headroom, because an oversized max-tokens reserves quota up front.

### Exam-Relevant Distinctions

| If the exam says... | The answer is... | Why |
|---|---|---|
| "Reduce FM cost without changing the model or infrastructure" | Token efficiency (lever 1) — compress prompt, prune context, bound output | It is free, reversible, applies on every call, and needs no commitment or model swap |
| "Shorten the prompt but quality dropped / model stopped following a rule" | Over-compression — instructional fidelity was lost | Compression must remove wording, not the instructions/constraints/examples the model relied on |
| "Pruned retrieved context and answers got worse" | Over-pruning — context the model needed was removed | Strike the right balance: enough context to answer, not more — cutting past it degrades the answer |
| "Multi-turn cost grows; must keep continuity of older turns" | Summarize history (not hard truncate) | Summarizing preserves important detail and coherence; truncation drops older turns outright |
| "Multi-turn cost grows; cheapest possible bound, detail not critical" | Truncate history to a recent-message window | Truncation adds no processing; it just discards oldest turns to fit the window |
| "Responses are long and verbose; cut cost" | Concise instruction + max-tokens limit | Output tokens are priced higher than input, so bounding output is the highest-leverage token saving |

- The terms "prompt compression" and "context pruning" are this guide's labels; AWS says "optimize prompt token length" and does not name context pruning — know the ideas, not the exact phrases.
- Token efficiency has a hard ceiling: reduce tokens only while output quality stays at or above the use case's quality bar; a drop below the bar is the signal to stop, not an acceptable trade.
- max-tokens is both a cost cap (bounds the priciest output dimension) and a quota mechanism (input + max-tokens is reserved up front, then reconciled to actual usage).
- The output-to-input price asymmetry is durable (output costs more); the exact multiple is fast-moving and must be verified live.
- Retrieved context and conversation history are usually the largest recurring token components — prune effort there, not on the short user message.

- B) It needs no infrastructure or model change, applies on every call, and compounds with later levers — bounded only by the requirement that output quality stay at or above the use case's quality bar

**Q2:** True or False — Prompt compression and context pruning fail in the same way, so the same caution applies to both.

**A: False.** They remove different things and fail differently. Prompt compression removes wording while preserving instructional content; over-applying it causes loss of instructional fidelity — you delete an instruction, constraint, or example the model relied on, and its behavior changes. Context pruning removes low-value context (stale conversation turns, irrelevant retrieved passages); over-applying it removes context the model actually needed, degrading the answer. The shared discipline is "strike the right balance," but the specific risk differs: compression risks changing what the model is told to do, pruning risks starving it of what it needs to know.

**Q3:** Scenario — A multi-turn legal-research assistant on Bedrock has rising per-call cost as sessions lengthen. Continuity matters: later questions routinely refer back to facts established many turns earlier. The team wants to bound token growth without losing that earlier context. Truncate or summarize, and what is the tradeoff?

**A: Summarize.** Because later turns depend on facts established early in the session, hard truncation (dropping the oldest turns to fit a recent-message window) would discard exactly the context the assistant needs, breaking continuity. A summarizing approach condenses the older turns into a running summary while keeping recent turns verbatim, preserving the important earlier facts and the conversation's coherence. The tradeoff is that summarizing costs an extra model call to build the summary (which can be pointed at a cheaper/faster model to limit cost) and any summary can lose some detail — but that is the right price to pay when continuity is the requirement. Truncation would be the cheaper choice only if losing older turns were acceptable.

**Q4:** What Went Wrong — A team capped cost by setting a very small max-tokens on a summarization endpoint. Cost per call did drop, but users complain that summaries are now cut off mid-sentence and miss key points. What happened, and how should output-length control be applied correctly?

**A:** max-tokens is a hard ceiling on generated output tokens — it does not tell the model to *finish* within that budget, it simply stops generation when the budget is hit. Setting it below what the task genuinely needs truncates the response and pushes output quality below the use case's quality bar, which violates the token-efficiency constraint (reduce tokens only while quality holds). The correct application of output-length control pairs the two halves: instruct the model to be concise (and use determinism — a short structured answer — where the full prose is not needed) so the *typical* response is short by design, and set max-tokens as a worst-case safety ceiling sized to the longest acceptable answer, not below the task's real need. The goal is to bound the costly output dimension without cutting answers off below the bar.

**Q5:** An engineer says, "Output and input tokens cost the same, so it doesn't matter whether I trim the prompt or the response — I'll just trim whichever is easier." Why is the cost reasoning wrong, and what should guide the choice?

**A:** Output and input tokens do *not* cost the same — output tokens are priced higher than input tokens across every model family (Section 1's asymmetry), because output is generated autoregressively one forward pass per token. So a token saved on the output side is worth more than a token saved on the input side, which makes output-length control the highest-leverage-per-token saving and a distinct lever. The choice should be guided by where the tokens are and what they cost: bound verbose output first (concise instruction + max-tokens) because it is the priciest dimension, then attack the largest recurring input components (usually retrieved context and conversation history). The exact output-to-input multiple is fast-moving and must be verified live, but the direction — output costs more — is durable.

## Section 3: Caching Strategies

Section 2 attacked the size of the request itself — fewer tokens per call. Caching, the second lever in the Cost Lever Priority, attacks a different waste: paying repeatedly to process and bill content the system has already seen. The two levers are complementary, not competing. Token efficiency makes each call leaner; caching avoids re-doing work across calls. Crucially, caching reaches its savings without touching the two things the later levers touch — it does not change the model (that is lever 3) and it does not rewrite or prune the prompt (that is lever 1). It leaves the request and the model exactly as they are and instead intervenes in the billing-and-compute path: content that was already paid for once is not reprocessed and re-billed at full rate the next time it appears. That is why it sits second — it is still low-commitment and infrastructure-light, a configuration-level or thin-service-level change rather than a model swap or a capacity commitment, but it ranks just below token efficiency because it carries one precondition that lever 1 does not.

That precondition is the whole story of when caching helps: caching only pays off when content actually repeats across calls. A cache returns value on a hit, and a hit requires that something seen before shows up again. If every prompt is unique and non-repeating — a stream of one-off, never-similar requests — there is no repeated prefix to reuse and no prior equivalent query to match against, so both caching mechanisms in this section yield nothing while still adding machinery. Before reaching for caching, then, the diagnostic question is not "is this workload expensive?" but "does this workload re-send the same content?" Long static system prompts re-sent on every call, the same reference document analyzed across many requests, a tool-definition block shared by every agent invocation, an FAQ answered the same way thousands of times a day — these repeat, and they are where caching earns its place. A workload with genuinely unique inputs on every call does not benefit, and recognizing that keeps you from adding a cache that can never hit. AWS endorses caching as a first-class cost practice in the Well-Architected Generative AI Lens (best practice GENCOST03-BP03, "Implement prompt caching to reduce token costs"), which places it squarely in the same operational-cost discipline as the token-efficiency levers of Section 2.

A concrete scenario carries the section. Return to the customer-support assistant from Section 2: a long system prompt (persona, tone rules, escalation policy, worked examples) and, increasingly, a product-manual PDF attached to many tickets. Two repetition patterns live here. The system prompt and the manual are the *same prefix* re-sent on call after call — a prompt-caching opportunity. And a large fraction of customers ask the *same questions* in slightly different words ("how do I reset my password?" / "I forgot my password, what now?") — a semantic-caching opportunity. The two caching strategies target these two different kinds of repetition, and the rest of the section is about telling them apart.

### Amazon Bedrock prompt caching

Amazon Bedrock prompt caching is an optional feature that lets you mark a contiguous, static prefix of a prompt — a "cache checkpoint" — so that on subsequent requests carrying the same prefix the model reads those input tokens from cache instead of recomputing them from scratch. The prefix is exactly the kind of repeated content Section 1 warned about: a long system prompt, a fixed instruction block, a set of tool definitions, or an uploaded document that does not change between requests. The defining requirement is that the cached portion must be static — the cache matches on the prefix, so any change to it (even a small edit near the front) produces a cache miss and the tokens are processed in full again. This is why you place the checkpoint after the stable content and keep the volatile, per-request content (the user's actual question) after it.

The payoff comes as two distinct benefits, and the exam rewards keeping them separate because they answer different questions. The first is a cost benefit: on a cache hit, the tokens read from the cached prefix are billed at a reduced rate rather than the full input-token rate, so the repeated prefix stops costing full price on every call. The second is a latency benefit: on a cache hit, the cached prefix is not reprocessed — the model skips the forward-pass computation over those tokens — so time-to-response drops. These are independent. The cost benefit is about what you are charged; the latency benefit is about how long the call takes. A workload can value either or both: a high-volume FAQ bot cares most about the cost benefit, while a latency-sensitive interactive assistant with a huge system prompt may care most about the latency benefit. AWS publishes benchmark figures of up to roughly 85% latency reduction and up to roughly 90% cost reduction for prompt caching, but these are best-case AWS benchmark claims, not guarantees — actual savings depend on how much of your prompt is cacheable prefix and how often it is reused, and the figures should be treated as marketing-grade upper bounds rather than expected results.

There is an important pricing nuance that prevents a common misconception. The savings come from cache *hits* on a reused prefix — not from the first call. Writing tokens into the cache can be charged at a rate *higher* than standard uncached input (this is model-dependent), so the initial request that populates the cache may cost slightly more, and the net benefit only accrues once the prefix is reused enough times for the cheap reads to outweigh the one-time write. Prompt caching therefore helps high-reuse prefixes and can be net-negative for a prefix used only once — which loops back to the section's precondition: repetition is what makes it pay.

Several specifics of prompt caching are fast-moving and must be verified against current AWS documentation rather than asserted from memory:

- **Supported models** (verify live): support has centered on the Anthropic Claude family and Amazon Nova (Nova offers automatic prompt caching for text), with most options generally available and some entries (such as Claude 3.5 Sonnet v2) listed in preview at various points. The feature launched in preview in December 2024 alongside Intelligent Prompt Routing and has since reached GA for Claude and Nova — but the exact supported-model list and each model's GA-versus-preview status change as the lineup evolves.
- **Minimum-token threshold** (verify live, per-model): a prefix must exceed a per-model minimum to be cacheable — on the order of ~1,024 tokens for some Claude models and ~4,096 for others, with a small maximum number of cache checkpoints per request (around four for Claude). Below the minimum, the request still succeeds but nothing is cached.
- **Cache lifetime / TTL** (verify live): the cache is short-lived and its TTL resets on each successful hit; most models use roughly a 5-minute TTL, while some Claude models support an extended (~1-hour) TTL on an opt-in basis. The cache is account-specific.
- **Inference-path constraint** (verify live): prompt caching is supported on on-demand inference and works with the Converse / ConverseStream and InvokeModel APIs, Prompt Management, and cross-region inference — but it is not supported with the batch inference API.

The *authoring* mechanics — how to place a cache checkpoint in a Converse or InvokeModel request, where the `cachePoint` block goes, and how checkpoint ordering interacts with the prompt structure — belong to Guide 05 (Prompt Engineering & Management), which is the prompt-authoring home. This guide treats prompt caching only as a cost-and-latency lever (what it caches, why, and the two benefits) and points to Guide 05 for the how-to rather than restating it.

### Semantic caching

Semantic caching solves the *other* repetition pattern — the same question asked in different words — and it works on a fundamentally different principle. Instead of matching a static prefix byte-for-byte, it matches by meaning. The mechanism is a vector cache: each time the system answers a query, it stores the vector embedding of that query paired with the response. When a new query arrives, the system embeds it and compares the new vector against the cached query vectors; if the similarity to some cached query exceeds a configured threshold, it returns that query's stored response directly; otherwise it falls through, invokes the model, and caches the new query-response pair for next time. Because the match is on embedding similarity rather than exact string equality, "how do I reset my password?" and "I forgot my password, what now?" can hit the same cached answer even though they share few exact words.

This is the distinction that separates semantic caching from prompt caching, and it is the single most testable point in the section: semantic caching avoids the model invocation entirely on a hit, while prompt caching reduces the cost of an invocation that still occurs. On a semantic-cache hit, no LLM call happens at all — the stored response is returned and the model is never invoked, so you save the *entire* cost and latency of generation, not just the input-prefix portion. On a prompt-cache hit, the model is still invoked — generation still runs — and you save only the reprocessing and full-rate re-billing of the cached prefix. Prompt caching makes an invocation cheaper and faster; semantic caching makes the invocation disappear. That is why semantic caching can show even larger headline savings (AWS references up to roughly 86% inference-cost reduction and up to roughly 88% latency reduction — again, benchmark claims to verify, not guarantees) — it is eliminating whole calls rather than trimming them.

The trade is that semantic caching is a customer-implemented pattern, not a single managed Bedrock toggle you switch on. AWS documents reference implementations built from its data services: Amazon ElastiCache for Valkey (version 8.2 and later) with vector search as a read-through semantic cache (using an HNSW index and cosine similarity, with the query embeddings produced by a model such as Amazon Bedrock Titan Text Embeddings), and Amazon MemoryDB vector search used as a persistent semantic cache. The embeddings can come from Bedrock, Amazon SageMaker, or another source. (Which service versions and configurations are current is fast-moving — verify the specific ElastiCache/MemoryDB version and vector-search details against current AWS documentation.)

### The correctness risk of semantic caching — and how to bound it

Prompt caching and semantic caching do not carry the same risk profile, and understanding why is essential. Prompt caching is low-risk for correctness: the same model still generates a fresh response on every call, so a cache miss merely costs you the savings — it never returns a wrong answer. Semantic caching is different, because it returns a *stored* answer in place of generating a new one. Its characteristic risk is that, because matching is by similarity and not by exact intent, the cache can return a stale or not-actually-equivalent answer for a query that is similar in wording but different in intent. "What is the return policy for electronics?" and "What is the return policy for opened electronics?" are textually close and may embed close together, yet they have different correct answers — and a too-permissive cache would serve the first answer for the second question. The risk is tunable in a dangerous direction: lowering the similarity threshold raises the hit rate (more queries match, more calls eliminated, more savings) but simultaneously raises the chance of serving a not-truly-equivalent answer. Dynamic or real-time content (live prices, account balances, anything that changes minute to minute) is a poor cache candidate for the same reason — even an exact-intent match can be stale by the time it is served.

Because the risk is real, semantic caching must be deployed with explicit mitigations rather than as a naive lookup. The primary one is a minimum embedding-similarity threshold: only return a cached response when similarity is high enough that the queries are genuinely equivalent, and tune the threshold to the cost of being wrong. AWS tuning guidance suggests a strict, high threshold (on the order of ~0.95) for low-risk-tolerance domains such as medical, legal, and financial use cases, relaxing toward ~0.75 for higher-risk-tolerance domains, and recommends starting high (around 0.90-0.95) and lowering gradually while monitoring answer accuracy — but treat these specific numbers as AWS guidance that varies by workload, not as fixed constants. A second mitigation is a cache-invalidation policy, typically a TTL matched to how fast the underlying data changes: a short TTL for dynamic data (on the order of 5-15 minutes for real-time prices) and a longer one for stable facts (on the order of ~24 hours), so stale answers expire before they can be served. (Treat those TTL figures as guidance, not rules.) Beyond those two, event-driven invalidation (purging cached entries when the underlying source data changes) and restricting what is cacheable to repeated queries with stable answers further reduce the staleness exposure. The discipline mirrors Section 2's quality-bar discipline: tune for savings, but measure answer accuracy, and let a drop in accuracy be the signal to raise the threshold or shorten the TTL — not an acceptable cost of a higher hit rate.

**Q1:** A support assistant on Amazon Bedrock attaches the same 40-page product manual and the same long system prompt to most requests, and the team wants to cut both cost and latency without changing the model or rewriting the prompt. **Q2:** True or False — Both prompt caching and semantic caching still invoke the foundation model on a cache hit; they just make the invocation cheaper.

**A: False.** This is the core distinction. On a *semantic*-cache hit the model is not invoked at all — the system returns the stored response and skips generation entirely, saving the full cost and latency of the call. On a *prompt*-cache hit the model *is* still invoked — generation still runs — and only the cached prefix is skipped, so the invocation is cheaper and faster but still happens. Prompt caching reduces the cost of an invocation that occurs; semantic caching eliminates the invocation.

**Q3:** A team deploys semantic caching on a financial-advice assistant and, to maximize the hit rate and savings, sets the embedding-similarity threshold very low. Users start receiving answers that are close to their question but not actually correct for what they asked. What went wrong, and how should it be fixed?

**A:** Lowering the similarity threshold raised the hit rate but also raised the chance of returning a not-actually-equivalent answer — a query that is similar in wording but different in intent now matches a cached entry and is served the wrong answer. This is semantic caching's characteristic correctness risk, and it is most dangerous exactly here, in a low-risk-tolerance domain like financial advice. The fix is to raise the minimum embedding-similarity threshold so only genuinely equivalent queries hit (AWS guidance points to a strict ~0.95 for medical/legal/financial use cases, starting high and lowering only while monitoring accuracy — treat the number as guidance, not a constant), and to add a cache-invalidation policy (a TTL matched to how fast the data changes, plus event-driven invalidation) so stale entries expire. The governing discipline is to tune for savings but measure accuracy, and let an accuracy drop trigger a stricter threshold — not accept it for a higher hit rate.

**Q4:** An engineer wants to enable prompt caching on a high-volume batch job that processes millions of documents overnight through the Bedrock batch inference API, reasoning that the shared instruction prefix will be cached. Why might this not work as expected?
- A) Batch inference already caches everything automatically, so it is redundant
- B) Prompt caching is supported on on-demand inference, not the batch inference API (verify against current docs), so the batch job cannot use it
- C) Prompt caching only works for image models, not text
- D) The instruction prefix is too short to ever be cached

**A: B** — Prompt caching's inference-path support has centered on on-demand inference (Converse/ConverseStream, InvokeModel, Prompt Management, cross-region inference) and is not supported with the batch inference API. So the assumption that the shared prefix will be cached during a batch job does not hold. (This is a fast-moving constraint — confirm against current AWS documentation.) A invents automatic batch caching; C is false (text models are the primary supported case); D is a possible but unstated concern — the documents and prefix here are large, so the binding reason is the batch-API constraint, not prefix length.

**Q5:** For each scenario, which caching strategy (if any) applies — and state the precondition that decides it. (a) A stream of entirely unique, never-similar one-off prompts. (b) Thousands of users asking the same FAQ in different phrasings, with stable answers. (c) The same tool-definition block sent on every agent call.

**A:** (a) **Neither.** Caching's precondition is that content actually repeats; a workload of genuinely unique, non-repeating prompts has no prefix to reuse and no prior equivalent query to match, so both strategies return nothing. (b) **Semantic caching.** The repetition is in the *meaning* of the questions (same FAQ, different words), and the answers are stable, so matching by embedding similarity lets the system return a stored answer and skip the model call — exactly semantic caching's case, provided the similarity threshold and TTL are tuned for the domain. (c) **Prompt caching.** The tool-definition block is a static prefix repeated across calls, so caching the prefix avoids reprocessing and full-rate re-billing while the model still runs the actual request. The deciding question in every case is *what repeats* — a static prefix points to prompt caching, repeated question meaning points to semantic caching, and nothing repeating points to neither.

## Section 4: Model Selection for Cost & Latency

The first two levers in the Cost Lever Priority leave the model untouched: token efficiency (Section 2) trims what you send and receive, and caching (Section 3) avoids re-paying for content that repeats. Both lower cost without ever changing *which* model runs the request. The third lever is the first one that does. Model selection asks a different question — not "how do I make this model cheaper to call?" but "is this the right model to be calling at all?" — and the answer reshapes the unit economics of the workload. Per-token prices differ by an order of magnitude or more across a provider's lineup: a small, fast model in a family is dramatically cheaper per token than the family's flagship. So switching a request from a larger model to a smaller or cheaper one changes the unit cost of *every* token that request consumes, on input and output alike. Because cost is token volume times per-token rate (Section 1), and the first two levers attacked volume, lever 3 attacks the rate. That is why it is the third lever and not the first: it is more powerful per request than trimming tokens, but it is also a behavioral change rather than a volume change, so it carries a constraint the volume levers do not.

That constraint is a quality bar, and it is non-negotiable. The AWS Well-Architected Generative AI Lens states the discipline directly in its cost-optimization best practice on right-sizing model selection (GENCOST01-BP01): start from the smallest, cheapest model that could plausibly do the job and scale *up* only until you reach an "acceptable model" — one that meets the quality bar your use case has defined — rather than starting from the flagship and hoping to trim later. The cheaper or smaller model is allowed to replace a larger one only if it still meets the task's defined quality acceptance criteria. This is the same quality-bar discipline Section 2 applied to token reduction, now applied to model choice: you optimize for cost up to the point where output quality would fall below the bar, and not one step past it. A model that is half the price but fails the task's accuracy or formatting requirements is not a saving — it is a regression that ships wrong answers cheaply. The practical implication is that lever 3 is inseparable from evaluation: you cannot responsibly downgrade a model without a quality bar to measure against and a test set that proves the cheaper model clears it (the evaluation mechanics for establishing that bar belong to Guide 06, Testing, Evaluation & Troubleshooting).

There are two structurally different ways to act on this lever, and the exam tests the difference between them. Both route work to a cheaper model when possible and a larger model when necessary; they differ in *when* the routing decision is made and what information it uses.

### Model tiering versus model cascading

A terminology note first, because it matters for how you read the rest of this section. The words **tiering** and **cascading** are teaching labels this guide uses to name two routing patterns clearly — they are not Amazon Bedrock product names, and you will not find a "tiering" or "cascading" toggle in the console. What *is* documented, in GENCOST01-BP01 and the broader cost-optimization guidance, is the underlying idea: route requests by predicted difficulty so that easy work lands on cheap models and only hard work reaches expensive ones. Tiering and cascading are two ways to implement that idea. (One of them, routing within a model family, *is* productized as Intelligent Prompt Routing, covered next — but the general tiering/cascading vocabulary is the guide's, not AWS's.)

**Model tiering** makes the routing decision *up front*, before any model runs. A classifier — which can be a rules engine, a small cheap model used as a router, or a learned scorer — predicts each request's difficulty or complexity and dispatches it directly to a designated model tier: trivial requests to the small tier, hard requests to the large tier, perhaps a middle tier in between. The defining property is that each request is sent to exactly one model, chosen in advance. There is no second attempt and no escalation; the routing cost is whatever it takes to classify, and then the request runs once. Tiering's strength is predictability — a request that is correctly classified as hard goes straight to the capable model with no wasted first attempt — and its weakness is that it is only as good as the up-front prediction. Misclassify a hard request as easy and you get a low-quality answer from the small tier with no automatic recovery.

**Model cascading** makes the routing decision *after the fact*, based on what the cheap model actually produced. The request goes to the smaller, cheaper model first; the system then checks the output against a predefined confidence or quality threshold (a self-reported confidence score, a validator model, a schema or business-rule check, or any signal that the answer is good enough); and only if that check *fails* does the system escalate the same request to a larger, more capable model. The defining property is the conditional second call. This produces a real cost benefit: the cheap model handles all the easy requests on its own, and those requests never incur a single large-model token, so a workload where most requests are easy pays mostly cheap-model prices. But the conditional second call also creates cascading's characteristic risk, which is about latency rather than correctness. When a request *does* escalate, it pays for both models in sequence — it waits out the cheap model's full latency, *then* waits out the large model's full latency — so an escalated request is slower than it would have been if it had been sent straight to the large model in the first place. Cascading trades average-case cost for worst-case latency: it is cheaper on the requests that succeed cheaply and slower on the requests that escalate. Whether that trade is worth it depends on the escalation rate. If 90% of requests clear the cheap model, the rare double-latency request is a fine price for the broad cost win; if half the requests escalate, you are paying double latency constantly and would have been better off tiering hard requests straight to the large model or just using the large model outright.

**Diagram (described):** The cascading pattern is a left-to-right decision flow. An incoming request first goes to a small, cheap model for an attempt. The model's output then reaches a decision point: is the confidence above the threshold? If yes, the system returns the answer immediately — this is the cheap path. If no, the request escalates to the large model, and only after that large-model call completes does the system return the answer — this is the slow, costly path. The key thing the diagram makes concrete is that the escalation branch visibly traverses *both* models in sequence (small model, then large model), and that longer two-model path is exactly the added-latency risk of cascading.

The contrast to hold onto: tiering decides once, up front, by *predicted* difficulty and runs exactly one model — fast and predictable, but vulnerable to misclassification. Cascading decides after a cheap first attempt, by *measured* output quality, and may run two models in sequence — cheaper on easy traffic, but slower on whatever escalates.

### Amazon Bedrock Intelligent Prompt Routing

Tiering and cascading are patterns you build. Amazon Bedrock Intelligent Prompt Routing is the managed feature that productizes the routing idea *within a single model family*. You configure a prompt router for a family, and for each incoming request Bedrock predicts the response quality each candidate model in that family would produce and routes the request to the most cost-appropriate model that should still meet quality — sending simpler prompts to the smaller, cheaper model in the family and harder prompts to the larger one, behind a single endpoint. The application calls the router, not a specific model, so the cost/quality routing happens server-side without the caller implementing a classifier or a cascade. It is, in effect, AWS-managed tiering scoped to one family.

The supported families and Regions are fast-moving and **must be verified live** before you rely on them. As of this research, Intelligent Prompt Routing supported routing within the Amazon Nova family (Nova Lite and Nova Pro), the Anthropic Claude family (Claude 3 Haiku, 3.5 Haiku, 3.5 Sonnet, and 3.5 Sonnet v2), and the Meta Llama family (Llama 3.1 8B and 70B, 3.2 11B and 90B, and 3.3 70B), with routing confined to within a family rather than across families. Region availability spans a mix of per-model single-Region access and cross-Region inference profiles across several US, EU, and Asia Pacific Regions plus select AWS GovCloud (US) Regions, and it varies by model — so there is no single "supported Regions" list to memorize.

There is also a status inconsistency worth flagging explicitly, because it is exactly the kind of detail an exam answer or a design decision can get wrong. An AWS Machine Learning blog dated April 22, 2025 announces Intelligent Prompt Routing as **generally available**, while the Bedrock User Guide page for the feature still carried preview-era wording at the time of this research. Treat the feature as GA on the strength of the GA announcement, but be aware the documentation is not fully consistent — and treat the supported families, the supported Regions, and the GA-versus-preview status as fast-moving facts to confirm against current AWS documentation rather than as settled constants.

### Latency-optimized inference

Model selection is usually framed as a cost lever, but it is also a latency lever, and latency-optimized inference is where the two meet. It is not a separate model — it is a configuration choice on a supported model that makes Bedrock serve that same model's inference faster than the standard path. You opt in per request by setting the inference configuration field `performanceConfig.latency` to `optimized` (the default is `standard`); on a supported model in a supported Region, Bedrock then runs that request on a latency-optimized serving path that reduces inference response time relative to standard inference for the same model. The key framing for the exam is that it lowers latency *without changing the model* — you get the same model's outputs, served faster — which makes it distinct from streaming (Section 6), which lowers *perceived* latency by emitting tokens as they generate, and distinct from switching to a smaller model, which lowers latency by changing what runs.

Two operational details matter. First, there is a graceful fallback: when a request asks for the optimized path but the optimized capacity quota is exceeded, Bedrock falls back to standard inference for that request and bills it at the standard rate rather than failing the call — so opting in is safe, but you cannot assume every request actually received the optimized treatment. Second, availability is narrow and **must be verified live**, because at the time of this research latency-optimized inference was marked **preview** in the Bedrock User Guide. As researched, the optimized path was available (via cross-region inference) for a small set of model-and-Region pairs — Anthropic Claude 3.5 Haiku in us-east-2 and us-west-2; Meta Llama 3.1 70B in us-east-2 and us-west-2; Llama 3.1 405B in us-east-2 (with a combined input-plus-output token cap on the order of ~11K, above which the request falls back to standard); and Amazon Nova Pro in us-east-1 and us-east-2. Read that as "a few specific models in select US Regions," and verify the exact model-and-Region pairs, the token caps, and the preview-versus-GA status against current AWS documentation before designing around it.

A note on the headline numbers you will see in marketing material. AWS blog and announcement content cites figures such as "up to 30%" or specific percentage latency reductions for latency-optimized inference, and similar percentage cost or latency claims appear for the other features in this section. Those are vendor benchmark and best-case claims, not guarantees — actual results depend on the model, Region, prompt, and load — so this guide does not present any specific percentage as a fact you can count on. Treat every such percentage as something to verify and benchmark for your own workload.

## Section 5: Inference Cost & Throughput Options

The first three levers changed *what you send* (token efficiency), *what you re-pay for* (caching), and *which model runs* (model selection). The fourth lever changes none of those. It leaves the prompt, the cached prefixes, and the model exactly as the earlier levers left them, and instead changes the *commercial and operational arrangement* under which the tokens are served — the billing mode and the throughput contract. That is why it sits last in the Cost Lever Priority: it is the highest-commitment lever, and the commitment is real in a way the first three are not. Token efficiency and caching are reversible configuration changes that cost nothing to adopt; model selection is a behavioral change you can validate and roll back. The fourth lever asks you to either *commit to paying for dedicated capacity by the hour whether or not you use it* (Provisioned Throughput) or *give up real-time responsiveness* (batch inference). Both are commitments you should make only after the cheaper, lower-risk levers are exhausted and only when the workload's shape justifies locking something in.

The two committed options are not interchangeable — each answers a different volume question, and naming the condition that selects each is the heart of this section. Provisioned Throughput is the right choice when volume is **steady, high, and sustained**: a production endpoint serving a predictable, continuous stream of real-time requests at a level where on-demand's shared-capacity quotas would throttle you, and where the hourly commitment is amortized across enough traffic to pay for itself. Batch inference is the right choice for the opposite traffic shape — a **large, bounded set of requests that does not need real-time answers**: tens or hundreds of thousands of prompts you can submit as one job and collect later, where the work is offline and the only thing you are trading away is immediacy. One commits to capacity for a relentless live stream; the other trades latency for a discount on a finite pile of offline work. If a workload is neither steady-and-high nor large-bounded-and-offline — if it is spiky, low, or unpredictable — then neither committed option fits and on-demand remains correct. This section develops on-demand as the default, the two committed options as the fourth lever, and Cross-Region Inference as a throughput-and-resilience mechanism that overlays the on-demand path. The deeper invocation mechanics of all of these — how to actually purchase Model Units, format batch files, and configure inference profiles — belong to Guide 01 (Foundation Models & Amazon Bedrock Core), which this section points to rather than restating.

### On-demand inference — the default, and what pushes you off it

On-demand is where every workload starts and where most should stay. It is the serverless, pay-per-use path established in Section 1: you call the model and are billed on input tokens plus output tokens at the model's published rates, with no upfront commitment, no minimum, and no idle capacity to pay for. Its cost behavior is the cleanest of any option — spend tracks usage exactly, so a quiet hour costs nothing and a busy hour costs in direct proportion to the tokens consumed. That property is exactly what makes on-demand the right answer for variable, unpredictable, or low-to-moderate traffic, and for any prototype or new workload whose volume you cannot yet predict. You do not commit to capacity you might not use.

What pushes a workload off on-demand is not cost per token — that is identical whether you send one request a day or a thousand a second — but **throughput limits**. On-demand capacity is shared and governed by per-model, per-Region service quotas on the `bedrock-runtime` inference endpoint. Three quota types matter. There is a tokens-per-minute quota (counted as input plus output tokens combined against a single per-model limit), a tokens-per-day quota (by default the per-minute quota multiplied across the day, and reduced for new accounts), and — for some models but not all — a requests-per-minute quota. When traffic exceeds the applicable quota, Bedrock throttles the excess and returns a throttling error, which the application must absorb with retries and backoff or by smoothing traffic. At steady high volume, throttling stops being an occasional burst event and becomes a structural ceiling: you are bumping the quota constantly, and even with quota-increase requests the shared-capacity path offers no *guarantee*. That structural ceiling is the pressure that motivates moving to a committed option — Provisioned Throughput for dedicated guaranteed capacity, or Cross-Region Inference to spread the load across more Regions' quotas. The exact quota values are per-model and per-Region, live in the Service Quotas console, and change as models and accounts change, so treat any specific tokens-per-minute, tokens-per-day, requests-per-minute, or concurrency number as **requiring live verification** rather than memorizing it.

### Provisioned Throughput — committed capacity for steady, high volume

Provisioned Throughput is the lever you pull when on-demand's shared-capacity ceiling is no longer acceptable and volume is high enough to justify a commitment. Instead of drawing from a shared pool subject to throttling, you purchase **dedicated** model-invocation capacity at a **fixed hourly price**. The capacity is expressed in Model Units (MUs): each MU delivers a specified, guaranteed throughput for the chosen model, defined as a number of input tokens it can process per minute and a number of output tokens it can generate per minute, aggregated across all your requests in that minute. You buy more MUs to guarantee more throughput. The cost/throughput tradeoff is the defining property — you exchange a committed, always-on hourly cost for guaranteed throughput that on-demand cannot promise, and crucially, the hourly meter runs whether or not you send any traffic. Billing continues until you delete the provisioned model.

The hourly price depends on three things: the model you choose, the number of MUs you provision, and the commitment term. The term is where the discount lives. You can provision with **no commitment** (delete any time), a **one-month** commitment, or a **six-month** commitment, with the longer terms earning a deeper hourly discount in exchange for the longer lock-in. There is also a prerequisite that catches people out: MUs are not available by default in arbitrary quantity, so before purchasing committed Provisioned Throughput you must request a Model Unit limit increase (through AWS Support / your AWS account manager). And one scope note that connects to Cross-Region Inference below: cross-Region inference profiles do not currently support Provisioned Throughput — the two throughput mechanisms do not stack.

The volume condition under which Provisioned Throughput becomes cost-effective relative to on-demand follows directly from the always-on hourly meter. Because you pay for the dedicated capacity continuously, the per-token effective cost only undercuts on-demand when you keep the provisioned capacity busy enough — that is, when traffic is **steady, high, and sustained** enough that the committed hourly cost is amortized across a large, continuous token volume. For spiky or low-volume traffic the meter runs through the idle periods and you end up paying for capacity you did not use, which is why Provisioned Throughput is the wrong answer for variable workloads even when they occasionally spike. The other condition that forces Provisioned Throughput has nothing to do with cost: a customized (fine-tuned, distilled, or imported) model cannot be invoked on-demand at all — serving a custom model in production *requires* Provisioned Throughput. The mechanics of purchasing MUs, the quota-increase request, and serving custom models through a provisioned model are owned by Guide 01 and cross-referenced here rather than restated. Treat the per-MU throughput figures, the per-MU and per-model pricing, the exact commitment-term discounts, and the supported models and Regions as **fast-moving details requiring live verification** — the AWS documentation itself directs you to your account manager for the MU specifics and pricing, and no dollar figures appear in this guide.

### Batch inference — discounted asynchronous bulk processing

Batch inference attacks cost from the opposite direction. Where Provisioned Throughput buys *more* responsiveness (guaranteed real-time capacity) at a committed cost, batch *gives up* responsiveness in exchange for a discount. You format your prompts as input files, upload them to Amazon S3, submit a single batch job that points at the bucket, and Bedrock processes the prompts asynchronously and writes the responses back to S3 — you can be notified of completion through Amazon EventBridge rather than polling. Because the work runs in bulk off the interactive path, batch is documented at a **lower price than on-demand for the same model**. The exact discount is an announced figure (described as 50% lower than on-demand at the time of this research), and per this guide's policy it is treated as a benchmark/announced value to **verify live**, along with which models are eligible — not recited as a settled fact.

The tradeoff that defines batch is latency: the results are **not real-time**. A batch job is throughput-oriented and asynchronous, so you submit it and collect the output later — there is no token-by-token response and no interactive turn. That makes batch the natural fit for a large, bounded set of requests where no individual answer is time-sensitive: the representative use case is **bulk document processing** — for example, summarizing or classifying a back catalog of tens of thousands of documents, enriching a dataset, or generating embeddings over a corpus — work that can run overnight and be picked up when it finishes. The same constraints that make batch cheap also limit it: batch is not supported for provisioned models, and each record is processed independently with no multi-turn interaction, so batch does not support tool calling (function calling) or structured-output response formatting. A scenario that needs an agent to call tools, or a strictly enforced JSON schema per request, rules batch out even if the volume looks batch-shaped. There is also a concurrency ceiling — a limit on how many batch jobs can run at once per model per Region (reported as on the order of ten at the time of research) — which large pipelines manage with a job queue. The discount rate, the supported models and Regions, and the exact concurrency limit are all **fast-moving and require live verification**; the batch *file format and job mechanics* are owned by Guide 01 and cross-referenced rather than restated.

### Cross-Region Inference — distributing inference across Regions

Cross-Region Inference is not a fourth billing mode; it is a throughput-and-resilience mechanism that overlays the on-demand path. Instead of calling a model in a single Region, you call a cross-Region inference profile — a construct that defines a model plus the set of Regions requests may be routed to — and Bedrock performs real-time capacity checks and routes each request to a Region with available capacity, first the source Region and then other Regions in the profile if needed. The payoff is twofold: it absorbs traffic bursts and raises effective throughput by drawing on more than one Region's capacity (directly relieving the on-demand throttling pressure described above), and it adds resilience by giving requests somewhere else to land when a single Region is constrained. Guide 01 (Foundation Models & Amazon Bedrock Core) is the home of the resilience story and the related dynamic-model-switching pattern (the AppConfig-driven abstraction layer that swaps models without code changes) — this guide points there for that depth rather than restating it, and treats Cross-Region Inference here specifically as a cost-and-throughput consideration.

Two considerations belong in the cost-and-performance picture. On **cost**: there is no additional routing or data-transfer charge for cross-Region inference — you pay normal inference pricing, calculated based on the Region from which you call the profile, so the cross-Region routing itself is free. Beyond that, AWS distinguishes geographic profiles (which route within a defined geography such as US or EU) from global profiles (which may route to any supported commercial Region worldwide), and global profiles are documented as carrying an approximate cost savings relative to the geographic/standard path (announced at roughly 10% at the time of research — treat that figure as a benchmark/announced value to **verify live**). On **data residency**: the geography-scoped profile exists precisely for compliance — it confines processing to Regions within the chosen geography, so when a regulatory or data-residency requirement says data may not leave a jurisdiction, the geographic profile (or single-Region inference) is the compliant choice, while the global profile, which can route anywhere worldwide, is appropriate only when there is no residency constraint. A related nuance worth holding: a cross-Region profile can route to Regions you have not manually enabled in your account, and all such traffic stays on the encrypted AWS network and is logged in CloudTrail in the source Region. The supported models, the Region membership of each profile, and the GA-versus-preview status of specific profiles are **fast-moving and require live verification**.

The comparison that the exam rewards is on-demand versus Provisioned Throughput versus batch inference — the three ways the *same model* can be billed and served. (Cross-Region Inference is omitted from this table on purpose: it is an overlay on the on-demand path, not a separate billing mode.)

## Section 6: Application Performance Optimization

The first five sections optimized for cost; this one optimizes for time. The two goals overlap — bounding output length cuts both the bill and the latency, and caching saves both money and milliseconds — but they are not the same goal, and conflating them is how engineers reach for the wrong fix. A workload can be cheap and slow, or fast and expensive, and "the application feels sluggish" is not one problem but a family of distinct problems, each with its own cause and its own remedy. The discipline this section teaches is diagnostic before it is corrective: before you pick a technique, you have to name precisely *which* latency is hurting, because the technique that fixes one kind of slowness does nothing for another. The single most important distinction in the whole section — the one the exam keeps returning to — is between the time it takes to produce the *complete* response and the time it takes to show the user *something*. Get that distinction right and the rest of the section is a lookup table.

Consider a concrete scenario that runs through this section. A team ships a customer-support assistant on Amazon Bedrock. Two complaints arrive in the same week. The first: "it just sits there for four or five seconds before anything appears, and it feels broken." The second, from a different team running a nightly job that classifies ten thousand tickets: "the batch takes hours and we keep getting throttled." These are both "performance" complaints, but they have almost nothing in common. The first is a perceived-latency problem on a single interactive request — the user is staring at a blank screen waiting for the first token. The second is a throughput problem across many requests — total wall-clock time and quota ceilings, with no human waiting on any individual answer. Streaming fixes the first and is irrelevant to the second; concurrency management and quota headroom fix the second and do nothing for the first. The same word, two unrelated cures. The sections below build the vocabulary to tell them apart.

### Total latency versus perceived latency

A foundation model produces a response in two phases, and the cost in time of each phase is governed by a different quantity. The first phase is *prefill*: the model ingests the entire input prompt — system prompt, retrieved context, conversation history, and the new message — in a single forward pass and produces the first output token. The time from the moment the request is received to the moment that first token is available is the time to first token, which Amazon publishes as the `TimeToFirstToken` (TTFT) metric. The second phase is *decode*: the model generates the rest of the output autoregressively, one token at a time, each token requiring its own forward pass. The time from request received all the way to the *last* output token — the complete response — is the total latency, which Amazon publishes as the `InvocationLatency` metric. These two metrics measure fundamentally different experiences. `TimeToFirstToken` measures responsiveness — how long the user waits before anything happens. `InvocationLatency` measures completeness — how long until the whole answer is done.

The relationship between them is not loose; AWS documents it as an additive model. Total latency is the time to the first token plus the time to decode all the remaining tokens: `InvocationLatency` ≈ `TimeToFirstToken` + (output token count ÷ output-tokens-per-second) × 1000 in milliseconds. The middle term is the part that matters for reasoning about performance. Decode time scales with the number of output tokens, divided by the model's output-tokens-per-second rate (the OTPS that Amazon surfaces as a runtime performance signal). The consequence is sharp: total latency grows with output length, but time to first token does not. A request that produces a 50-token answer and a request that produces a 2,000-token answer can have nearly identical `TimeToFirstToken` — the prefill phase is the same shape — yet wildly different `InvocationLatency`, because the long answer spends far longer in the decode loop. This is why output length is simultaneously a cost lever (Section 1) and a total-latency lever: every output token you do not generate is a forward pass you do not wait for and a token you do not pay for.

**Diagram (described):** A simple left-to-right timeline of a single invocation. It starts at "Request received." The first segment runs from request received to the first output token — this interval is the TimeToFirstToken (TTFT). From there the flow enters the "decode loop," which produces output one token at a time. The decode loop ends at the "last token," and the full span from request received all the way to that last token is the total latency (InvocationLatency). The diagram visually reinforces that TTFT is just the opening segment, while total latency covers the whole timeline including the per-token decode loop that grows with output length.

Naming the metric tells you which technique applies, and this is the governing logic of the entire section. If the symptom is "the screen is blank too long before anything appears," the hurting quantity is `TimeToFirstToken`, and the fixes are the ones that get a token in front of the user sooner — streaming above all, and reducing the prefill workload (a smaller retrieved context, a shorter prompt). If the symptom is "the complete answer takes too long to finish," the hurting quantity is `InvocationLatency`, and the fixes are the ones that shorten the decode phase — bounding output length with `maxTokens`, choosing a faster model or latency-optimized inference (Section 4), or simply asking for less. If the symptom is "we cannot process enough requests per unit time," the hurting quantity is neither single-request metric but aggregate throughput, and the fixes live in concurrency and quotas. One diagnostic question — *which clock is the user actually watching?* — routes you to the right family of techniques every time.

### Response streaming

Streaming is the highest-leverage fix for a perceived-latency problem, and understanding exactly what it does and does not do is the crux of this section. By default, a Bedrock invocation buffers the entire response and returns it in one block when generation is complete — the caller waits the full `InvocationLatency` before seeing anything. The streaming APIs change the *delivery* of the response, not its generation: `ConverseStream` and `InvokeModelWithResponseStream` return tokens to the caller incrementally as they are generated, emitting a sequence of partial-result events (`contentBlockDelta` events) so the application can render text the instant the first token is produced rather than waiting for the last. The user sees output begin at `TimeToFirstToken` instead of at `InvocationLatency` — and since TTFT is typically a small fraction of the total, the perceived wait collapses from seconds to a moment. For any human-facing, conversational, or long-form generation surface, this is the single most effective responsiveness improvement available, which is why interactive assistants stream by default.

Here is the part the exam tests, and it is a logical consequence of the additive latency model rather than a verbatim AWS sentence: streaming does **not** reduce total generation time. The model still performs exactly the same prefill and exactly the same per-token decode passes in exactly the same wall-clock time — `InvocationLatency` is unchanged. What streaming changes is *when the user first sees output*, not *when the model finishes*. The last token still arrives at the same moment it would have without streaming; the difference is that the user was reading along the whole time instead of staring at a blank screen. Put precisely: streaming improves perceived latency (`TimeToFirstToken` becomes the felt wait) and leaves total latency (`InvocationLatency`) untouched. A scenario that complains about *total* processing time, batch completion time, or the time until a downstream system can consume the *whole* response will not be fixed by streaming — that is the trap. The mechanics of the streaming APIs themselves (event shapes, SDK handling, and which models support streaming, which you can check via `GetFoundationModel`'s `responseStreamingSupported` flag) are established in Guide 01 and the streaming-API material; this guide points there rather than re-teaching them, and treats streaming purely as the perceived-latency lever.

### Throughput, concurrency, and pre-computation

When the problem is volume rather than the latency of a single request, the levers change. Throughput is raised by issuing requests in parallel and managing concurrency — running many invocations at once rather than serially, so the aggregate completes in a fraction of the wall-clock time a sequential loop would take. The nightly ten-thousand-ticket job from the opening scenario is the canonical case: no individual classification is latency-sensitive, but the *set* must finish in a window, so you fan the requests out across concurrent workers. The primary constraint on this technique is not your client — it is the service. Concurrent throughput against Bedrock is bounded by per-account, per-model service quotas on the `bedrock-runtime` endpoint: a requests-per-minute (RPM) limit and a tokens-per-minute (TPM) limit (TPM counts input and output tokens combined). Push past either ceiling and Bedrock rejects the excess with a `ThrottlingException`, which the application must absorb with retries and exponential backoff or by smoothing the submission rate. So the real shape of "go parallel" is "go parallel up to the quota, then throttle gracefully" — raising concurrency past the quota does not raise throughput, it just generates throttling errors. (For high, steady volume, the structural fix is the Section 5 lever: Provisioned Throughput for dedicated capacity, or Cross-Region Inference to draw on more than one Region's quota. The `EstimatedTPMQuotaUsage` CloudWatch metric lets you watch how close you are running to the ceiling.)

Pre-computation and caching attack latency from a different angle: do the expensive work once, ahead of time or on the first request, and reuse the result instead of recomputing it. If an expensive result — a model response, an embedding, a retrieved-and-reranked context — is going to be needed repeatedly, computing it once and serving it from a store turns every subsequent need into a fast lookup instead of a full invocation. The constraint that defines this technique is the reuse precondition, and it is the same precondition that governs caching in Section 3: pre-computation reduces latency *only when the same expensive result is actually reused across requests*. If every request needs a unique result, there is nothing to reuse, the pre-computed store is never hit, and you have added complexity for no gain. This is exactly the prompt-caching and semantic-caching story from Section 3 viewed through the latency lens rather than the cost lens — prompt caching avoids reprocessing a repeated prefix (saving prefill time and therefore TTFT), and semantic caching avoids the invocation entirely (saving the full `InvocationLatency`) when a query repeats. Section 3 carries the depth; here the point is simply that caching is a performance technique *conditioned on repetition*, not a universal speed-up.

### Inference-parameter tuning

The inference parameters split cleanly into two groups by their effect on performance, and the exam rewards knowing which group a given parameter belongs to. The first group controls *length*, and the member that matters here is the maximum-tokens setting (`maxTokens`). It bounds the maximum number of output tokens the model may generate before it must stop. Because total latency grows with output-token count (the decode term of the additive model) and output tokens are the costlier billing dimension (Section 1), `maxTokens` is a single setting that caps *both* the worst-case total latency *and* the worst-case output-token cost of a request. It does not make a short answer faster — a response that naturally ends at 80 tokens is unaffected by a 500-token ceiling — but it guarantees that no single request can run away into a multi-thousand-token generation that blows out both the latency budget and the bill. This is the direct mechanical connection back to the output-token cost lever from Section 1: the same knob that bounds cost bounds total latency, because both are driven by the number of tokens the model decodes.

The second group controls *randomness and diversity* — temperature, top-k, and top-p. These shape the candidate-token pool the model samples from at each decode step: temperature flattens or sharpens the probability distribution, top-k restricts sampling to the k most-likely tokens, and top-p (nucleus sampling) restricts it to the smallest set of tokens whose cumulative probability exceeds p. (In the Converse API, top-k is supplied through `additionalModelRequestFields` because not every model exposes it as a first-class field.) The performance-relevant fact is what these parameters do *not* do: they primarily shape output randomness and quality, not latency or cost directly. This is a logical consequence of how AWS documents them — they are described purely as controls on the output distribution, with no mechanism that changes the number of forward passes, the number of tokens generated, or the per-token rate. Lowering temperature does not make the model faster or cheaper; it makes the output more deterministic. The only indirect path by which a sampling parameter could touch latency is if it changed how *long* the model tends to generate before stopping, which is incidental and unreliable, not a control you tune for performance. The exam trap is to offer "lower the temperature" as a fix for a latency or cost problem — it is a quality/determinism knob, and the latency/cost knob is `maxTokens`.

### Retrieval and index optimization for RAG

For a RAG workload, a major performance lever lives upstream of the model entirely, in the retrieval step. Before the model is ever called, the retrieval pipeline runs a query against the vector index and injects the retrieved passages into the prompt as context — and both halves of that operation have performance consequences. The retrieval configuration itself adds latency: the vector search, any reranking pass, and (for advanced configurations) query decomposition all take time before the model invocation begins. More consequentially, the *volume* of retrieved context directly drives the input-token count. The number of retrieved results (`numberOfResults` in an Amazon Bedrock Knowledge Bases retrieval configuration), the chunking strategy that determines how large each retrieved chunk is, the search type (semantic versus hybrid), and whether reranking is applied all determine how many tokens of context land in the prompt. And input-token volume is exactly what drives prefill — more retrieved chunks mean a larger prompt, a longer prefill phase, a higher `TimeToFirstToken`, and a higher input-token cost on every call. So tuning retrieval is a two-for-one performance-and-cost lever: retrieving fewer, better-targeted chunks lowers both the perceived latency (smaller prefill) and the per-call token bill, provided you do not starve the model of the context it needs to answer correctly. The retrieval-pipeline mechanics — how chunking, search type, reranking, and `numberOfResults` are configured, and how the vector store is built — belong to Guide 02 (RAG, Vector Stores & Knowledge Bases); this guide points there for the depth and treats retrieval tuning here strictly as a latency-and-input-token lever. The `numberOfResults` default and maximum are version-sensitive (default 5, configurable up to 100 at research time) and should be verified live against current documentation rather than memorized.

### Matching the symptom to the technique

The whole section reduces to a routing table: name the symptom, identify which latency dimension it touches, and the technique follows. The dimension column is the diagnostic key — it forces you to decide whether the user is waiting on the *first* token (perceived), the *last* token (total), or on *aggregate* capacity (neither single-request dimension) before you pick a fix.

| Symptom / goal | Recommended technique | Latency dimension improved | Primary tradeoff or constraint |
|---|---|---|---|
| High perceived latency — blank screen before any output on an interactive request | Response streaming (`ConverseStream` / `InvokeModelWithResponseStream`) | Perceived (`TimeToFirstToken`) | Does not reduce total generation time; model must support streaming; client must handle incremental events |
| High total latency — the complete response takes too long to finish | Bound output with `maxTokens` (plus a faster/latency-optimized model from Section 4) | Total (`InvocationLatency`) | Truncates output if set too low; sampling parameters (temperature/top-k/top-p) do not help here |
| Low throughput — cannot process enough requests per unit time | Parallel requests / managed concurrency (and Section 5 committed capacity) | Neither — raises aggregate throughput | Bounded by per-account/per-model RPM and TPM quotas; exceeding them causes `ThrottlingException` |
| Repeated identical or equivalent work across requests | Pre-computation / caching (prompt caching, semantic caching — Section 3) | Perceived and/or total, when content repeats | Helps only when the same expensive result is actually reused; stale-answer risk for semantic caching |
| RAG: high TTFT and high input-token cost from oversized context | Retrieval/index tuning — fewer/better chunks, reranking, tuned `numberOfResults` (Guide 02) | Perceived (`TimeToFirstToken`) and input-token cost | Retrieving too little starves the model of needed context, degrading answer quality |

### Exam-Relevant Distinctions

| If the exam says... | The answer is... | Why |
|---|---|---|
| "App feels slow — blank screen for several seconds before any output" | Response streaming | The hurting metric is perceived latency / `TimeToFirstToken`; streaming shows the first token sooner |
| "Reduce the *total* time to produce the complete response" | Bound output length / faster model — not streaming | Streaming leaves `InvocationLatency` unchanged; total latency is driven by output-token count in the decode phase |
| "Lower the temperature to make the model respond faster / cheaper" | Distractor | Temperature, top-k, top-p shape randomness/quality, not latency or cost; the latency/cost knob is `maxTokens` |
| "Nightly bulk job is slow and keeps getting throttled" | Increase concurrency within quotas; request quota increase / Provisioned Throughput | Throughput is bounded by per-model RPM/TPM quotas; exceeding them returns `ThrottlingException` |
| "Cap the worst-case cost and latency of any single request" | Set `maxTokens` | One setting bounds both worst-case output-token cost and worst-case total latency (decode term) |
| "RAG answers are slow and prompts are huge" | Reduce retrieved context (`numberOfResults`, chunking, reranking) | More chunks → more input tokens → longer prefill (higher TTFT) and higher cost |
| "Caching will make every request faster" | Distractor unless content repeats | Pre-computation/caching reduces latency only when the same expensive result is reused across requests |

- `TimeToFirstToken` is published only for streaming operations; `InvocationLatency` measures end-to-end time to the last output token. The two answer different questions — responsiveness versus completeness.
- The published latency model is additive: total latency ≈ time to first token + (output tokens ÷ output-tokens-per-second). Decode time, and therefore total latency, scales with output-token count; TTFT does not.
- Streaming reduces perceived latency without reducing total generation time (a logical consequence of the additive model, not a verbatim AWS claim).
- `maxTokens` is the only inference parameter that bounds latency and cost; temperature, top-k, and top-p are output-distribution controls with no direct latency or cost effect.
- Exact RPM/TPM/tokens-per-day quota values are per-model, per-Region, adjustable, reduced for new accounts, and must be read from the Service Quotas console — never memorized.

### Knowledge Check

**Q1:** Users complain that a Bedrock-backed chat assistant "sits on a blank screen for four or five seconds before anything shows up." A developer proposes enabling response streaming. Will this help, and what exactly does it change?
- A) No — streaming only matters for batch jobs, not interactive chat
- B) Yes — streaming returns tokens incrementally, so the first token appears at `TimeToFirstToken` instead of the user waiting the full `InvocationLatency`; it improves perceived latency without changing total generation time
- C) Yes — streaming reduces the model's total generation time by parallelizing token production
- D) No — the only fix is to switch to a larger, faster instance type

**A: B** — The complaint is a perceived-latency problem: the user is waiting for the *first* token. `ConverseStream` / `InvokeModelWithResponseStream` deliver tokens incrementally (as `contentBlockDelta` events), so output begins at `TimeToFirstToken` rather than at the end of the full response. C is the classic trap — streaming changes *delivery*, not *generation*; the model still performs the same prefill and per-token decode passes, so `InvocationLatency` is unchanged. A is false (streaming is most valuable for interactive surfaces). D invents compute that does not exist on the serverless on-demand path.

**Q2:** True or False — Lowering the temperature parameter is an effective way to reduce both the latency and the cost of a Bedrock invocation.

**A: False.** Temperature, along with top-k and top-p, is a sampling parameter that shapes the randomness and diversity of the output distribution — it does not change the number of forward passes, the number of tokens generated, or the per-token rate, so it has no direct effect on latency or cost. (This follows from how AWS documents these parameters: purely as output-distribution controls.) The parameter that actually bounds latency and cost is `maxTokens`, which caps the number of output tokens the model may generate — and since total latency and output-token cost both scale with output-token count, that one setting bounds the worst case of both.

**Q3:** Fill in the relationship — AWS describes total latency as an additive model: `InvocationLatency` ≈ ___ + (___ ÷ output-tokens-per-second). Which term does response streaming improve, and which does it leave unchanged?

**A:** `InvocationLatency` ≈ `TimeToFirstToken` + (output token count ÷ output-tokens-per-second). Streaming improves the *perceived* wait by surfacing the first token at `TimeToFirstToken`, but it leaves `InvocationLatency` (the total time to the last token) unchanged — the model still runs the full decode loop in the same wall-clock time. The decode term grows with output-token count, which is why bounding output length with `maxTokens` is the lever for *total* latency, while streaming is the lever for *perceived* latency.

**Q4:** A nightly job classifies 10,000 support tickets through Bedrock. The team parallelizes aggressively to finish faster, but now the job fails partway through with repeated `ThrottlingException` errors. What is the constraint they hit, and what are the correct responses?

**A:** They hit the per-account, per-model service quotas on the `bedrock-runtime` endpoint — the requests-per-minute (RPM) and/or tokens-per-minute (TPM) limits. Raising client-side concurrency past those ceilings does not raise throughput; it just produces `ThrottlingException`. The correct responses are: (1) absorb throttling with retries and exponential backoff and smooth the submission rate to stay under the quota; (2) request a quota increase via Service Quotas; and (3) for sustained high volume, move to a Section 5 lever — Provisioned Throughput for dedicated capacity, or Cross-Region Inference to draw on more than one Region's quota. Watching `EstimatedTPMQuotaUsage` shows how close the job runs to the ceiling. (Exact quota values are per-model/per-Region and must be verified live.)

**Q5:** A RAG-backed assistant has slow first-token times and high per-call cost. Investigation shows each query retrieves 50 chunks and injects all of them into the prompt. A junior engineer suggests "switching to a lower temperature and enabling streaming" to fix the cost and slowness. What went wrong with that proposal, and what actually addresses it?

**A:** The proposal misdiagnoses both problems. Lower temperature does nothing for cost or latency (it is a randomness control). Streaming improves *perceived* latency but does nothing for the *cost* problem and nothing for the oversized prefill that is driving the high `TimeToFirstToken`. The real cause is retrieval volume: 50 chunks is a large input-token payload, which lengthens the prefill phase (raising TTFT) and inflates the input-token bill on every call. The fix is retrieval/index tuning — reduce `numberOfResults`, improve chunking, and add reranking so fewer, better-targeted chunks carry the needed context (Guide 02 owns the mechanics) — being careful not to retrieve so little that the model loses the context it needs to answer correctly. Streaming can still be added on top to improve responsiveness, but it is not what fixes this.

## Section 7: Monitoring & Observability Program

The first six sections handed you levers to pull — token efficiency, caching, model selection, committed throughput, and the performance techniques. This section builds the instrument panel that tells you *which* lever to pull, *when* to pull it, and whether pulling it actually worked. A foundation model workload is not a thing you optimize once and walk away from. Token volume drifts as usage patterns change, a model swap upstream quietly shifts latency, a new feature triples retrieved context, a prompt change inflates output length, and spend creeps month over month with no single event you can point at. The discipline that catches all of this is a *standing monitoring program*: the continuous, ongoing operation of logging, metrics, and cost instrumentation over the entire life of the workload, watched by dashboards, guarded by alarms, and audited by scheduled quality re-runs — running every day whether or not anything is wrong, precisely so that you find out the moment something starts to go wrong.

It is worth being exact about what makes this *standing* program different from the monitoring you already met in Guide 06 (Testing, Evaluation & Troubleshooting), because the exam draws the line here and so does real operational practice. Guide 06 used Model Invocation Logging, the `AWS/Bedrock` CloudWatch metrics, and X-Ray/tracing *diagnostically* — a specific incident has occurred (a spike in errors, a hallucination complaint, a latency regression on one endpoint), and you read the signals after the fact to root-cause that one incident. That is reactive and bounded: you open the logs because something broke, you find the cause, you close the investigation. This guide owns the other half — the *standing-program* scope that Guide 06 explicitly deferred to "future Guide 07." Here the very same signals are operated continuously and proactively: the dashboards are always up, the alarms are always armed, the golden-dataset re-runs are always scheduled, and the cost instrumentation is always attributing spend — so that drift, cost creep, and slow degradation surface on their own, without anyone first noticing a problem. The distinction to carry into the exam is simple: Guide 06 reads the signals to explain one incident; Guide 07 operates the same signals forever to prevent incidents and to detect the slow problems that have no single triggering event. Where this section needs the diagnostic mechanics — what a golden dataset *is*, how to read a trace to root-cause a bad answer — it points back to Guide 06 rather than restating them.

Consider the real-world shape of the problem. A team ships a production support assistant on Amazon Bedrock and it works well at launch. Six months later, three things have happened, none of which fired a pager. Average latency has crept from 1.2 seconds to 3 seconds because an upstream change enlarged the system prompt. Monthly spend has roughly doubled, hidden inside a steadily rising bill that never spiked on any single day. And answer quality on a class of edge-case questions has quietly degraded after a model version change. No incident occurred — there was nothing to *diagnose* — yet the workload is now slow, expensive, and less accurate. That is exactly the failure mode a standing monitoring program exists to catch: latency dashboards and a `TimeToFirstToken` alarm would have flagged the creep, AWS Cost Anomaly Detection (or a budget) would have surfaced the cost climb, and a scheduled golden-dataset re-run would have caught the quality drift. The rest of this section builds those three instrumentation paths — operational metrics, logging at scale, and cost instrumentation — and the dashboards, alarms, and anomaly detection that sit on top of them.

### The standing monitoring data flow

Before the individual instruments, here is the map of how they connect.

**Diagram (described):** Every Bedrock invocation is a single source event that fans out into three parallel instrumentation paths. The first path is operational metrics — the `AWS/Bedrock` CloudWatch namespace. The second is logging — Model Invocation Logging. The third is cost instrumentation. From the metrics path, three things branch off: CloudWatch dashboards (at-a-glance health), static-threshold alarms (known limits), and CloudWatch anomaly detection (for the unknown ones). The logging path splits into two destinations, Amazon S3 and CloudWatch Logs; the S3 destination in turn feeds the scheduled golden-dataset re-runs (the evaluation jobs that watch for quality drift). The cost-instrumentation path branches into Cost Explorer, AWS Budgets, and Cost Anomaly Detection. Finally, the alarming outputs converge: the static-threshold alarms, CloudWatch anomaly detection, Cost Anomaly Detection, and AWS Budgets all send notifications to an SNS topic, which is the single fan-out point to email, chat, ticketing, or automated remediation.

### CloudWatch metrics in the AWS/Bedrock namespace

Amazon Bedrock publishes operational metrics automatically to Amazon CloudWatch under the `AWS/Bedrock` namespace, with no logging configuration required — they are emitted for every account that invokes a model, and each is dimensioned by `ModelId` so you can break health down per model. These metrics are the always-on heartbeat of the standing program, and they fall into four categories that map directly onto the four signals you watch.

The **invocation-volume signal** is the `Invocations` metric — the count (as `SampleCount`) of successful `Converse`, `ConverseStream`, `InvokeModel`, and `InvokeModelWithResponseStream` requests. It tells you how much traffic the workload is actually serving and is the denominator for any error-rate or cost-per-call calculation. The **token signal** is two metrics — `InputTokenCount` and `OutputTokenCount` — which together are the cost-and-usage core of the entire guide: because cost is a function of token volume (Section 1), these two metrics *are* your cost-driver telemetry, and watching their sum over time is how you see token consumption trend before it shows up on the bill. When prompt caching is in use, two further token metrics appear — `CacheReadInputTokens` and `CacheWriteInputTokens` — letting you confirm that a cache is actually being hit rather than perpetually re-written. The **latency signal** is `InvocationLatency` (milliseconds from request sent to the last token received — total latency) and, for streaming operations only, `TimeToFirstToken` (milliseconds to the first token — perceived latency); these are the two clocks Section 6 taught you to tell apart, now surfaced as first-class metrics.

The **error and throttle signal** is the category most likely to be tested, because its structure is counterintuitive. There is **no single aggregate "error" metric** in the `AWS/Bedrock` namespace. Instead there are three distinct metrics: `InvocationClientErrors` (4xx-class problems caused by the request — bad input, access denied), `InvocationServerErrors` (5xx-class problems on the service side), and `InvocationThrottles` (requests rejected because a quota was exceeded), each counted as `SampleCount`. The trap is the throttle accounting: a throttled invocation is counted as **neither an `Invocation` nor an error** — it sits in its own metric entirely. So if you want to alarm on capacity pressure you must alarm on `InvocationThrottles` specifically; it will not show up in `Invocations` (the request never succeeded) and it will not show up in either error metric. A related capacity-trending metric is `EstimatedTPMQuotaUsage`, which surfaces how close you are running to your tokens-per-minute quota — but AWS documents this explicitly as an **approximation**, so treat it as a headroom-trending signal for "are we getting close to the ceiling," not as an exact quota ledger. Finally, the model-invocation-logging subsystem publishes its own *delivery* metrics — `ModelInvocationLogsCloudWatchDeliverySuccess` / `ModelInvocationLogsCloudWatchDeliveryFailure`, `ModelInvocationLogsS3DeliverySuccess` / `ModelInvocationLogsS3DeliveryFailure`, and `ModelInvocationLargeDataS3DeliverySuccess` / `ModelInvocationLargeDataS3DeliveryFailure` — which let you alarm on the logging pipeline itself, so that a silent log-delivery failure does not leave you blind.

Two flags belong on this material. First, `TimeToFirstToken` and `EstimatedTPMQuotaUsage` are comparatively newer metrics, and the AWS documentation does not enumerate the per-Region availability of every individual `AWS/Bedrock` metric — so do not assert that a given metric is present in a given Region without verifying it live; confirm availability in the CloudWatch console for the Region you operate in. Second, `EstimatedTPMQuotaUsage` is an approximation by AWS's own description and must not be presented as an exact figure.

### Model Invocation Logging operated at scale

CloudWatch metrics tell you *how much* and *how fast*; they do not tell you *what was actually said*. For the full request and response payloads you need Model Invocation Logging, which captures the complete request data, response data, and invocation metadata for `Converse`, `ConverseStream`, `InvokeModel`, and `InvokeModelWithResponseStream` calls. In the standing program, Model Invocation Logging is not a switch you flip during an incident — it runs **continuously, at scale, for the life of the workload**, producing the durable record that three different standing functions draw on: token-usage analysis and cost attribution (every logged call carries its token counts and `ModelId`), quality signals (the actual prompts and completions are what a scheduled evaluation re-scores, and what a human reviewer audits), and an operational audit trail of exactly what the model was asked and what it produced. This is the standing-source use; Guide 06's use of the very same logs to crack open a *single* misbehaving response is the diagnostic use, and this section points there for that depth rather than repeating it.

Operating it at scale means understanding its configuration and its destinations. Model Invocation Logging is **disabled by default** — a deliberate choice, since the logs contain full prompt and response content — and is enabled either from the Bedrock console or programmatically through the `PutModelInvocationLoggingConfiguration` API, after you have provisioned a destination. The destinations are Amazon S3, Amazon CloudWatch Logs, or both, in the same account and Region as the invocations. Enabling S3 requires a bucket policy granting the `bedrock.amazonaws.com` service principal `s3:PutObject`; enabling CloudWatch Logs requires a log group and an IAM role Bedrock can assume to write to it. You can also toggle which data types are logged — text, image, embedding, video, and audio — so a workload can capture text payloads while excluding large binary modalities. One scale detail matters for destination choice: CloudWatch Logs has a payload-size limit, so when a request or response is too large to fit, Bedrock writes that large payload to S3 instead — which is exactly why the separate `ModelInvocationLargeDataS3Delivery*` delivery metrics exist. The practical consequence is that for a high-volume workload with large prompts or multimodal payloads, S3 is the destination that reliably captures everything, while CloudWatch Logs is convenient for searchable, near-real-time text logging within its size limits — and most production programs enable both.

### Dashboards, alarms, and anomaly detection

The metrics and logs are raw material; dashboards, alarms, and anomaly detection are what turn them into an always-on watch. A CloudWatch dashboard is the at-a-glance health view — a single pane that graphs invocation volume, the token-count metrics, the two latency metrics, and the three error/throttle metrics side by side so an operator can see the workload's state without running a query. Alarms are the active layer: a CloudWatch static-threshold alarm watches one metric (or a metric-math expression) against a fixed comparison threshold over a defined number of evaluation periods, transitions to the `ALARM` state when the metric breaches that threshold for the configured periods, and notifies an Amazon SNS topic (which fans out to email, chat, a ticketing webhook, or an automated remediation function). The standing program needs, at minimum, one alarm on each of the three core signals, and each alarm must state the condition that fires it:

- **Token-usage / cost signal.** Alarm on the *sum* of `InputTokenCount` and `OutputTokenCount` using a metric-math expression, breaching a tokens-per-period ceiling — for example, fire when combined tokens exceed 5,000,000 in any one-hour period for one period. Because tokens are the cost driver, this is your earliest warning that spend is about to rise, well ahead of the billing data.
- **Latency signal.** Alarm on `InvocationLatency` at a high percentile against a millisecond target — for example, fire when the p99 of `InvocationLatency` stays above 4,000 ms for three consecutive five-minute periods — or, for a streaming surface, alarm on the p90 of `TimeToFirstToken` above a responsiveness target. The percentile and the sustained-periods condition keep a single slow outlier from paging anyone while still catching a real regression.
- **Error / throttle signal.** Alarm on `InvocationThrottles` `SampleCount` above a count-per-period — for example, fire when throttles exceed 10 in any five-minute period — or alarm on an error *rate* with metric math, such as `InvocationServerErrors ÷ Invocations` exceeding 1% for two consecutive periods. (The example threshold numbers here are illustrative starting points; tune them to each workload's baseline.)

Static-threshold alarms are perfect when you know the line — a hard latency SLO, a budgeted token ceiling, an unacceptable error rate. They are *bad* at a different job: catching the unexpected spike in a metric whose "normal" value cycles predictably. A workload whose token volume triples every weekday morning and falls to near zero overnight cannot be guarded by a single static line — set it high enough to avoid false alarms at the daily peak and it will miss a genuine anomaly during a trough; set it low enough to catch the trough anomaly and it will page you every morning at peak. This is what anomaly detection solves, and AWS offers **two distinct services**, both built on a *learned dynamic baseline* rather than a fixed threshold.

The first is **CloudWatch anomaly detection**, which operates on *metrics*. It applies machine learning to a metric's own history to build a prediction band — a model of expected values that accounts for hourly, daily, and weekly seasonality — and you create an anomaly-detection alarm that fires when the metric goes above, below, or outside that band. Because the band rises and falls with the metric's normal rhythm, it catches the weekday-trough spike *and* stays quiet at the expected morning peak. The cost of this adaptability is training time: a newly created anomaly detector may take up to two weeks of history to train a stable band. Use CloudWatch anomaly detection for the operational `AWS/Bedrock` metrics — latency, token volume, error spikes — where "normal" is a moving target.

The second is **AWS Cost Anomaly Detection**, which operates on *spend*. It applies machine learning to your billing data to detect anomalous cost, evaluating roughly three times a day against net unblended cost, with up to about a 24-hour detection delay (so it is decidedly not real-time). You configure monitors segmented by AWS service, by account, by cost allocation tag, or by cost category, and when it detects an anomaly it ranks the contributing root causes by dollar impact and alerts you by email or SNS. It is generally available and free to use. The two services are complementary, not redundant: CloudWatch anomaly detection watches operational metrics in near-real-time; Cost Anomaly Detection watches the bill a few times a day. Both beat a static threshold for the same reason — they alarm on *deviation from a learned normal*, not on a line a human guessed in advance.

There is one verified nuance about AWS Cost Anomaly Detection that is high-value on the exam and must be stated as fact: **it does not monitor third-party AWS Marketplace products.** This explicitly includes third-party large language models offered through Amazon Bedrock — for instance, Anthropic Claude charges, which are billed under the legal entity "Anthropic, PBC" — as well as models obtained through Amazon Bedrock Marketplace. If a meaningful share of your FM spend is a third-party model, Cost Anomaly Detection will not catch an anomaly in those charges. The documented alternative for alerting on that spend is **AWS Budgets**: create a cost budget that filters on the relevant Billing entity (the third-party legal entity) so you receive an alert when those charges cross a threshold. Reach for Budgets here precisely because the anomaly service has a blind spot for Marketplace charges.

### Recurring golden-dataset re-runs for drift detection

Metrics and alarms watch *how the system runs*; they say nothing about *whether the answers are still good*. Quality has its own slow failure mode — drift — where output accuracy degrades gradually as the underlying model version changes, as the data distribution the workload sees shifts, or as prompt and retrieval changes accumulate. No latency or error metric will catch a model that is still fast and still returns 200s but has quietly started hallucinating more on a class of questions. The standing-program answer is the recurring golden-dataset re-run: on a schedule, re-run a curated, ground-truth-labeled evaluation set (the "golden dataset") through the live workload and score the results, so that a drop in quality scores surfaces on its own — the same way a cost anomaly or a latency breach surfaces — rather than waiting for a user to complain. This is the explicit hand-off that Guide 06 deferred to "future Guide 07": Guide 06 defined what a golden dataset is and used it *diagnostically* to investigate a quality complaint; this guide receives that hand-off and operates the golden dataset *continuously* as a drift sentinel. For the definition of a golden dataset and its diagnostic use, see Guide 06 — this section does not restate them.

The scoring engine for these re-runs is a real, named AWS feature: **Amazon Bedrock Evaluations**. It supports automatic metrics (including BERT Score, F1, semantic robustness, and retrieval-correctness measures), human evaluation workflows, and an LLM-as-a-judge mode, and it scores outputs across dimensions that include correctness, completeness, hallucination detection, and harmfulness. For RAG workloads, Bedrock Evaluations specifically measures retrieval relevance and generation quality against ground truth — which makes it the natural instrument for re-scoring a golden set on a schedule. The one thing to flag clearly: the *recurrence* — running these evaluations automatically on a cadence to detect drift — is an operational **pattern**, not a single one-click named feature. You assemble it by pairing Bedrock Evaluations with a scheduler (such as an Amazon EventBridge scheduled rule that launches the evaluation job and routes the results to a dashboard or an alarm). Treat "scheduled golden-dataset drift re-runs" as this pattern, not as a built-in Bedrock toggle.

### Cost monitoring instrumentation

The cost-instrumentation path is its own suite of four tools, and the exam reliably tests whether you know that each answers a *different* question about spend. They are not interchangeable, and the cleanest way to hold them is by the question each one answers.

| Tool | The distinct question it answers | How it answers it |
|---|---|---|
| AWS Cost Explorer | Where did spend go, what drives it, and what is the trend or forecast? | Visualizes and analyzes cost and usage over time; filters and groups by service, Region, and tags; projects a forecast |
| AWS Budgets | Am I about to exceed a planned limit? | Custom cost or usage budgets that alert (up to five alerts per budget) when actual or forecasted spend crosses a threshold; the documented tool for third-party-LLM and Bedrock Marketplace spend alerts |
| Cost allocation tags | Which team, application, or project owns this spend? | User-defined and AWS-generated tags activated in the Billing console to attribute spend; not retroactive; managed only by the management or standalone account |
| AWS Cost Anomaly Detection | Has spend deviated unexpectedly from its learned normal pattern? | ML monitors (segmented by service, account, tag, or cost category) run ~3x/day on net unblended cost and rank root causes by dollar impact; does not cover third-party Marketplace products |

Cost Explorer is the *analytical* tool — you go to it to understand history and trend, to answer "what is driving the bill and where is it heading." Budgets is the *guardrail* — proactive, threshold-based, the answer to "warn me before I overshoot a planned number," and (per the section above) the right tool for third-party-LLM spend that Cost Anomaly Detection cannot see. Cost allocation tags are the *attribution* mechanism — without them, a shared Bedrock bill is an undifferentiated lump; with them, you can split FM spend across the teams, applications, and environments that incurred it, which is what makes chargeback and per-app accountability possible. (Remember that tags are not retroactive — activate them before the spend you want to attribute, not after.) Cost Anomaly Detection is the *spend sentinel* — the learned-baseline watch on the bill itself.

These tools are not just for reporting — they close the loop back to the Cost Lever Priority. Monitoring is what tells you *which lever to pull*. Cost allocation tags plus Cost Explorer reveal *which* application or team is driving the spend; once you know that, the token metrics on its dashboard tell you *why*. If the `InputTokenCount` and `OutputTokenCount` trend is the culprit, the answer is lever 1 (token efficiency) or lever 2 (caching) — trim the prompts, bound the output, cache the repeated prefix. If a single expensive model is the driver and a cheaper one would clear the quality bar, the answer is lever 3 (model selection). If the spend reflects genuinely high, steady, predictable volume on an expensive on-demand path, the answer is lever 4 (Provisioned Throughput or batch). Without the instrumentation you are guessing which lever to pull; with it, the data points straight at the lever. That is the whole purpose of the standing program: not to admire the dashboards, but to know — continuously and with evidence — exactly where the cost and latency are going and what to do about it.

### Knowledge Check

**Q1:** An operations team wants to be alerted when their Bedrock workload's token consumption deviates from its normal pattern. Token volume triples every weekday morning and drops to near zero overnight. They set a single static CloudWatch alarm on the token-count sum. What is the problem, and what fits better?
- A) Nothing wrong — a static threshold is the standard approach for any metric
- B) A single static line cannot fit a metric with daily seasonality; use CloudWatch anomaly detection, which learns a dynamic baseline and alarms on deviation from it
- C) Switch to AWS Cost Anomaly Detection, which watches the token metric in real time
- D) Disable the alarm and review the dashboard manually each morning

**A: B** — A metric whose normal value cycles predictably cannot be guarded by one fixed line: set it high enough to avoid false alarms at the morning peak and it misses an overnight-trough anomaly; set it low enough to catch the trough and it pages every morning at peak. CloudWatch anomaly detection applies ML to the metric's own history to build a prediction band that accounts for hourly/daily/weekly seasonality and alarms when the metric falls outside the band. C is wrong because AWS Cost Anomaly Detection operates on *spend* (billing data, ~3x/day), not on CloudWatch operational metrics, and is not real-time. D abandons the automated watch the standing program is built to provide.

**Q2:** True or False — To monitor all failures of a Bedrock workload, you should alarm on the single `AWS/Bedrock` error metric, which includes throttled requests.

**A: False — on both counts.** There is no single aggregate error metric in the `AWS/Bedrock` namespace. Failures are split across three distinct metrics: `InvocationClientErrors` (request-side 4xx problems), `InvocationServerErrors` (service-side 5xx problems), and `InvocationThrottles` (quota-exceeded rejections). And throttles are *not* counted as errors — a throttled invocation is counted as neither an `Invocation` nor an error; it lives only in `InvocationThrottles`. To watch capacity pressure you must alarm on the throttle metric specifically, and to watch a true error rate you build a metric-math ratio over the error metrics and `Invocations`.

**Q3:** Scenario — A company's Bedrock bill is dominated by Anthropic Claude usage, which appears on the invoice under the legal entity "Anthropic, PBC." They enable AWS Cost Anomaly Detection and assume they are now covered for spend spikes on that model. Are they? What should they do?

**A: No.** AWS Cost Anomaly Detection does not monitor third-party AWS Marketplace products, and that exclusion explicitly covers third-party LLMs on Bedrock such as Anthropic Claude (billed under "Anthropic, PBC") and Bedrock Marketplace models. The anomaly service has a blind spot for exactly the charges that dominate their bill. The documented fix is AWS Budgets: create a cost budget filtered on the relevant Billing entity (the third-party legal entity) so an alert fires when those charges cross a threshold. Cost Anomaly Detection is correct for first-party AWS spend; Budgets is the tool for the third-party-Marketplace spend it cannot see.

**Q4:** What went wrong? — A team enabled Model Invocation Logging to Amazon CloudWatch Logs only. Their text prompts log fine, but they later find that some large multimodal requests never appeared in the logs, and they were never alerted to the gap. What happened, and how should the standing program be configured?

**A:** CloudWatch Logs has a payload-size limit, so when a request or response is too large to fit (typical of large multimodal payloads), Bedrock writes that large payload to Amazon S3 instead — but if no S3 destination is configured, that large data has nowhere to land and is effectively missing from the logs. The fix is to enable an S3 destination (most production programs enable both S3 and CloudWatch Logs, with S3 reliably capturing everything and CloudWatch Logs giving searchable near-real-time text). The "never alerted" gap is also avoidable: the logging subsystem publishes delivery metrics — including `ModelInvocationLargeDataS3DeliverySuccess` / `...Failure` — so the standing program should alarm on log-delivery failures rather than discovering the blind spot after the fact.

**Q5:** Fill in the number / name — A single AWS Budget supports up to ___ alerts. The AWS feature that scores golden-dataset re-runs (automatic metrics, human evaluation, and LLM-as-a-judge) is named ___, but making those re-runs recurring for drift detection is achieved by pairing it with a ___.

**A:** Up to **five** alerts per budget. The scoring feature is **Amazon Bedrock Evaluations** (it supports automatic metrics such as BERT Score and F1, human evaluation workflows, and LLM-as-a-judge, scoring correctness, completeness, hallucination, and harmfulness, and measuring retrieval relevance and generation quality for RAG). Making it recurring is an operational *pattern*, not a one-click feature — you pair Bedrock Evaluations with a **scheduler** (such as an Amazon EventBridge scheduled rule) that launches the evaluation on a cadence and routes the scores to a dashboard or alarm. The recurrence-as-pattern point is the part to remember: there is no single "scheduled drift re-run" toggle.

### Exam Pattern 4 — Optimize cost / latency for an FM workload

Pattern 4 lives in Domain 4 (Operational Efficiency and Optimization), and almost every cost or performance question you meet is a variation on it. The reason it deserves a dedicated dissection is that the wrong answer is not random — it is the answer your traditional-infrastructure instincts will hand you first, and the question is built to reward you for overriding that instinct. Treat the three parts below as a single reflex: the signature tells you you are in Pattern 4, the answer shape tells you where to go, and the trap tells you what to walk past on the way.

#### Part (a) — The scenario signature

Recognizing Pattern 4 is half the battle, because once you have named it the answer space collapses to the Cost Lever Priority. The signature is a question that describes a foundation-model workload on Amazon Bedrock and asks you to reduce cost, reduce latency, or raise throughput. The surface phrasing varies, but the tells are consistent. On the cost side you will see "the Bedrock bill is rising month over month," "reduce inference cost," "the model is expensive," or the subtler "costs tripled but traffic only doubled" — spend creeping with no single spike to point at. On the latency side: "the app sits on a blank screen before responding," "reduce latency for an interactive assistant," or "responses take too long to complete." On the throughput side: "the high-volume workload keeps getting throttled" or "needs more throughput." There is one more tell that is almost diagnostic by itself: the presence of a compute-scaling option among the answers — "use a bigger instance," "enable compute autoscaling," "add nodes or GPUs." Because the default on-demand Bedrock path is serverless, that option exists in the answer set for one reason only, which is to be the distractor. Seeing it is itself a signal that you are in Pattern 4.

#### Part (b) — The canonical answer shape

The canonical answer is the Cost Lever Priority from Section 1, applied in order, cheapest win first, escalating to a higher-commitment lever only when the cheaper one is genuinely exhausted. The ordering is the answer to "where do I start," and on the exam the earlier lever is almost always the better answer unless the scenario explicitly rules it out.

The first lever is token efficiency (Section 2): compress the prompt, prune low-value context, summarize or truncate conversation history, and bound the output with a concise instruction and a `maxTokens` ceiling. It is free, reversible, requires no infrastructure commitment and no model change, and it applies to every single call — which is exactly why it sits first. The second lever is caching (Section 3), and it splits into two mechanisms the exam expects you to keep separate: Bedrock prompt caching reuses a repeated static prefix so it is neither reprocessed nor re-billed at full rate, while the model is still invoked; semantic caching returns a stored answer when a new query is close enough in meaning by embedding similarity, so the model is not invoked at all on a hit. Both require that content actually repeats — the precondition that makes lever 2 a no-op on a stream of unique prompts. The third lever is model selection (Section 4): route by difficulty with tiering (decide once, up front, by predicted difficulty, run one model) or cascading (try the cheap model first, escalate on a failed quality check), use Amazon Bedrock Intelligent Prompt Routing for managed routing within a single model family, and use latency-optimized inference to serve the same model faster. The hard constraint on lever 3 is the quality bar (the GENCOST01 well-architected guidance): a cheaper model is only a saving if it still clears the task's quality acceptance criteria. The fourth and highest-commitment lever is Provisioned Throughput and batching (Section 5): Provisioned Throughput buys dedicated Model Units for steady, high, sustained real-time volume (and is required to serve custom models), while batch inference processes a large bounded set asynchronously at a documented discount; Cross-Region Inference overlays the on-demand path to add throughput and resilience. You escalate to lever 4 only when the volume is real, steady, and large enough to justify the commitment.

#### Part (c) — The canonical trap

The trap built into Pattern 4 is to scale compute — "use a larger instance type," "enable compute autoscaling," "reserve a GPU" — when the real lever is token efficiency or a smaller model. This is the single most important distractor in Domain 4, and it is tempting precisely because it is the correct answer for almost every *traditional* performance or cost problem. For a Bedrock foundation-model workload on the default on-demand path it is a category error: Bedrock is serverless, there is no instance for you to size, no cluster to scale, and no compute-hour meter to turn down. The lever acts on token volume (generate and ingest fewer tokens) or on billing mode (cache repeated tokens, commit to throughput, or batch), never on machine size. When a Pattern 4 scenario offers a compute-scaling fix, that option is there to be rejected — the right answer touches tokens or billing mode.

### The cost-and-performance decision tree

The Cost Lever Priority is an ordered list, but a live scenario also branches on what kind of problem you have. The tree below renders the whole decision space, covering the four decision points the exam tests: which lever to apply first, whether to stay on-demand or move to Provisioned Throughput or batch, whether to run a single smaller model or route across models, and which latency lever fits the symptom.

**Diagram (described):** The decision tree starts from "FM cost or latency problem" and immediately splits on the top question — is this a cost problem or a latency problem?

On the **cost** branch, the first sub-question is "does content repeat?" If yes, go to caching (prompt or semantic). If no, apply token efficiency first. Both of those then converge on the next question: "can a cheaper model clear the quality bar?" If yes, you reach a model-choice fork — either a single smaller-model swap (when one model fits the whole workload) or routing (tiering, cascading, or Intelligent Prompt Routing) when difficulty varies. If a cheaper model cannot clear the bar, you branch on volume shape: steady high volume goes to Provisioned Throughput, bulk offline work goes to batch inference, and spiky or low volume stays on-demand.

On the **latency** branch, the question is "which clock hurts?" If it is the first token (blank-screen, perceived latency), the answer is response streaming. If it is total time (the whole response is slow), the answer is bounding output with `maxTokens` or using latency-optimized inference. If it is throughput (not enough requests per unit time), the answer is raising concurrency within the quotas.

The four decision points map onto the tree as follows. Decision point one — which lever first — is the cost branch's opening cluster: ask whether content repeats, send repeated content to caching and everything else to token efficiency first, and only then test whether a cheaper model clears the quality bar. Decision point two — on-demand versus Provisioned Throughput versus batch — is the volume-shape node: steady high real-time volume justifies Provisioned Throughput, a large offline set goes to batch, and spiky or low volume stays on-demand. Decision point three — a single smaller model versus tiering, cascading, or Intelligent Prompt Routing (abbreviated IPR in the diagram) — is the one-model-or-route split: a flat swap to a smaller model when one model fits the whole workload, or routing when difficulty varies enough that easy and hard traffic should land on different models. Decision point four — streaming versus latency-optimized inference — is the latency branch: a blank-screen, time-to-first-token complaint is solved by response streaming, a total-time complaint by bounding output with `maxTokens` or by latency-optimized inference, and a throughput ceiling by raising concurrency up to the quota.

### Common exam traps and distractors

Examiners reuse a small set of distractors across Domain 4, and each one is tempting for a specific, predictable reason. The catalog below states each trap as the tempting-but-wrong choice, the correct lever or answer, and why the tempting choice is wrong. The first two are mandatory to know cold — the canonical Pattern 4 trap and the pricing trap — and the rest span the remaining topics so that no part of the domain catches you out.

1. The compute-scaling trap (the canonical Pattern 4 trap, token economics). Tempting: fix an expensive or slow FM workload by scaling up the instance, enabling compute autoscaling, or reserving a GPU. Correct: reduce token volume (token efficiency or a smaller model) or change the billing mode (caching, Provisioned Throughput, batch). Why it is wrong: on-demand Bedrock is serverless — there is no instance to scale, and cost is a function of tokens, not machine size, so the compute-scaling option is a category error no matter how natural it feels.

2. The pricing trap (token economics). Tempting: choose the answer that quotes a precise current per-token figure, or treat a price you have memorized as a present fact. Correct: recognize that prices are fast-moving and must be verified live against the Bedrock pricing page; reason about the pricing *model* (input versus output, the output-priced-higher asymmetry) rather than any specific number. Why it is wrong: a memorized dollar figure is not a current fact, and a question that hinges on one is testing whether you know prices must be looked up rather than recalled.

3. Caching applied to non-repeating work (caching). Tempting: "turn on caching" to fix any expensive workload, including a stream of unique prompts. Correct: caching only pays off when content repeats — prompt caching for a repeated prefix, semantic caching for repeated query meaning, neither for genuinely unique inputs. Why it is wrong: with no repetition there is no prefix to reuse and no prior query to match, so the cache never hits and the lever does nothing.

4. Confusing the two kinds of caching (caching). Tempting: assume both prompt caching and semantic caching skip the model call, or that both merely make the call cheaper. Correct: semantic caching avoids the invocation entirely on a hit, whereas prompt caching reduces the cost of an invocation that still happens. Why it is wrong: the two intervene at different points — prefix reuse within a call versus elimination of the call — and exam answers exploit the conflation.

5. Ignoring the quality bar on a model downgrade (model selection / routing). Tempting: "switch to the cheaper model to cut cost," with quality unverified. Correct: downgrade only if the smaller model still clears the task's quality acceptance criteria. Why it is wrong: a cheaper model that fails the bar ships wrong answers cheaply — that is a regression, not a saving, and the well-architected guidance makes the quality bar a precondition of the lever.

6. Provisioned Throughput for spiky or low traffic, and batch for interactive work (inference options). Tempting: buy Provisioned Throughput to "guarantee throughput" for a low or bursty workload, or run a real-time, tool-calling, or strict-JSON request as a batch job to get the discount. Correct: use on-demand for variable traffic and reserve Provisioned Throughput for steady high sustained volume (or custom models); use batch only for large, bounded, offline sets, never for real-time or tool-calling work. Why it is wrong: the Provisioned Throughput meter runs whether or not you use the capacity, so idle commitment costs more than on-demand; and batch is asynchronous and processes records independently, so it supports neither real-time response nor tool calling.

7. The wrong latency lever (performance). Tempting: "lower the temperature" or "enable streaming" to reduce total processing time. Correct: streaming reduces *perceived* latency (time to first token) only; total latency is cut by bounding output with `maxTokens` or by a faster/latency-optimized model; temperature, top-k, and top-p do not affect latency or cost at all. Why it is wrong: streaming changes delivery, not generation time, and the sampling parameters shape randomness, not speed — so both are answers to a different question.

8. The single error metric, and a static alarm on a seasonal metric (monitoring). Tempting: alarm on "the" Bedrock error metric (assuming it includes throttles), or set one static threshold on a token metric that triples every morning. Correct: there is no aggregate error metric — `InvocationClientErrors`, `InvocationServerErrors`, and `InvocationThrottles` are distinct, and a throttle is counted as neither an invocation nor an error — so capacity pressure must be watched on the throttle metric directly; and a metric with daily seasonality belongs under CloudWatch anomaly detection (a learned dynamic baseline), not a fixed line. Why it is wrong: a generic error alarm silently misses throttling, and a static line on a seasonal metric either false-alarms at peak or misses a trough anomaly.

9. Cost Anomaly Detection for third-party LLM spend (monitoring). Tempting: rely on AWS Cost Anomaly Detection to catch a spike in Anthropic Claude (billed as "Anthropic, PBC") or a Bedrock Marketplace model. Correct: use AWS Budgets with a Billing-entity filter for that spend. Why it is wrong: AWS Cost Anomaly Detection does not monitor third-party AWS Marketplace products, so it has a blind spot for exactly the charges that may dominate a Bedrock bill.

### Exam-Relevant Distinctions

| If the exam says... | The answer is... | Why |
|---|---|---|
| "FM workload is expensive or slow — scale the compute" | Trap — touch tokens or billing mode instead | Serverless on-demand Bedrock has no instance to scale; cost is driven by token volume, not machine size |
| "Where do I start to cut FM cost?" | Token efficiency first, then caching, then model, then commitment | The Cost Lever Priority is cheapest-win-first; the earlier lever is the better answer unless the scenario rules it out |
| "Avoid calling the model for repeated questions" vs "stop re-billing a repeated prefix" | Semantic caching vs prompt caching | Semantic caching skips the invocation on a hit; prompt caching cheapens an invocation that still occurs |
| "Decide the model up front" vs "escalate only on failure" | Tiering vs cascading | Tiering runs one model chosen by predicted difficulty; cascading may run two in sequence, adding latency on escalation |
| "Steady high real-time volume" vs "large offline set" | Provisioned Throughput vs batch inference | Provisioned Throughput is committed real-time capacity; batch is asynchronous bulk at a discount |
| "Blank screen before output" vs "whole response is slow" | Streaming vs `maxTokens` / latency-optimized inference | Streaming fixes perceived latency only; total latency is set by output-token count and serving path |
| "Alert on spend that cycles daily" vs "third-party LLM spend" | Anomaly detection vs AWS Budgets | Anomaly detection learns a seasonal baseline; Budgets covers third-party Marketplace spend the anomaly service excludes |

- The presence of a "bigger instance / add compute" option in a Pattern 4 question is itself a tell that it is the distractor.
- Caching is a no-op without repetition — a stream of unique prompts has nothing to reuse.
- A model downgrade is only a saving if the smaller model clears the quality bar; otherwise it is a regression.
- `maxTokens` is simultaneously a cost lever (Section 1) and a total-latency lever (Section 6); temperature, top-k, and top-p are neither.
- Any exact per-token dollar figure is fast-moving — reason about the pricing model and verify the number live.

### Knowledge Check

**Q1:** A team's Amazon Bedrock bill has been rising for three months while traffic grew only modestly. They ask which change to make first. Which answer is correct, and which is the engineered trap?
- A) Move the workload to larger, autoscaled compute instances to handle the cost
- B) Start with token efficiency — compress prompts, prune context, and bound output with `maxTokens` — before any higher-commitment lever
- C) Immediately purchase Provisioned Throughput to lock in capacity
- D) Lower the temperature parameter to reduce cost

**A: B** — The Cost Lever Priority is cheapest-win-first, and token efficiency is lever 1: free, reversible, no commitment, applied to every call. A is the canonical Pattern 4 trap — on-demand Bedrock is serverless, so there is no instance to scale and cost is a function of token volume, not machine size. C jumps straight to the highest-commitment lever before the cheaper ones are exhausted, and idle committed capacity costs more than on-demand. D is wrong because temperature shapes output randomness, not cost.

**Q2:** True or False — Both Bedrock prompt caching and semantic caching avoid invoking the model on a cache hit.

**A: False.** Only semantic caching avoids the invocation entirely — on a hit it returns a stored answer for a semantically similar query without calling the model. Prompt caching does still invoke the model; what it avoids is reprocessing and full re-billing of a repeated static prefix within that invocation. The two intervene at different points — whole-call elimination versus prefix reuse — and conflating them is a common distractor. Both also share one precondition: content must actually repeat, or neither helps.

**Q3:** Scenario — An interactive assistant "sits on a blank screen for several seconds before any text appears," but once text starts it completes quickly. A teammate proposes lowering the temperature to speed it up. What is the right fix, and why is the proposal wrong?

**A:** Enable response streaming. The complaint is about perceived latency — time to first token — and streaming surfaces tokens as they are generated, so the user sees output immediately instead of waiting for the whole response. Lowering the temperature does nothing here: temperature, top-k, and top-p shape output randomness, not latency or cost. (Had the complaint been that the *whole* response takes too long, the lever would be bounding output with `maxTokens` or using a faster / latency-optimized model — not a sampling parameter either.)

**Q4:** A company's Bedrock spend is dominated by Anthropic Claude, billed under "Anthropic, PBC." They enable AWS Cost Anomaly Detection and consider the spend covered for spikes. Are they covered? What should they use?
- A) Yes — Cost Anomaly Detection monitors all Bedrock spend including third-party models
- B) No — Cost Anomaly Detection excludes third-party AWS Marketplace products; use AWS Budgets with a Billing-entity filter
- C) Yes, but only if they add a cost allocation tag
- D) No — they must use CloudWatch anomaly detection on a billing metric instead

**A: B** — AWS Cost Anomaly Detection does not monitor third-party AWS Marketplace products, and that exclusion explicitly covers third-party LLMs on Bedrock such as Anthropic Claude. The documented alternative for alerting on that spend is AWS Budgets, filtered on the relevant Billing entity. A is the trap. C does not change the exclusion (tags attribute spend, they do not add anomaly coverage). D confuses CloudWatch anomaly detection, which watches operational metrics, with billing-data monitoring.

**Q5:** Match each volume shape to the inference option that fits — (i) steady, high, predictable real-time volume; (ii) 200,000 documents to summarize overnight with no latency requirement; (iii) spiky, low, unpredictable traffic; (iv) a fine-tuned custom model in production.

**A:** (i) Provisioned Throughput — steady high sustained real-time volume justifies committing to dedicated Model Units. (ii) Batch inference — a large bounded offline set tolerates asynchronous turnaround and earns the documented discount. (iii) Stay on-demand — variable traffic does not justify a commitment, and the on-demand meter only charges for what you use. (iv) Provisioned Throughput — required, because custom (fine-tuned) models cannot be served on the on-demand path. The trap to avoid is buying Provisioned Throughput for case (iii) or running case (ii)-style real-time/tool-calling work as a batch job.

### Multiple-Response Knowledge Check

The AIP-C01 exam leans heavily on multiple-response items (choose two or more correct options from five or more). The single-answer checks above build recall; these four train the harder skill the exam actually measures — holding several true-or-false judgments at once and rejecting the engineered distractors. Work each before revealing the answer.

**Q1:** A team's on-demand Amazon Bedrock bill for a customer-support assistant is rising fast. Each call carries a long static system prompt plus a product-manual PDF, conversations run many turns, and responses are verbose. They want the cheapest, lowest-commitment wins first, with no model change yet. Which TWO actions fit that brief? (Select TWO)
- A) Bound output with a `maxTokens` ceiling and a concise-response instruction
- B) Move the workload onto larger, autoscaled compute instances
- C) Enable Bedrock prompt caching on the repeated static prefix (system prompt + manual)
- D) Purchase Provisioned Throughput to lock in dedicated Model Units
- E) Lower the temperature parameter to reduce per-call cost

**A: A, C** — The brief asks for the cheapest, lowest-commitment levers with no model swap, which points at lever 1 (token efficiency) and lever 2 (caching). A is output-length control — output tokens are the priciest dimension, so a `maxTokens` ceiling plus a concise instruction is a free, reversible, every-call win. C is prompt caching: the system prompt and manual are a repeated static prefix, exactly what prompt caching targets, and it is config-level and infrastructure-light without changing the model. B is the canonical Pattern 4 compute-scaling trap — on-demand Bedrock is serverless, so there is no instance to scale and cost is driven by token volume, not machine size. D is lever 4, the *highest*-commitment lever, evaluated last and only for steady high volume — it violates the "lowest-commitment first" brief. E is a distractor: temperature shapes output randomness, not cost or latency.

**Q2:** An engineer is choosing between Amazon Bedrock prompt caching and a customer-built semantic cache for two different workloads. Which TWO statements correctly capture how the two mechanisms differ? (Select TWO)
- A) On a semantic-cache hit the foundation model is not invoked at all; on a prompt-cache hit the model is still invoked and only the cached prefix is skipped
- B) Both mechanisms eliminate the model invocation on a hit and differ only in what they match on
- C) Semantic caching carries a genuine correctness risk (a similar-but-not-equivalent query can be served a stale or wrong answer), while prompt caching is low correctness risk because the model still generates a fresh response
- D) Prompt caching is a customer-built pattern on ElastiCache or MemoryDB, whereas semantic caching is a single managed Bedrock toggle
- E) Lowering the semantic-cache similarity threshold reduces the correctness risk while raising the hit rate

**A: A, C** — A is the cleanest one-line separator in the section: semantic caching makes the invocation disappear (the stored response is returned, the model is never called), whereas prompt caching cheapens an invocation that still occurs (the static prefix is skipped but generation still runs). C captures the asymmetric risk profile: because semantic caching returns a stored answer matched by similarity, a query similar in wording but different in intent can be served a wrong or stale answer, while prompt caching always generates fresh output so a miss merely costs the savings. B is false — it conflates the two, the exact distractor the exam exploits. D inverts reality: semantic caching is the customer-built pattern (ElastiCache for Valkey, MemoryDB), and prompt caching is the managed Bedrock feature. E is doubly wrong: lowering the threshold *raises* the hit rate but *increases* the correctness risk, not reduces it.

**Q3:** A workload sends a real-time, tool-calling agent request that must return strict JSON, and separately needs to summarize 200,000 archived documents overnight with no latency requirement. A teammate proposes running both through batch inference to capture the documented discount. Which TWO statements are correct about inference-option selection here? (Select TWO)
- A) Batch inference is appropriate for the overnight 200,000-document summarization because it is a large, bounded, latency-tolerant offline set
- B) The real-time tool-calling strict-JSON request should not run as a batch job, because batch is asynchronous and processes records independently
- C) Batch inference is the right choice for both, since the discount applies to any high-volume workload
- D) The real-time agent request should move to Provisioned Throughput only because batch is cheaper
- E) Provisioned Throughput would be the correct lever for the overnight batch job to guarantee throughput

**A: A, B** — A correctly matches batch inference to its use case: a large, bounded, offline set with no real-time requirement, processed asynchronously through Amazon S3 at a documented discount. B correctly rejects batch for the real-time path — batch is asynchronous and processes records independently, so it supports neither real-time responsiveness nor tool calling, which is exactly trap 6 in the catalog. C is the proposal's error: the discount does not make batch suitable for interactive or tool-calling work. D misstates the reason and the lever — the real-time request's binding constraint is latency/tool-calling, not relative price, and nothing in the scenario establishes the steady high sustained volume that justifies Provisioned Throughput. E is wrong because the overnight job is a bounded offline set that fits batch, not a steady real-time stream that would justify a Provisioned Throughput commitment.

**Q4:** An operations team is setting up the standing monitoring program for a Bedrock workload. Token spend roughly triples every weekday morning and falls overnight, most of the spend is Anthropic Claude (billed as "Anthropic, PBC"), and the team also wants to catch capacity pressure. Which TWO choices are correct? (Select TWO)
- A) Use CloudWatch anomaly detection (a learned dynamic baseline) for the token metric that cycles daily, rather than a single static threshold
- B) Rely on AWS Cost Anomaly Detection to catch spikes in the Anthropic Claude spend
- C) Use AWS Budgets with a Billing-entity filter to alert on the third-party Anthropic Claude spend
- D) Alarm on the single aggregate Bedrock error metric, since it includes throttles
- E) A single static threshold alarm on the daily-cycling token metric is the most reliable approach

**A: A, C** — A is correct because a metric with strong daily seasonality belongs under CloudWatch anomaly detection, which learns a dynamic baseline that follows the morning peak and overnight trough; a static line would false-alarm at peak or miss a trough anomaly. C is correct because AWS Cost Anomaly Detection does *not* monitor third-party AWS Marketplace products — including third-party LLMs like Anthropic Claude — so the documented path for that spend is AWS Budgets filtered on the Billing entity. B is exactly that blind-spot trap (trap 9): Cost Anomaly Detection cannot see the Marketplace spend that dominates this bill. D is wrong on two counts — there is no aggregate Bedrock error metric (`InvocationClientErrors`, `InvocationServerErrors`, and `InvocationThrottles` are distinct, and a throttle is counted as neither an invocation nor an error), so capacity pressure must be watched on the throttle metric directly. E directly contradicts A — a static threshold on a seasonal metric is the failure mode anomaly detection exists to fix.
