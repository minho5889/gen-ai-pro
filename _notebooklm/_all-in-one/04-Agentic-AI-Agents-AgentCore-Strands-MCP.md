## Section 1: Agentic AI Foundations

### What Makes an Agent an Agent

Everything you have built so far in this series has been a single shot at the model. In Guide 01 you sent a prompt to a foundation model through the Converse API and read back a completion; in Guide 02 you wrapped that call in a retrieval step so the model answered from your own data. Both are powerful, and both share one structural property: the application decides, in code, exactly what happens and in what order. The model is consulted once (or a fixed number of times), and the control flow around it is written by you. An agent breaks that assumption. An agent hands the control flow itself to the model.

The cleanest working definition, and the one the exam rewards, is this: an agent is a foundation model placed inside a reasoning loop, given a set of tools it can call and a memory it can read from and write to, and pointed at a goal rather than a single question. The foundation model is the cognitive core — it reasons about what to do next. The loop is what lets it take more than one step. The tools are how it acts on the world beyond generating text: querying a database, calling an API, running code, searching a Knowledge Base. The memory is what lets it carry context forward, both within a single task and, for some agents, across separate sessions. Strip any one of these away and you have something simpler. A model with no loop and no tools is a single invocation. A model with retrieval bolted on the front but still no loop is a RAG pipeline. It is the combination — model plus loop plus tools plus memory, aimed at a goal — that earns the name agent.

The word itself comes from agency, the capacity to act independently toward an objective. AWS Prescriptive Guidance frames agentic systems around three pillars: autonomy (the system pursues a goal without step-by-step human direction), asynchronicity (it can run as a longer background process rather than a single blocking request), and agency. Of the three, agency is the defining differentiator — it is what separates a true agent from a clever script. Agency means goal-directed behavior, decision-making about which action to take, the ability to act on a delegated intent rather than an explicit instruction, and contextual reasoning that adapts the plan as new information arrives. A useful mental decomposition that AWS uses for a traditional agent is perceive, reason, act: the agent perceives its environment and the current state, the reasoning module (its cognitive core, drawing on short- and long-term memory, a knowledge base, and its goals) decides what to do, and the act module carries that decision out through APIs, commands, or tools.

A critical framing to carry through this entire guide: agency is a spectrum, not a switch. You do not flip a workload from non-agentic to agentic. You grant a system more or less latitude to decide for itself, and the central engineering discipline of agentic AI — the thing Sections 7 and 9 return to repeatedly — is to grant only as much agency as the task genuinely requires. More autonomy is not better; it is more powerful and more dangerous, and the right amount is the smallest amount that solves the problem.

### Single Prompt, RAG Pipeline, and Agent — Three Points on a Spectrum

Because the exam loves to offer an agent as the answer to a problem that a single prompt or a RAG pipeline would solve more cheaply and more predictably, you have to be able to place all three on the same axis and articulate exactly what changes as you move along it.

A single prompt is one-shot inference. The application sends a request, the model returns a completion, and the interaction is over. The control flow is trivial and fully deterministic at the orchestration level — there is exactly one model call, and your code decides everything around it. This is the right tool when the task is a self-contained transformation of the input: summarize this text, classify this ticket, rewrite this paragraph in a friendlier tone. Nothing about the task requires the model to fetch external information or take a sequence of dependent actions.

A RAG pipeline adds a retrieval step in front of the model, but the flow is still fixed: retrieve relevant context, then generate an answer grounded in that context. Retrieve-then-generate is a straight line. The application, not the model, decides to retrieve, decides what to retrieve against, and decides to generate exactly once afterward. RAG is the right tool when the task is fundamentally a question-answering or grounded-generation problem — the model needs facts it does not carry in its weights, but once it has them, a single grounded generation answers the question. Guide 02 built this pattern in depth with Amazon Bedrock Knowledge Bases.

An agent replaces that fixed flow with a loop in which the model itself chooses what to do at each step. It might decide to call a tool, observe the result, decide the result is insufficient, call a different tool, consult its memory, and only then produce a final answer — or it might answer immediately. The number of steps is not known in advance, because the model determines it at runtime based on what it observes. This is the right tool when the task requires multi-step reasoning, when the sequence of actions cannot be hardcoded because it depends on intermediate results, or when the system must adapt its approach to information it discovers along the way. The defining shift is that an agent puts the foundation model in a loop where it selects tools and iterates, whereas RAG runs a fixed retrieve-then-generate path and a single prompt runs once.

| Dimension | Single prompt | RAG pipeline | Agent |
|---|---|---|---|
| Control flow | One model call | Retrieve then generate - fixed | Model-chosen loop - dynamic |
| Who decides next step | Your code | Your code | The model |
| Steps known in advance | Yes - one | Yes - two | No - runtime |
| External actions | None | Retrieval only | Many tools |
| Best for | Self-contained transform | Grounded Q and A | Multi-step adaptive tasks |
| Predictability | Highest | High | Lowest |
| Cost and latency | Lowest | Low to moderate | Highest |

### The Reasoning Loop: ReAct and Chain-of-Thought

The engine inside an agent is the reasoning loop, and the pattern that AWS documentation uses to describe it is ReAct — short for reason and act. The mechanics are straightforward once you see them as a cycle. The application invokes the model with the goal, the available tools, and the reasoning guidelines (these live in the system prompt). The model reasons about the goal and decides whether it needs a tool. If it does, it emits a tool-use request — the act. The application executes that tool and feeds the result back into the model on the next invocation. The model now perceives that result, reasons again, and either calls another tool or decides it has enough to produce the final answer. Each iteration accumulates context, so the model's understanding compounds as the loop turns. AWS describes this as a cognitive cycle of perceive, reason, act that closes the loop: the output of an action feeds back in and is perceived again on the next pass. That feedback step — taking the result of an action and folding it back into the next round of reasoning — is what the broader practitioner community commonly calls the observe step, giving the popular three-word framing reason-act-observe. AWS documentation predominantly writes it as reason and act (ReAct) and as perceive, reason, act; treat reason-act-observe as the same loop described with the observation step named explicitly.

Underneath ReAct sits chain-of-thought reasoning, and it helps to keep the two distinct. Chain-of-thought prompting is a technique that instructs the model to show its intermediate reasoning steps before committing to a final answer, rather than jumping straight to a conclusion. It is a prompting strategy, not a training method — you elicit it with instructions, either zero-shot ("Let's think step by step") or with a few worked exemplars in the prompt. Its value is that it transforms an opaque, single-shot prediction into an auditable chain of reasoning. ReAct then interleaves that reasoning with acting and observing: the model reasons (chain-of-thought), acts (calls a tool), and observes (reads the result), repeating the cycle. Chain-of-thought is the reasoning substrate; ReAct is the loop that wraps reasoning around real-world actions. When you reach Section 2, you will see this made visible — Amazon Bedrock Agents expose a trace that lets you inspect the agent's chain-of-thought reasoning as it works through the ReAct framework.

A loop needs a way to end, and this is where agentic AI introduces a risk that single prompts and RAG pipelines simply do not have. The agent iterates until a stopping condition is met. The natural stop is that the model decides the goal is satisfied and returns a final answer. But because the model controls the loop, an agent can also fail to stop — reasoning in circles, calling tools repeatedly, burning tokens and latency without converging. So an agent run can terminate for several reasons. The Strands Agents SDK (covered in Section 6) enumerates a concrete set of stop reasons that illustrate the categories well: end turn (the model produced a final response), tool use (the model is requesting a tool, so the loop continues), max tokens (the response was truncated against the token ceiling, which can be an unrecoverable state), stop sequence (a configured sequence was emitted), and content filtered or guardrail intervention (a safety control halted generation); a run can also be cancelled. The end-turn case is the clean finish; the others are guards. The crucial design point — developed fully in Section 7 — is that you must impose bounds on the loop yourself through timeouts, maximum iterations, and circuit breakers, because an unbounded reasoning loop is a cost, latency, and safety liability, not just an inconvenience.

From the reasoning step, a decision branch asks "Tool needed?" If yes, flow moves to "Act — call tool," then to "Observe — tool result," and that observation feeds back into Memory (closing the loop so the next reasoning pass sees it). The tools the model calls in the act step are increasingly connected through the Model Context Protocol, the open standard Guide 02 introduced and Section 5 develops in full. For now, hold the idea that the loop is the reasoning engine, tools are how it reaches outside itself, memory is how it carries state across iterations, and the stopping condition is what keeps the whole thing bounded.

### Agentic Autonomy Versus Deterministic Orchestration

To choose an agent wisely you have to understand what you give up when you adopt one, and the cleanest contrast is against the systems you already know: the traditional request/response service and the deterministic orchestration workflow.

A traditional request/response system — and by extension a hand-coded orchestration like a Step Functions state machine or a chain of Lambda functions — is deterministic. Given the same input, it follows the same code path and produces the same behavior every time. That predictability is a feature: it is easy to test, easy to audit, easy to reason about, and easy to bound in cost and latency because the number of steps is fixed in the code. AWS positions this as the low-autonomy end of a three-way comparison. Traditional automation is tool-centric and narrow, runs in batch or synchronous request/response, carries limited autonomy, and is typically stateless. Software agents sit in the middle — event-driven, rule-bound, holding some state, with medium autonomy. Agentic AI sits at the high-autonomy end — asynchronous, event- and goal-driven, reasoning dynamically with the foundation model, carrying memory and context, and often coordinating multiple agents with a human in the loop.

The trade is autonomy for predictability. An agent's great strength is flexibility: it pursues goals and makes contextual decisions without constant human oversight, adapting its sequence of actions to what it discovers. But that same dynamic interpretation is the cost. Moving from a deterministic system to one that interprets and decides at runtime introduces nondeterministic behavior — the same request can take a different path on a different run. AWS calls this out directly as a new challenge that agentic systems bring: behavior you cannot fully predict in advance. There is a useful related distinction here between generative AI and agentic AI. Generative AI creates content for a human to review and act on; agentic AI autonomously executes tasks within policy-defined boundaries and keeps an audit trail of what it did. The autonomy is what makes it valuable and what makes it require governance.

The cost and latency dimension follows the same curve, and the exam tests the tension directly. Higher autonomy delivers the strongest efficiency, scalability, and speed at scale — an agent that handles a task end-to-end without human hand-offs can be dramatically faster and cheaper per task than a human-led process. But higher autonomy comes with the weakest inherent risk management, because the system is making more decisions on its own. Lower autonomy (a human-led process with the agent only assisting) gives the strongest risk management but the lowest efficiency and scalability and the slowest throughput. There is no free lunch: you are choosing a point on a curve that trades control against speed and scale, and the right point depends entirely on the task's complexity and its tolerance for error.

### When Is an Agent Warranted?

This is the highest-yield decision in the entire guide, because it is the core of exam Pattern 3 ("build an autonomous agent that calls tools") and the source of its most reliable trap. The discipline is to reach for the simplest construct that solves the problem and to escalate to an agent only when the task genuinely demands it.

AWS frames the choice as a three-question decision framework. First, task assessment: is this task actually right for an agent? Second, risk impact assessment: what are the risks if the agent acts wrongly? Third, ROI: is an agent cost-effective here, given that it is the most expensive option per task? The task-assessment question turns on a few factors. Complexity is the big one — if the task requires multi-step reasoning and contextual interpretation, an agent fits; if it is mechanical and rule-based, it does not. Standardization is the mirror image — if the task needs contextual understanding and adapts case by case, an agent fits; if it is so standardized that it needs no adaptation, traditional automation or robotic process automation is the better, cheaper, more predictable choice. Volume and value round out the picture. The core rule to memorize: agentic AI excels at contextual reasoning and adaptive decision-making, while highly standardized, simple, rule-based tasks belong to traditional automation.

Layered on top of task fit is the question of how much autonomy to grant, which AWS expresses as four deployment approaches keyed to risk and error tolerance. Fully autonomous deployment suits low-risk tasks that tolerate a small error rate (roughly one to two percent) — data categorization, document routing, report generation. Human-in-the-loop suits medium-risk tasks needing tighter accuracy (under about half a percent error) — drafting responses, content moderation, claims processing — where the agent does the work but a human approves it. Co-pilot suits high-risk tasks demanding near-zero error where a human makes the final decision and the agent only advises — strategic planning, risk and investment decisions. Human-led with agent support suits critical, zero-tolerance domains — legal, medical diagnosis, regulatory compliance — where the human leads and the agent merely assists. The principle threading through all four is the one from earlier: match the degree of agency to the task's complexity and its cost of being wrong.

There is also a spectrum within agentic design itself, which AWS describes as three patterns. LLM-augmented workflows carry low agency — deterministic code with the model making decisions only at specific, bounded points. Autonomous agents carry high agency — the full ReAct loop, with the model driving the control flow. Hybrids (such as plan-and-solve) sit between, letting the model plan while keeping execution more controlled. Recognizing that "agentic" itself ranges from a tightly bounded workflow to a fully autonomous loop is what lets you pick the lightest design that works.

This brings us squarely to the Pattern 3 trap, which appears in two complementary forms. The first form is choosing more agency than the task warrants — proposing a fully autonomous agent for a job a single prompt, a RAG pipeline, or a deterministic workflow would handle more cheaply and predictably. The second form is the opposite reflex: deciding an agent is appropriate but then hand-rolling the orchestration loop, the tool wiring, the memory, and the safeguards yourself when a managed agent primitive — Amazon Bedrock Agents, the subject of Section 2 — already provides all of that. The exam consistently rewards the candidate who picks the managed primitive over the bespoke build when the managed primitive fits the requirements, and who picks the simpler non-agentic construct when the task does not actually need autonomy at all. Section 9 consolidates this into an explicit decision tree; for now, internalize the two-sided trap, because it is the single most common way agentic questions are written to mislead.

### Knowledge Check

**Q1:** A team needs a system that, given a customer question, looks up the order in a database, checks the shipping API, and — only if the package is delayed — issues a refund through a billing API, then replies. The exact sequence depends on what each lookup returns. Which construct fits best?
- A) A single prompt with a detailed system message
- B) A RAG pipeline over the order documents
- C) An agent with tools for the database, shipping, and billing APIs
- D) A fixed Lambda chain that always calls all three APIs

**A: C** — The sequence of actions is not knowable in advance; it branches on intermediate results (issue a refund only if the package is delayed). That runtime-dependent, multi-step, tool-using control flow is exactly what an agent's reasoning loop provides. A single prompt (A) cannot call external APIs. A RAG pipeline (B) is a fixed retrieve-then-generate flow with no actions. A fixed Lambda chain (D) cannot adapt — it would issue refunds unconditionally or require you to hand-code every branch, which is brittle compared to letting the model decide.

**Q2:** True or False — Adding a retrieval step in front of a model invocation turns the application into an agent.

**A: False.** Adding retrieval gives you a RAG pipeline — a fixed retrieve-then-generate flow. It is not an agent because the model is not placed in a loop where it chooses tools and iterates until a stopping condition; your code still decides every step and the model is invoked once. An agent requires the foundation model to drive the control flow through a reasoning loop with tools and memory, not just to receive retrieved context.

**Q3:** A document-routing task is highly standardized, rule-based, tolerates roughly a 1-2% error rate, and runs at high volume. According to the AWS decision framework, what is the most appropriate choice?
- A) A fully autonomous agent with broad tool access
- B) Traditional automation or RPA, not an agent
- C) A co-pilot pattern with a human making every decision
- D) A human-led workflow with the agent only assisting

**A: B** — The task is highly standardized and rule-based with no need for contextual adaptation, so it belongs to traditional automation or RPA rather than an agent; agentic AI is reserved for tasks that require contextual reasoning and adaptive decision-making. (If the task did warrant an agent, its low risk and 1-2% error tolerance would justify the fully autonomous deployment approach — but the standardization test rules out an agent first.) C and D over-apply human oversight to a low-risk, mechanical task.

**Q4:** A developer decides their use case needs an autonomous agent, then builds the reasoning loop, tool dispatch, session memory, and iteration limits from scratch on raw model calls. What is the most likely exam critique?

**A: This is the Pattern 3 trap — hand-rolling orchestration when a managed agent primitive fits.** Amazon Bedrock Agents already provides the orchestration loop, tool invocation through action groups, short- and long-term memory, and safeguards, so reinventing them on raw model calls adds cost, risk, and maintenance for no benefit. The exam rewards selecting the managed primitive when it satisfies the requirements. (The mirror-image trap is choosing an autonomous agent at all when a single prompt, RAG, or a deterministic workflow would do.)

**Q5:** Why is an unbounded agent reasoning loop a problem, and what is the standard mitigation?

**A:** Because the model controls the loop, an agent can fail to converge — reasoning in circles and calling tools repeatedly — which runs up token cost, increases latency, and creates a safety risk if it keeps taking real-world actions. The standard mitigation is to bound the loop with explicit stopping conditions: maximum iterations, timeouts, and circuit breakers (developed in Section 7). The clean termination is the model deciding the goal is met and emitting a final answer (the end-turn stop reason); the bounds exist to catch every other case.

### What a Bedrock Agent Is

Section 1 ended on the Pattern 3 trap: you decide a task genuinely needs an agent, and then you reach for raw model calls and start hand-building the reasoning loop, the tool dispatch, the memory, and the safeguards yourself. Amazon Bedrock Agents is the managed primitive that makes that build unnecessary. It is a fully managed capability that lets you build autonomous agents which orchestrate interactions between foundation models, your data sources, your applications, and the ongoing conversation — automatically calling APIs to take actions and invoking Knowledge Bases to retrieve grounding information. The point worth internalizing for the exam is what Bedrock manages on your behalf: prompt engineering, memory, monitoring, encryption, user permissions, and API invocation. You do not write infrastructure or orchestration code. You declare what the agent can do, and Bedrock runs the loop. That is the precise managed-primitive answer to the Pattern 3 trap from Section 1.

It helps to anchor the service against the perceive-reason-act framing from Section 1 by naming the four things a Bedrock Agent does. It extends the foundation model so it can break a task into a sequence of steps rather than answering in one shot. It collects additional information from the user through natural conversation when it needs more to proceed. It takes actions in the real world by making API calls to your systems. And it augments its own accuracy by querying your data sources. Those four behaviors — decompose, converse, act, retrieve — are the whole value proposition, and they map directly onto the reasoning loop you already understand.

Working with an agent splits across two API sets, and keeping them straight prevents a class of confusion. The build-time API is how you create and configure the agent — defining its components and assembling them. The runtime API is how your application actually invokes the finished agent to handle a user request. At build time you assemble four ingredients: a foundation model the agent uses to reason and generate; instructions, a natural-language description that tells the agent what it is for and how it should behave; at least one of action groups or knowledge bases (an agent that can neither act nor retrieve is not useful); and prompt templates that govern how the agent constructs its prompts at each step. Bedrock assembles these into base prompts. Crucially, after any configuration change you must Prepare the agent — this is the step that compiles your configuration into a testable, working version. The Prepare step recurs throughout this section, so flag it now: nothing you change about an agent (an action group, a Knowledge Base association, memory, a guardrail) takes effect until you Prepare it.

### The Orchestration Loop

At runtime your application calls `InvokeAgent`, and the agent runs through three stages: pre-processing, orchestration, and post-processing. Pre-processing comes first — it contextualizes and categorizes the user input and validates it, filtering and shaping the request before the real work begins. Orchestration is the core loop where the agent reasons, acts, and observes. Post-processing formats the final response for the user, and the single most testable fact about it is that post-processing is disabled by default. If an exam question hinges on whether the agent reformats its answer in a final pass, the default answer is no — you must explicitly enable post-processing.

**Diagram (described):** This is a simple left-to-right pipeline of the runtime flow. "User input" flows into "Pre-process — validate," which flows into "Orchestration — ReAct loop," which flows into "Post-process — off by default," which flows into "Final response." The key takeaway it conveys is the linear order of the three runtime stages, with post-processing explicitly labeled as disabled by default.

The orchestration stage is where Section 1's ReAct pattern becomes concrete, and AWS documents the correspondence explicitly: the rationale is the reason step, the action is the act step, and the observation is the observe step. Each turn of the loop proceeds in a predictable order. First, the agent interprets the input with the foundation model and generates a rationale — a chain-of-thought explanation of what it intends to do and why. Second, it predicts which action-group action to invoke, or which Knowledge Base to query. Third, it invokes that action: it sends the parameters to the backing Lambda function, or it returns control to your application (the two execution models covered below); if it lacks the information to proceed, it instead queries a Knowledge Base or reprompts the user for more detail. Fourth, it generates an observation from the action result or the Knowledge Base summary, folds that observation back into the base prompt, and re-interprets the augmented prompt with the model. The loop continues this way — reason, act, observe, augment, reason again — until the agent has enough to return a response or determines it must prompt the user. This is exactly the bounded reasoning loop from Section 1, now running inside a managed service.

### Action Groups: How an Agent Knows What It Can Do

An agent takes actions through action groups. An action group is a set of operations the agent is allowed to perform, and each action group defines its operations using exactly one of two schema types — and different action groups on the same agent may use different types. The first is an OpenAPI schema that defines a set of API operations. The second is a simplified function-definition schema that just lists each function's parameters. Both serve the same purpose: they describe to the agent what each operation does, when to use it, and what parameters it requires, so the model can map a user's intent onto the right operation and gather the right arguments.

The OpenAPI route is the richer of the two. You author the schema in JSON or YAML and either upload it to Amazon S3 or paste it inline in the console, where it is validated on entry. Several details here are exam-grade. The `openapi` version field must be exactly `"3.0.0"` — not 3.0.1, not 3.1. Bedrock supports only a subset of OpenAPI 3.0, and the most consequential omission is that `enum` is not supported; if an operation accepts only a fixed set of values, you describe the allowed values in the field's `description` rather than constraining them with `enum`. The schema's `paths` (required, each starting with `/`) and HTTP `method` (required) define the operations; the `description` on each operation is doing real work, because it tells the agent both what the operation does and when to call it — a weak description leads to a confused agent. The `operationId` is required for newer tool-use models such as Claude 3.5 Sonnet and Llama. The `responses` matter more than they first appear: the agent reads the response properties to build subsequent prompts, which is how it can feed the output of one operation into a later step of the loop. You define `parameters` and, for operations with a body, a `requestBody` (omit the body for GET and DELETE). One optional field is a genuine safeguard: `x-requireConfirmation` (values ENABLED or DISABLED, default DISABLED) makes the agent request explicit user confirmation before invoking that action — a defense against prompt injection driving an unintended action, and a concrete human-in-the-loop control point that Section 7 returns to.

The function-definition alternative trades expressiveness for simplicity. Instead of a full API contract, you list each function's parameters by name, type (one of string, number, integer, boolean, or array), description, and whether each is required. The agent behaves the same way at the intent-mapping level: it determines which function the user's request maps to, follows up with the user to gather any missing required parameters, and then either sends them to a Lambda function or returns them to your application. The intent-mapping mechanic is identical across both schema types — the agent uses the schema to decide which operation to invoke and which parameters are needed, then dispatches. You create action groups with the `CreateAgentActionGroup` API, where the `APISchema` field is a union that accepts only one of an inline `payload` string or an `s3` location.

A subtle trap lives in tool naming. With Claude 3.5 Sonnet, each action is exposed to the model under a tool name of the form httpVerb__actionGroupName__apiName, and that whole name must match the pattern of letters, numbers, underscores, and hyphens up to 64 characters. The double underscore is the delimiter, which means your action-group name and API name must not themselves contain double underscores — if they do, the delimiter breaks and the tool name is malformed. It is the kind of detail that surfaces as a "why won't this agent invoke the tool" troubleshooting question.

### Lambda-Backed Execution Versus Return of Control

When the agent decides to invoke an action, something has to actually run it, and Bedrock gives you a choice between two execution models. The first is the Lambda-backed model. When you create an action group you can quick-create a new Lambda (the recommended path), select an existing Lambda, or choose return of control. With the Lambda model, the agent packages the action's schema information plus metadata as an input event and sends it to your Lambda function; the function runs your business logic and returns a response, which becomes the agent's observation. Two facts here are easy to test. First, each action group has exactly one executor, so you write only one Lambda function per action group — that single function receives one operation at a time and must be written to handle all of the group's operations (typically by branching on which operation was invoked). Second, you must attach a resource-based policy to the Lambda granting the Amazon Bedrock service principal permission to invoke it; without that policy the agent cannot call your function. Watch the scope of the agent's quotas here, because the exam likes to mis-attribute them: the limits are defined *per agent*, not per action group. The adjustable `APIs per Agent` quota is **11** — the maximum number of API operations you can add across *all* of an agent's action groups combined — and an agent can have up to **20** action groups. AWS publishes no maximum number of operations within a single action group, so a distractor that says "up to 11 operations *per action group*" is wrong on the scope: 11 is the agent-wide API ceiling. The shape of the input event depends on the schema type: an OpenAPI-schema action group sends an `apiPath` and `httpMethod`, while a function-definition action group sends a `function` name.

The second model is return of control, and it exists because Lambda is not always the right home for an action. Instead of invoking a Lambda, the agent returns control by placing the predicted API or function parameters directly in the `InvokeAgent` response. You configure this by setting the action-group executor to RETURN_CONTROL as a custom-control value rather than a Lambda ARN — and the two are mutually exclusive, an action group's executor is either a Lambda ARN or RETURN_CONTROL, never both. With return of control, the details come back in the response as invocation inputs carrying a unique `invocationId`; your application performs the call itself, then sends the results back through `sessionState`, reusing the same `invocationId` and action group so the agent can resume the loop. A behavioral detail worth remembering: when `returnControlInvocationResults` is present in `sessionState`, the `inputText` is ignored — the agent treats that turn as the continuation of the prior action, not as new user input.

Knowing when to choose return of control over Lambda is the part the exam rewards. Return of control is the right answer when you want to leverage existing application APIs without rebuilding their authentication and networking inside a Lambda; when the task exceeds Lambda's 15-minute execution limit; or when your application already performs orchestration in a distributed, application-level way and the agent is just one participant. It is also, conceptually, a human-or-application control point in the agentic workflow — the agent proposes the action, but your code decides how and whether to carry it out, which Section 7 treats as a safeguard pattern.

| Dimension | Lambda-backed | Return of control |
|---|---|---|
| Who runs the action | Bedrock-invoked Lambda | Your application |
| Configured as | Lambda ARN in executor | RETURN_CONTROL custom control |
| Parameters delivered to | The Lambda input event | The InvokeAgent response |
| Time limit | Lambda 15-minute cap | No Lambda cap - app controls |
| Best when | New self-contained business logic | Reuse existing APIs and auth |
| Results returned via | Lambda return value | sessionState with invocationId |

### Agent Memory: Short-Term Session Versus Long-Term

Memory in Bedrock Agents comes in two distinct scopes, and the exam wants you to keep them apart. Short-term memory is within-session context. A session is a group of `InvokeAgent` requests that share the same session ID, and a single `InvokeAgent` call is a turn. As long as the requests share that session ID and the session has not gone idle past its `idleSessionTTLInSeconds`, the conversation history is preserved and used to augment the orchestration prompt on every turn. That is what makes a back-and-forth conversation coherent — but it lives only for the life of the session.

Long-term memory persists across sessions, and in native Bedrock Agents it takes one specific form: session summarization. You enable the SESSION_SUMMARY memory type, and the agent retains a persistent summary across separate sessions for a given user, so a conversation next week can build on a summary of what happened this week. The configuration knobs are worth recognizing: `memoryConfiguration` carries `enabledMemoryTypes` set to `["SESSION_SUMMARY"]`, a `storageDays` value between 1 and 365 (default 30) controlling how long summaries are retained, and a `sessionSummaryConfiguration.maxRecentSessions` controlling how many recent sessions feed the summary. A practical gotcha: a `memoryId` must be specified at invocation for a turn to actually be stored into long-term memory. And memory is not universally available — it is not supported on Amazon Titan Text Premier or Claude Instant, which is exactly the kind of model-compatibility detail an exam distractor exploits.

One clarification will save you from a guaranteed trap. The richer memory model — the one with named strategy types such as semantic, summarization, and episodic memory — is Amazon Bedrock AgentCore Memory, which is a separate service covered in Section 4. Do not conflate it with native Bedrock Agents memory. In Section 2's world, long-term memory means SESSION_SUMMARY; the multi-strategy memory belongs to AgentCore.

### Prompt Templates and Advanced Prompts

Every prompt the agent sends to the foundation model is built from a template, and Bedrock ships default base prompt templates that define how the agent constructs its prompt at each step of the loop. The four templates the exam cares about align one-to-one with the stages you have already seen: pre-processing, orchestration, knowledge-base response generation, and post-processing (disabled by default). Current documentation lists two additional templates — memory summarization and a routing classifier — which you should recognize as real but not let crowd out the foundational four.

Advanced prompts are the mechanism for customizing these templates, and they unlock four capabilities that map onto common real-world needs. You can edit a template to provide additional configuration and hand-curated few-shot examples that steer the model toward more accurate behavior on your domain. You can turn off a step entirely — useful when post-processing is unnecessary, or when disabling pre-processing helps you troubleshoot what the orchestration step is actually doing. And you can supply a custom parser Lambda that parses the raw output of each step, when the default parsing does not fit your model's output format. One compatibility detail matters here: the template format depends on the model. Older Claude models (Instant, v2.0, v2.1) use raw text templates, while Claude 3 and 3.5 use the Anthropic Claude Messages API JSON format — so a template authored for one will not drop cleanly into the other.

### Associating Knowledge Bases and Applying Guardrails

Two of the most important things you attach to an agent are covered in depth in other guides, so this section explains how they bind to the agent and points you to that depth. Associating an Amazon Bedrock Knowledge Base is how you bring retrieval-augmented generation to an agent. You build the Knowledge Base — a data source plus a vector store — exactly as Guide 02 describes, then associate it with the agent either at creation or afterward. When you associate it, you provide natural-language instructions that tell the agent when and how to query that Knowledge Base; the agent folds those instructions into its prompt context and uses them to decide, during orchestration, whether retrieval is warranted on a given turn. The build-time call is `AssociateAgentKnowledgeBase`, and you can adjust the Knowledge Base query configuration per invocation through `sessionState`. As always, you must Prepare the agent for the association to take effect. For the retrieval mechanics themselves — chunking, embeddings, vector stores, and query configuration — refer back to Guide 02; this section only covers the binding.

Applying a Bedrock Guardrail to an agent is how you prevent unwanted content in both user messages and agent responses. You attach a guardrail when you create or update the agent — in the console under the guardrail details, or via the API by supplying a `GuardrailConfiguration` (carrying a `GuardrailIdentifier` and a `GuardrailVersion`) on `CreateAgent` or `UpdateAgent`. The mental model to carry from Guide 06 is that Bedrock Guardrails apply uniformly across four Bedrock features — model inference, agents, Knowledge Bases, and flows — safeguarding both the prompts going in and the responses coming out. For the guardrail policy types (content filters, denied topics, word filters, PII redaction, contextual grounding) and how to configure them, see Guide 06; here, just remember that a guardrail is a first-class attachment to an agent and that it screens both directions of the conversation.

### Versioning, Aliases, and Tracing

The last piece is the agent lifecycle, and it follows a clean, deliberate pattern. When you create an agent, Bedrock gives you a working draft — the DRAFT version — and a test alias (TSTALIASID) that points at it. Every change you make applies to the working draft, and you iterate against the test alias until you are satisfied. When you are ready to deploy, you create an alias; Bedrock automatically creates an immutable numbered version (versions are numbered in order starting from 1) and points the new alias at it — or you can point an alias at a version that already exists. Your application always calls an alias, never the draft, which is the whole point: a version is a frozen snapshot, and an alias is a movable pointer to one. To push an update to production you create a new version and repoint the alias at it; to roll back, you repoint the alias at a previous version. Because the application only ever knows the alias, none of this requires the app to track version numbers. There is even a way to pause a deployed agent without touching IAM or deleting anything: set the alias's `aliasInvocationState` to REJECT_INVOCATIONS (and later ACCEPT_INVOCATIONS to resume) via `UpdateAgentAlias`.

| Concept | What it is | Mutable? | Who calls it |
|---|---|---|---|
| DRAFT | The working draft you edit | Yes - changes land here | Test alias only |
| Numbered version | Immutable snapshot, numbered from 1 | No | An alias points to it |
| Alias | Movable pointer to a version | Yes - repoint anytime | Your application |
| TSTALIASID | Built-in test alias to the draft | Points to DRAFT | Testing and iteration |

Tracing is how you see inside the loop, and it cross-references Guide 06's Responsible AI treatment. Each agent response can carry a trace that details the orchestrated steps — letting you follow the agent's reasoning from the input all the way to the response, including the inputs it sent to action groups, the Knowledge Bases it queried, the outputs it received, and the chain-of-thought rationale behind each step. On failure, the trace returns the failure reason, which makes it the primary troubleshooting tool for agents. You enable tracing at runtime in `InvokeAgent`, and each streamed chunk can carry a trace field (a TracePart) tagged with identifiers like agent ID, agent alias ID, session ID, and agent version. The trace is structured by step type, and recognizing the names helps when a question describes a symptom: PreProcessingTrace, OrchestrationTrace (which contains the Rationale, the InvocationInput, and the Observation), PostProcessingTrace, plus CustomOrchestrationTrace, RoutingClassifierTrace, FailureTrace, and GuardrailTrace. When an agent misbehaves, the trace — especially the OrchestrationTrace rationale and the FailureTrace — is where you look first.

### Knowledge Check

**Q1:** A developer is building an action group and sets the OpenAPI `openapi` field to "3.0.1", then uses an `enum` to restrict a status parameter to OPEN or CLOSED. The agent fails to build. What are the two problems?
- A) The version must be exactly "3.0.0", and enum is not supported — describe allowed values in the description instead
- B) The version is fine, but enum must be lowercase
- C) OpenAPI is not allowed at all — only function definitions work
- D) Both are fine; the failure is unrelated to the schema

**A: A** — Bedrock Agents require the OpenAPI version field to be exactly `"3.0.0"`, and they support only a subset of OpenAPI 3.0 that excludes `enum`. To constrain a parameter to a fixed set of values, you spell out the allowed values in the field's `description` so the model learns them from natural language rather than from an `enum` constraint. OpenAPI is fully supported (ruling out C), and the case of `enum` is irrelevant because `enum` is unsupported entirely (ruling out B).

**Q2:** True or False — By default, an Amazon Bedrock Agent runs a post-processing step that reformats its final response before returning it to the user.

**A: False.** Post-processing is disabled by default. The agent runs pre-processing and orchestration by default, but the post-processing template that reformats the final response must be explicitly enabled through advanced prompts. This is a favorite distractor: if a question assumes the agent always reformats its answer in a final pass, the default behavior says otherwise.

**Q3:** An agent can have up to ___ total API operations across all of its action groups and up to ___ action groups; you write exactly ___ Lambda function(s) per action group, and the Lambda needs a ___ policy granting the Amazon Bedrock service principal permission to invoke it.

**A:** 11 API operations *per agent* (the `APIs per Agent` quota, across all action groups combined), 20 action groups per agent, 1 Lambda function per action group, and a resource-based policy. Note the scope: 11 is an agent-wide ceiling, **not** a per-action-group limit — AWS publishes no maximum number of operations within a single action group, so "up to 11 operations per action group" is a scope trap. The single Lambda per group receives one operation at a time (identified by `apiPath`/`httpMethod` for an OpenAPI schema or by `function` for a function-definition schema) and must handle all of that group's operations. Without the resource-based policy for the Bedrock principal, the agent cannot invoke the function at all.

**Q4:** Scenario — Your agent must trigger a multi-hour data-reconciliation job that runs through an existing internal API with its own authentication and networking, and the job far exceeds 15 minutes. Which execution model fits, and why?

**A: Return of control.** Two of its three classic justifications apply directly: the task exceeds Lambda's 15-minute execution limit, and you want to reuse an existing application API without rebuilding its authentication and networking inside a Lambda. With return of control, the agent places the predicted parameters in the `InvokeAgent` response (with a unique `invocationId`), your application executes the long-running job itself, and you return the result through `sessionState` reusing the same `invocationId` so the agent can resume. A Lambda-backed action group would hit the 15-minute cap and force you to re-implement the API's auth.

**Q5:** A team deployed an agent behind an alias that the application calls. They ship a new version, and minutes later discover it misbehaves. They need to revert immediately and, separately, pause the agent during investigation — all without changing application code or IAM. What two actions accomplish this?

**A:** Repoint the alias at the previous numbered version to revert instantly, and set the alias's `aliasInvocationState` to REJECT_INVOCATIONS (via `UpdateAgentAlias`) to pause invocations during investigation (then ACCEPT_INVOCATIONS to resume). Because the application only ever calls the alias and versions are immutable snapshots, rolling back is just repointing the movable pointer — no app change, no IAM change, no version deletion. To diagnose the misbehavior itself, the agent trace (especially the OrchestrationTrace rationale and the FailureTrace) is the first place to look.

## Section 3: Function Calling & Tool Use

### From the Managed Loop to the Raw Primitive

Section 2 showed you the reasoning loop running inside a managed service: you declare action groups, and Amazon Bedrock Agents drives the reason-act-observe cycle for you. This section drops one level down to the primitive that the whole idea is built on. Strip away Bedrock's orchestration, its memory, and its prompt templates, and what remains is a single contract between your application and the model: the model can ask to use a tool, and your application can hand back the result. That contract lives in the Converse API, the unified, model-agnostic interface Guide 01 established for invoking foundation models. Function calling — also called tool use — is a Converse feature, which is why it inherits Converse's defining property: you write the code once and run it against different models by changing a model ID, not by rewriting your integration.

Understanding this primitive matters for two reasons the exam tests directly. First, it is the raw material of a self-managed agent loop: when you do not use Bedrock Agents and instead own the loop yourself, this contract is exactly what you implement — detect the model's tool request, run the tool, return the result, and re-invoke. Second, it clarifies what Bedrock Agents actually automates. An agent's action group maps onto a tool definition here; the agent's orchestration loop is the managed version of the detect-run-return cycle you are about to see by hand. Knowing the primitive is what lets you reason about when the managed loop earns its keep and when raw Converse tool use is the simpler answer.

### The Converse Tool-Use Contract

Tool use in Converse is configured through a single optional request field, `toolConfig`, which you pass alongside your `messages` (and alongside `guardrailConfig` if you are also applying a guardrail). `toolConfig` tells the model what tools exist and how much freedom it has to call them, and it has just two members. The first, `tools`, is a required array with at least one entry — each entry is a `Tool` object wrapping a `toolSpec`. The second, `toolChoice`, is optional and governs whether the model may, must, or must not call a tool on this turn.

A `toolSpec` is the description of one tool, and it has exactly the fields you would expect a model to need in order to decide whether and how to call it. The `name` is required (one to sixty-four characters, restricted to letters, numbers, underscores, and hyphens) and identifies the tool. The `inputSchema` is required and is a JSON-format schema describing the arguments the tool accepts. The `description` is optional but does real work — it is the natural-language explanation the model reads to decide when and how to use the tool, exactly as an OpenAPI operation's description did for a Bedrock Agent in Section 2. A weak description produces a confused model that calls the wrong tool or fills in the wrong arguments. There is also an optional `strict` boolean that opts the tool into structured-output enforcement, covered below.

The `inputSchema` deserves a careful look because it is the spine of structured tool input. It is a union whose member is `json`, and the JSON Schema you place there must have a top-level type of `object`, with `properties` describing each argument and `required` listing the mandatory ones. This is the same object-with-properties shape you author for any structured request, and it is the contract the model is bound to when it produces arguments.

Example `toolSpec` (JSON): a tool named `get_order_status` with the description "Look up the current status of a customer order by its ID." Its `inputSchema.json` is an object with one property `orderId` (a string described as "The order identifier"), and `orderId` is listed as required.

`toolChoice` is where a lot of exam value hides, because it is a union — you set exactly one of its three members, and they express three different policies. The default is `auto` (`AutoToolChoice`): the model decides for itself whether to call a tool or just generate text. The second is `any` (`AnyToolChoice`): the model must request at least one tool and is not allowed to produce a plain text answer this turn. The third is `tool` (`SpecificToolChoice`): the model must request the one named tool you specify. The critical caveat is that forcing behavior is model-dependent — the documentation phrases it as "if supported by the model" — and the only forcing mode with a verified model list is the specific `tool` choice, which is supported by Anthropic Claude 3 and Amazon Nova. Do not assume `any` and `tool` work universally across every tool-use-capable model; treat `auto` as the safe, portable default and forced choices as a capability you confirm per model.

| toolChoice member | Behavior | Notes |
|---|---|---|
| auto - AutoToolChoice | Model decides - call a tool or answer in text | The default |
| any - AnyToolChoice | Model must call at least one tool - no text | Model-dependent support |
| tool - SpecificToolChoice | Model must call the one named tool | Verified on Claude 3 and Nova only |

### The Request/Response Flow

The single most important behavioral fact about client-side tool use is that the model never runs the tool. The model can only ask. Your application executes the tool and feeds the answer back. The whole exchange is a short, well-defined conversation, and walking it step by step makes the contract concrete.

It begins when your application calls `converse` with a `messages` array containing the user's request (role `user`) and the `toolConfig` describing the available tools. The model reasons about the request and, when it concludes it needs a tool, signals that in two coordinated ways. The response's `stopReason` comes back as `tool_use`, and the assistant message's content carries a `toolUse` block. That `ToolUseBlock` has three required fields: a `toolUseId` (one to sixty-four characters, same character rules as a tool name) that uniquely tags this particular request, the `name` of the tool to call, and the `input` — a JSON object that conforms to that tool's `inputSchema`. The model has, in effect, filled out the form your schema defined.

Your application now does the work. It appends the assistant's tool-use message to the conversation so the history stays intact, executes the tool (calls the API, runs the query, whatever the tool does), and then sends a new message back with role `user` whose content is a `toolResult` block. The `ToolResultBlock` must carry a `toolUseId` that matches the request's `toolUseId` exactly — this pairing is how the model knows which outstanding request this result answers, which matters when more than one tool call is in flight. It carries a required `content` array (the actual result, discussed next) and an optional `status` of `success` or `error` — though that status field is itself supported only on Nova and Claude 3 and 4, another model-dependent detail.

With the result appended, your application calls `converse` again. The model consumes the tool result, reasons over it, and either produces its final answer or asks for another tool. A multi-tool task simply repeats the request-execute-return cycle: each turn the model may emit another `toolUse`, you run it and return another `toolResult`, and the loop continues until the model stops asking for tools. The clean terminal signal is a `stopReason` of `end_turn` — the model is done and has produced its final response. This is the same reason-act-observe loop from Section 1, except here your application is the loop's engine rather than Bedrock Agents.

**Diagram (described):** This diagram shows the client-side tool-use flow. "App — converse request" feeds into "Model reasons," and so does "toolConfig with toolChoice." The model's output reaches a decision node "stopReason." If the stop reason is `tool_use`, flow goes to "toolUse — id name input," then "App executes the tool," then "toolResult — matching id," which feeds back into the model for another reasoning pass. If the stop reason is `end_turn`, flow goes to "Final answer." The picture emphasizes that the application — not the model — executes the tool, and that the loop continues until an end_turn stop reason.

The `stopReason` field is worth knowing in full, because several of its values are diagnostic. The complete set is `end_turn`, `tool_use`, `max_tokens`, `stop_sequence`, `guardrail_intervened`, `content_filtered`, `malformed_model_output`, `malformed_tool_use`, and `model_context_window_exceeded`. The two tool-relevant error values — `malformed_tool_use` and `malformed_model_output` — are exactly what you look at when an agent loop fails to advance: the model tried to call a tool but produced arguments that did not parse against the schema, or its output was otherwise unusable.

A compact picture of the two messages at the heart of the exchange: when the stop reason is `tool_use`, the assistant content carries a `toolUse` block such as `{ "toolUse": { "toolUseId": "tooluse_a1", "name": "get_order_status", "input": { "orderId": "X-42" } } }`. The user-role message you send back next carries a `toolResult` block such as `{ "toolResult": { "toolUseId": "tooluse_a1", "content": [ { "json": { "status": "shipped" } } ], "status": "success" } }`. Note that the `toolUseId` matches across the two messages.

### Structured Tool Input and Output

Both ends of this exchange are structured, and that is the deeper theme connecting tool use back to the structured-output material in Guides 01 and 06. On the input side, the model does not emit free text when it calls a tool — it emits `toolUse.input` as JSON that must match the tool's `inputSchema.json`. The schema (top-level object, properties, required) is a constraint on what the model is allowed to produce, which is what makes tool arguments reliable enough to feed straight into an API call. On the output side, the result you return is also structured. A `ToolResultContentBlock` is a union, so each block in the result is exactly one of several types: `json` (a JSON value), `text` (a string), `image` (Nova and Claude 3 and 4 only), `document`, `video`, or `searchResult`. A tool can therefore return clean structured data as `{ "json": { ... } }` or simple prose as `{ "text": "..." }`, depending on what the model needs to consume next.

By default, the schema is a strong hint rather than a hard guarantee — a capable model usually honors it, but nothing in the base contract forces it to. Bedrock's Structured Outputs feature closes that gap using constrained decoding, and it offers two mechanisms you can use together or independently. The first is strict tool use: add `strict: true` to a tool's definition to turn on schema validation of the tool name and its inputs, so the model is constrained to produce valid tool calls. The second is JSON Schema output format, configured through Converse's `outputConfig.textFormat` with a type of `json_schema` and a `jsonSchema` carrying the schema, a name, and a description — this constrains the model's ordinary text response (not a tool call) to a schema you define. The mechanics carry a few exam-relevant edges: schemas are validated against a supported subset of JSON Schema Draft 2020-12, and using an unsupported feature returns an immediate 400 rather than failing silently; a new schema compiles (which can take up to a few minutes) and is then cached for twenty-four hours, so the first call against a fresh schema may be slower. Structured Outputs is available on Converse and ConverseStream and on InvokeModel. The throughline to remember is that tool use gives you structured input by schema, the tool-result union gives you structured output by type, and Structured Outputs upgrades the schema from a hint into an enforced guarantee.

### Model-Agnostic Tool Use and the Three Modes

The reason this primitive is worth learning once is that you only learn it once. Converse is a single, consistent interface that works with every model that supports messages, and tool use rides on top of that interface — so the same `toolConfig` contract works across any tool-use-capable model. Switching from one model to another is a `modelId` change, not a rewrite of your tool integration. That portability is the agentic facet of exam Pattern 5 (switch models without code changes), which Section 8 develops in full. One important honesty note the exam respects: there is no single authoritative matrix in the documentation listing exactly which models support tool use, so treat tool-use support as model-dependent and check the relevant model card rather than asserting a fixed list.

AWS describes three modes of tool use, and distinguishing them prevents a common category error. The first and most common is client-side tool use: your application runs the tool after the model returns a tool-call request — this is the entire flow above, available through Converse and the other inference interfaces. The second is server-side tool use: Bedrock itself invokes the tool through a registered Lambda or an AgentCore Gateway, rather than handing the request back to your code. Server-side tool use is currently scoped to the Responses API (not Converse) and is fast-moving, so treat it as a Responses-API capability for now; the custom Lambda behind it implements MCP by handling the `tools/list` and `tools/call` operations, and Bedrock ensures that Lambda runs under the same IAM policy as the calling application. This is the forward link to AgentCore Gateway in Section 4 and to MCP server hosting in Section 5. The third mode is Anthropic Claude tool use, where your code uses Anthropic-defined tool types directly.

Place the three modes on the autonomy spectrum from Section 1 and the relationship to Bedrock Agents snaps into focus. Client-side Converse tool use is the primitive of a self-managed agent loop — your application owns the entire reason-act-observe cycle: detect `tool_use`, run the tool, return the `toolResult`, re-invoke. Bedrock Agents is the managed orchestration of that exact same idea — Bedrock owns the loop, the action invocation, and the conversation state, so you declare tools instead of writing the loop. Server-side tool use sits in between: Bedrock runs the tool for you, but you are not handing over the whole orchestration the way a full agent does. The same tool-use concept, in other words, appears at three levels of management, and choosing among them is choosing how much of the loop you want to own.

| Mode | Who runs the tool | Scope | Relationship to the loop |
|---|---|---|---|
| Client-side | Your application | Converse and other inference APIs | The self-managed agent-loop primitive |
| Server-side | Bedrock - via Lambda or Gateway | Responses API today - fast-moving | Midpoint - Bedrock runs the tool |
| Bedrock Agents | Bedrock - managed | Agents service | Fully managed orchestration of tool use |

Two permissions facts round out the practical picture. Converse requires the `bedrock:InvokeModel` action, while ConverseStream requires `bedrock:InvokeModelWithResponseStream`, and both Converse operations live on the bedrock-runtime endpoint only — not the control-plane endpoint you use to manage agents and models.

### Streaming Tool Use

When you call ConverseStream instead of Converse, a tool call does not arrive as one complete block. It streams as partial JSON deltas, and your application has to reassemble it. The stream opens a content block with a `contentBlockStart` event, delivers a series of `contentBlockDelta` events whose `delta.toolUse` carries fragments of the input JSON, and closes the block with `contentBlockStop`; a terminal `messageStop` event carries the `stopReason`. The operational consequence is the one to remember: you must accumulate the partial JSON across the deltas and wait for the block to finish before you execute the tool, because no single delta contains a complete, valid argument object. This is the streaming counterpart to the same contract — same `toolUse` and `toolResult` shapes, delivered incrementally.

### Knowledge Check

**Q1:** A Converse response comes back with `stopReason` of `tool_use` and an assistant message containing a `toolUse` block. What must the application do next, and how does the model learn the outcome?
- A) The model already ran the tool; just read the result from the toolUse block
- B) Execute the tool itself, then send a new user-role message with a toolResult whose toolUseId matches the request, then call converse again
- C) Call a separate ExecuteTool API that Bedrock provides
- D) Set toolChoice to any and re-invoke so the model retries

**A: B** — In client-side tool use the model never runs the tool; it only emits a request. The application executes the tool, appends the assistant's tool-use message, and returns a new message with role `user` carrying a `toolResult` block whose `toolUseId` exactly matches the request's `toolUseId`. It then calls `converse` again so the model can consume the result and continue. A is the classic misconception — the `toolUse` block holds the requested input, not an executed result. There is no separate execute API (C), and changing `toolChoice` (D) does not deliver the result the model is waiting on.

**Q2:** True or False — Setting `toolChoice` to the specific `tool` option to force a single named tool is supported uniformly across all foundation models that support tool use.

**A: False.** Forcing a specific tool with `toolChoice`'s `tool` (SpecificToolChoice) member is model-dependent — the documentation qualifies forcing behavior with "if supported by the model," and the verified support for the specific-tool choice is limited to Anthropic Claude 3 and Amazon Nova. The portable, always-safe option is `auto` (the default), where the model decides whether to call a tool. Do not assume `any` or `tool` work on every tool-use-capable model.

**Q3:** Fill in the blanks — A `toolSpec` requires a `name` and an `inputSchema` whose top-level JSON Schema type must be ___, and a returned `toolResult` must include a `toolUseId` that ___ the request's toolUseId.

**A:** the top-level type must be `object` (with `properties` and `required`), and the `toolResult`'s `toolUseId` must exactly match the request's `toolUseId`. The object-typed schema is what constrains the arguments the model produces, and the matching ID is what pairs a result to the specific tool request it answers — essential when multiple tool calls are outstanding.

**Q4:** Scenario — A team built a tool-using feature directly on client-side Converse, owning the detect-tool-use, execute, return-result, re-invoke loop themselves. Their requirements then grow to need managed conversation state, automatic loop orchestration, and built-in memory, with tools declared rather than hand-wired. What is the natural move, and why is the underlying contract the same?

**A: Move to Amazon Bedrock Agents.** Bedrock Agents is the managed orchestration of the very same tool-use idea — Bedrock owns the reason-act-observe loop, the action invocation, and the conversation state, so the team declares tools (action groups) instead of coding the loop. The contract is conceptually identical: an agent's action group maps onto the same tool definition, and the agent's orchestration loop is the managed version of the detect-execute-return-reinvoke cycle they were running by hand. Client-side Converse tool use is the raw primitive; Bedrock Agents is that primitive with the loop, memory, and invocation managed for you.

**Q5:** A developer wants a hard guarantee — not just a strong hint — that the model's tool calls always conform to the tool's schema, and separately wants a plain-text response constrained to a JSON schema. Which feature and which two mechanisms apply?

**A: Bedrock Structured Outputs, via its two mechanisms.** For tool calls, set `strict: true` on the tool definition (strict tool use), which turns on constrained-decoding validation of the tool name and inputs. For a schema-constrained text answer, use Converse's `outputConfig.textFormat` with type `json_schema` and a `jsonSchema`. The two mechanisms work together or independently. Remember the edges: schemas validate against a supported subset of JSON Schema Draft 2020-12, unsupported features return an immediate 400, and a new schema compiles (up to a few minutes) then caches for 24 hours.

## Section 4: Amazon Bedrock AgentCore

> **Fast-moving topic notice.** Amazon Bedrock AgentCore is a new and rapidly evolving service. Every factual claim in this section was verified against current AWS documentation via MCP research before it was written, and the points that are changing fastest — the component set, the Region count, and the general-availability dates — are explicitly flagged in the prose and consolidated under "Fast-Moving Flags" below. ### Where AgentCore Fits

You have now seen two ways to put a foundation model in a reasoning loop. Section 2 gave you Amazon Bedrock Agents — the managed primitive that lives fully inside Amazon Bedrock and runs the orchestration loop, memory, and tool invocation for you. Section 3 gave you the raw Converse tool-use contract — the self-managed loop you implement yourself when you own the detect-run-return cycle. AgentCore is a third thing, and the cleanest way to avoid the most common confusion on the exam is to understand that it is not a competitor to either of those. It is the platform you run agents on once you decide to take them to production.

Amazon Bedrock AgentCore is a platform to build, connect, deploy, and operate AI agents at scale. Its defining properties are that it is framework-agnostic and model-agnostic and fully managed — AWS handles the infrastructure (the "backend plumbing") so you focus on the agent's logic, and it delivers enterprise-grade security and reliability "without giving up open-source flexibility." AWS Prescriptive Guidance frames it as "an agentic platform to build, deploy, and operate highly capable agents securely at scale using any framework, model, or protocol," all without managing infrastructure. The phrase to hold onto is operate at scale: AgentCore's reason for existing is the operational layer — deployment, identity, memory, tool access, and observability — that an agent needs once it leaves a developer's laptop and starts serving real traffic.

Three flexibility claims are worth stating precisely because they are exactly what distinguishes AgentCore from a single-vendor managed agent. It supports custom frameworks plus any open-source framework, and AWS documentation explicitly names CrewAI, LangGraph, LlamaIndex, Google ADK, the OpenAI Agents SDK, and Strands Agents. It is model-agnostic — any foundation model inside or outside Amazon Bedrock, with OpenAI, Google Gemini, Anthropic Claude, Amazon Nova, Meta Llama, and Mistral all named. And it speaks open protocols: it supports the Model Context Protocol (MCP, the subject of Section 5) and Agent-to-Agent (A2A) communication, with A2A currently available in AgentCore Runtime and broader A2A support across other services described as coming. The point of all three is the same: you are not locked into one framework, one model, or one protocol to get a managed, production-grade operating environment.

One more structural property shapes how you adopt it. AgentCore is composable — its capabilities can be used together or independently. AWS frames this directly: "use what you need today and add more as your agents take on more responsibility." You do not have to adopt the whole platform to use one piece of it. A team can put a Strands agent on AgentCore Runtime and nothing else, or use AgentCore Memory under an agent that otherwise runs on its own infrastructure. That composability is why the rest of this section walks through the components as a menu rather than a monolith.

### AgentCore Versus Bedrock Agents — the Key Distinction

This is the comparison the exam tests most reliably, so anchor it firmly. The AgentCore FAQ answers the question directly under the heading "I am using Amazon Bedrock Agents today. Should I switch to AgentCore?" — and the answer is that if you use Bedrock Agents today, you can continue to do so. There is no forced migration and the two are not mutually exclusive.

The distinction is one of layer, not of rivalry. Bedrock Agents is the managed agent primitive inside Amazon Bedrock: action groups, the orchestration loop, and native session and summary memory, all the way back in Section 2. It is the fastest path to a managed agent that lives entirely within Bedrock, and the price of that speed is that you accept Bedrock's managed orchestration. AgentCore is the operational platform that is deliberately broader: it runs agents built with any open-source framework (Strands, LangChain, LangGraph, LlamaIndex, CrewAI), backed by any foundation model (inside or outside Bedrock), with MCP support for standardized tool access, VPC connectivity for private resources, and A2A for inter-agent communication. The framing to carry into every exam question is this: Bedrock Agents is the managed primitive, AgentCore is the production operating platform, and open-source frameworks like Strands are the authoring layer that runs on AgentCore. They are complementary layers of the same stack, not three options you choose between.

A timeline detail belongs here, flagged as point-in-time. AgentCore entered preview in July 2025 and became generally available in October 2025. [FLAG — fast-moving] Treat those dates as a snapshot; for the exam, the durable fact is the relative ordering (preview, then GA) and the positioning, not the calendar.

### The Component Set

AgentCore is best understood as a set of composable services, each solving one operational problem that an agent faces in production. The seven that the AIP-C01 objectives name — Runtime, Gateway, Memory, Identity, Observability, Code Interpreter, and Browser — are all verified present and generally available. Walk through them in the order an agent encounters them, because the purpose of each becomes obvious once you see the problem it solves.

Runtime is where the agent actually executes. It is a secure, serverless, session-isolated environment to deploy and scale agents and tools, accepting any framework, model, or protocol. Two capabilities make it production-grade rather than a toy: it supports long-running asynchronous workloads (up to eight hours), which matters because real agentic tasks are not always sub-second request/response, and it supports bi-directional streaming. Session isolation is the security-critical property — one user's agent session cannot bleed into another's.

Gateway is how the agent reaches tools. It turns APIs, AWS Lambda functions, and existing services into agent-compatible tools and connects to existing MCP servers, all behind one unified, secured endpoint — in other words, a managed MCP endpoint. This is the component that removes the need to build and operate MCP server infrastructure yourself, and because it is central to both this section and Section 5, it gets its own treatment below.

Memory gives the agent state that outlives a single model call. It is managed short-term memory (the multi-turn context within a conversation) and long-term memory (knowledge that persists across separate sessions), and it handles the hard parts for you: vector embeddings, consolidation, reflection, namespace isolation between users, and either pre-defined or custom extraction strategies. This is the component that resolves the memory trap seeded back in Section 2 — the richer, multi-strategy memory model belongs here, in AgentCore Memory, not in native Bedrock Agents, a point developed below.

Identity solves authentication and authorization for agents specifically. It is identity and credential management built for agents, with a secure token vault, support for OAuth two-legged and three-legged flows, and integration with Amazon Cognito, Microsoft Entra ID, and Okta; it is powered by Amazon Cognito under the hood. The problem it addresses is real and often overlooked: an agent acting on a user's behalf needs to hold and present credentials to downstream services securely, and doing that by hand is both tedious and dangerous.

Observability lets you see inside the agent in production. It is the ability to trace, debug, and monitor agents, it is OpenTelemetry-compatible, and it is powered by Amazon CloudWatch, surfacing metrics, spans, and traces. Because tracing is both an exam topic in its own right and a cross-reference to Guide 06, it gets a dedicated subsection below.

Code Interpreter and Browser are the two built-in tools. Code Interpreter is a secure, sandboxed environment for executing code (Python, JavaScript, and TypeScript), with support for large files and Amazon S3 and optional internet access — it is what lets an agent run code to compute an answer rather than only describe one. Browser is a fast, secure, cloud-based browser runtime with VM-level isolation that lets agents operate websites at scale — it is what lets an agent navigate and act on a web application that exposes no API. [FLAG — framing only, no contradiction] AWS documentation presents these two in two ways: the Prescriptive Guidance page groups Code Interpreter and Browser together under "AgentCore built-in tools," while the FAQ lists them as separate top-level components. Both framings are accurate; present them as the platform's two built-in tools.

### AgentCore Gateway as a Managed MCP Endpoint

Gateway deserves a closer look because it is the bridge between an agent and the tools it needs, and because it is the concrete answer to a problem Section 5 frames in full. The problem is this: the Model Context Protocol standardizes how an agent discovers and calls tools, but someone still has to run the MCP servers that expose those tools — and operating that infrastructure (hosting, scaling, securing, authenticating) is real work. Gateway removes that work. It gives agents a single, unified, secure endpoint to discover and call tools, it transforms APIs and Lambda functions into agent-compatible (MCP-compatible) tools, and it connects to existing MCP servers. That is precisely the "managed MCP endpoint" framing: you point your agent at Gateway, and Gateway handles the tool plumbing behind it.

The breadth of tool sources is what makes it powerful. Gateway can expose AWS services (Amazon S3, DynamoDB, Aurora, Amazon Redshift, Lambda), third-party services, and custom tools defined via API specs, function code, MCP servers, OpenAPI, Smithy, Lambda, or ECR container images. At scale, an agent might have access to hundreds of tools, so Gateway adds built-in semantic search for tool selection plus metadata-based filtering (for example, filtering by risk level), so the model is offered the right tools rather than drowned in all of them. On security, Gateway supports multiple authentication methods — IAM-based, OAuth 2.1, and API keys — and integrates with AgentCore Identity for authentication and authorization and with Observability for audit, and it supports AWS WAF web ACLs. It also ships one-click integrations with common SaaS tools such as Salesforce, Slack, Jira, Asana, and Zendesk.

[FLAG — terminology, anti-confusion] Do not conflate two similarly named things. The AgentCore Gateway is the managed MCP endpoint that agents call at runtime to reach tools. The open-source "AgentCore MCP Server" is a completely different artifact — a developer-tooling MCP server that AI coding assistants (such as Kiro, Claude Code, or Amazon Q Developer) use to scaffold and deploy agents during development. One is a runtime tool-access service; the other is a build-time developer aid. An exam distractor can exploit the shared "MCP" and "AgentCore" words to make them sound interchangeable; they are not.

### AgentCore Observability — OpenTelemetry and CloudWatch

Observability is where this section connects back to Guide 06's agent-tracing material, and the relationship is worth stating plainly: AgentCore Observability is the platform-level analogue of the built-in trace you saw on Bedrock Agents in Section 2. Where a Bedrock Agent gives you a trace of its own orchestration steps, AgentCore Observability gives you tracing, debugging, and monitoring for any agent — built with any framework, backed by any model — across all the AgentCore services.

The mechanics rest on two pillars the exam pairs together: OpenTelemetry and Amazon CloudWatch. AgentCore Observability is powered by Amazon CloudWatch and is OpenTelemetry-compatible. Telemetry for traces, session count, latency, duration, token usage, and error rates surfaces on the CloudWatch GenAI Observability page as real-time dashboards. Built-in metrics for Runtime, Memory, Gateway, the built-in tools, and Identity are available in CloudWatch by default — you get a baseline of visibility without instrumenting anything. To see the full range of telemetry, including custom runtime metrics, you instrument your agent code with the AWS Distro for OpenTelemetry (the ADOT SDK, packaged as `aws-opentelemetry-distro` and run via `opentelemetry-instrument`). One piece of setup is easy to forget and therefore exam-worthy: you must enable CloudWatch Transaction Search once, which is what lets you view the service-generated metrics, spans, and traces (it ingests spans as structured logs and uses an AWS X-Ray trace-segment destination of CloudWatch Logs).

Two consequences of the OpenTelemetry foundation matter. First, Observability is not limited to agents hosted on AgentCore Runtime: agents on Runtime get automatic instrumentation, but agents hosted elsewhere — on Amazon EC2, Amazon EKS, AWS Lambda, or even other clouds — can also be observed with some additional setup. Second, because the telemetry is OpenTelemetry-standard, it integrates with third-party observability tools such as Arize Phoenix, Datadog, Dynatrace, LangSmith, Langfuse, and Braintrust, rather than locking you into one viewer. The memory hook for the exam is the pairing: AgentCore Observability means OpenTelemetry (ADOT) for the instrumentation standard and CloudWatch for the metrics, spans, and traces, with Transaction Search as the one-time enablement step.

### AgentCore Memory and the Multi-Strategy Distinction

Section 2 deliberately left a flag planted: native Bedrock Agents long-term memory is session summarization (SESSION_SUMMARY), and the richer, named-strategy memory model belongs to a separate service. That separate service is AgentCore Memory, and this is where the distinction pays off. AgentCore Memory provides both managed short-term memory (multi-turn context) and long-term memory (knowledge that persists across sessions), and it supports multiple extraction strategies — semantic, summarization, and episodic — with namespace isolation between users and either pre-defined or custom strategies. Episodic memory specifically was called out as a verified addition in a later AWS update. The exam-relevant rule is crisp: if a question describes semantic, summarization, or episodic memory strategies, that is AgentCore Memory; if it describes a persistent session summary in native Bedrock Agents, that is SESSION_SUMMARY. Conflating the two is a deliberate trap.

### When AgentCore Is the Right Choice

The decision rule for AgentCore is not "is this an agent?" — Sections 1 and 2 already answered that. The decision rule is "I have an agent; where do I operate it?" AgentCore is the right choice for organizations moving agents from proof-of-concept — typically built on an open-source or custom framework — to production, where they now need enterprise-grade security and reliability but do not want to give up the framework and model flexibility that got them this far. That is the scenario the exam will paint: a team has a working agent in a notebook or on a single server, and the question is how to run it at scale, securely, without rewriting it into a single vendor's managed primitive.

A few signals point specifically at AgentCore. It fits when you want to lower operational overhead through a fully managed service rather than running agent infrastructure yourself. It fits when you need modular services that work together or independently with any framework and model — the composability point from earlier. It fits when you have stateful conversational agents that need durable memory. And it fits when agents must integrate with diverse applications, data sources, and APIs, which is exactly what Gateway and Identity are built for. AWS Prescriptive Guidance describes the implementation as a set of pillars that double as a checklist: deploy securely on serverless infrastructure with session isolation and built-in IAM and identity; enhance the agent with persistent Memory, Gateway tools, and the Browser and Code Interpreter built-in tools; trace, debug, and monitor with Observability via CloudWatch and OpenTelemetry; and accelerate development with the modular, composable building blocks.

### Bedrock Agents, AgentCore, and Open-Source Frameworks — the Three Layers

Pulling the section together, the comparison the exam wants you to be able to draw is between three layers of the agentic stack, not three competing products. The managed primitive is a fully managed agent inside Amazon Bedrock — action groups, orchestration loop, native memory — and Amazon Bedrock Agents is its embodiment; you reach for it when you want the fastest path to a managed agent fully inside Bedrock and you accept the managed orchestration. The authoring framework is an open-source, model-driven SDK for defining an agent (a model plus a system prompt plus a set of tools) and multi-agent patterns; Strands Agents is the AWS-favored example (with LangGraph, CrewAI, and LlamaIndex as peers), and you reach for it when you want code-level control of the agent loop and portability across environments. ## Section 5: Model Context Protocol

### From Custom Glue Code to an Open Standard

By now you have seen the same idea — let a foundation model reach outside itself and call a tool — surface at several levels of management. Section 3 showed you the raw Converse tool-use contract, where your application declares each tool's name, description, and input schema by hand and owns the detect-run-return cycle. Section 4 showed you AgentCore Gateway, which turns APIs and Lambda functions into agent-callable tools behind one managed endpoint. Guide 02 introduced the Model Context Protocol briefly, as one of the consistent, model-agnostic ways an agent gets access to tools and data. This section develops MCP in full, because it is the open standard that ties the whole tool-access story together — and it is a named Task 2.1 topic the exam tests directly.

Start with the problem MCP exists to solve, because the protocol only makes sense once you feel the pain it removes. Imagine you have M different AI applications — a customer-support agent, a coding assistant, an internal analytics bot — and N different systems each of them might need to touch: a GitHub repository, a Slack workspace, a product-catalog database, an internal API. Without a standard, every application needs its own bespoke integration to every system it uses. Wire all of them together and you are building and maintaining M × N custom integrations, an integration matrix that grows multiplicatively and becomes unmanageable. Each new tool means re-integrating it into every application that wants it; each new application means re-integrating every tool. This is the source of the information silos, integration complexity, and scalability bottlenecks that plague enterprise AI.

MCP collapses that M × N problem into an M + N problem. It is an open standard — a protocol, not a product — that defines one common language for how an AI application discovers what tools are available, requests their execution with specific parameters, and receives and processes the results. AWS describes it as a universal connector, or a universal translator, between models and the external world. Once the language is standard, you build M clients (one per application, each speaking MCP) and N servers (one per tool or data source, each speaking MCP), for M + N implementations total. A new tool, exposed once as an MCP server, instantly becomes available to every MCP-speaking application with no custom glue. AWS draws the analogy to two earlier standardizing protocols that did the same thing in their domains: APIs standardized how web applications talk to backends, and the Language Server Protocol (LSP) standardized how IDEs talk to language tooling so that any editor could support any language through one interface. MCP aims to do for AI-to-tool integration what LSP did for editor-to-language integration. That M × N → M + N framing is the single most useful way to explain why MCP matters, and it is worth being able to reproduce on demand.

**Diagram (described):** This diagram illustrates the M + N collapse. Two AI applications, "AI app A" and "AI app B," both connect into a central node labeled "MCP — common language." From that central node, three arrows fan out to three servers: "Server — GitHub," "Server — database," and "Server — Slack." The picture conveys that every application speaks one common protocol to every server, so applications and servers connect through a shared standard rather than through bespoke pairwise integrations.

Two points of origin and structure round out the foundation. MCP was originated by Anthropic and released as an open-source project, which is why it is genuinely an open standard rather than a single vendor's API — AWS, among others, now contributes to its evolution. And underneath the headline "tools" idea, the protocol actually standardizes three kinds of capability a server can expose. Tools are functions the model can call to retrieve information or perform an action. Resources are data that can be pulled into the model's context — database records, file contents, images. Prompts are reusable templates that guide how the model should interact with a given tool or resource. For the exam, the durable trio to remember is the function MCP performs — discover, execute, process results — and the fact that it standardizes tools, resources, and prompts. (At the wire level, MCP rides on JSON-RPC 2.0, using standard input/output streams for local servers and a streamable HTTP transport for remote ones; you rarely need that detail for the exam, but it becomes relevant in a moment because the stateless-versus-stateful nature of that transport is exactly what drives the hosting choice.)

### The Client/Server Model

MCP is a client-server protocol, and getting the two roles straight is the foundation for everything that follows. The MCP client is the AI application — the agent or the host application — that needs access to external data and tools. Anthropic's Claude Desktop is a client; so is a custom agent built on Amazon Bedrock, an AI coding assistant like Kiro or Amazon Q Developer, or an agent authored with a framework like the Strands Agents SDK. The MCP server is the process that exposes a specific set of tools and data to clients in the standardized MCP form — a server fronting a GitHub repository, a Slack workspace, an AWS service, or an S3-backed product catalog. The division of labor is clean: the server advertises its capabilities (its tools, resources, and prompts), and the client discovers them and decides which to call. The same well-defined protocol governs the conversation whether the client and server run on the same machine or across a network — and that "local or remote" flexibility is the hinge that the entire hosting discussion turns on.

One subtlety inside the client's job is worth pulling out, because it is both a real design decision and an exam-relevant one: how does the client decide which of a server's tools to register and offer to the model? AWS Prescriptive Guidance describes three tool-discovery approaches, and they trade precision against convenience against context-window cost. The first is static definition — the client manually specifies individual tools by name. This is precise and minimizes context-window usage because only the named tools are loaded, but it is brittle (it breaks if the server's tools change) and it requires knowing the tool names up front. The second is dynamic discovery — the client automatically registers all of a server's available tools by listing them. This is the simplest to operate, but every registered tool's name, description, and schema consumes context-window space (on the order of a few hundred tokens per tool), so twenty tools can quietly cost thousands of tokens before the user has even asked anything, and the cost grows linearly as the tool catalog expands. The third is a search function, or semantic search — the client lists all tools but then semantically matches them against the user's actual prompt at runtime and registers only the relevant subset. This is the best-practice approach at scale because it preserves the context window while still giving the model access to a large catalog, and it is exactly the capability AgentCore Gateway provides natively (Section 4). The reason this matters for the exam is the context-window tradeoff: registering every tool from every server does not scale, and the fix is filtering or semantic tool selection.

The server then leads to "Return results," which flows back to the "MCP client." ### Hosting MCP Servers: Lambda, ECS/Fargate, and AgentCore

Here is the net-new core of this section, and the part most likely to appear as a "pick the architecture" question. Abstracting your tools into MCP servers decouples agent development from the tools themselves — but it raises an immediate operational question: where do you run the server? There are two layers to the answer, and it helps to keep them separate. The first layer is where the server lives relative to the user (its hosting strategy); the second layer is what compute you run a remote server on. Both layers are exam-relevant, so take them in turn.

The hosting-strategy layer has three options. Local hosting runs the MCP server as a subprocess on the end user's own machine, talking to the client over standard input/output. It requires no authentication between client and server — the server simply uses the user's locally stored credentials and inherits their local network access — which makes it the simplest possible setup and ideal for rapid iteration, such as a coding assistant's local Git server. Its weaknesses are operational: every user must discover, download, and configure the server themselves, and you cannot control the server's lifecycle, so users may run outdated or vulnerable versions, which complicates compliance. Remote hosting puts the MCP server behind an HTTP/HTTPS endpoint reachable by any networked client, which is what enables central access control, authentication and authorization, and version and update control — at the cost of having to actually implement that authentication and authorization (a point Topic 5 develops). An MCP gateway is the third strategy: a centralized proxy that sits between clients and a fleet of remote servers, giving an agent a single endpoint that handles authentication, authorization, routing, and protocol translation, and lets new servers be added without the agent re-registering each one. AgentCore Gateway is AWS's MCP gateway, treated below.

When you choose remote hosting, the compute layer is the next decision, and this is where the task's Lambda-versus-Fargate distinction lives. AWS enumerates the options explicitly, and the cleanest way to hold them is by the nature of the tool the server exposes.

AWS Lambda is the right home for lightweight, stateless tool endpoints with bursty or intermittent traffic. Lambda is serverless: it automatically scales to match the incoming request rate, runs across multiple Availability Zones for fault tolerance, scales to zero when idle, and bills per use. That profile fits a tool where each call is short and self-contained — a price lookup, a record fetch, a unit conversion — because there is no server-side session to maintain between calls and nothing to keep warm. The flip side is the constraint that defines "lightweight": a Lambda invocation runs for a bounded, relatively short duration and does not hold long-lived connections or in-memory session state across calls, which is exactly why it suits stateless tools and not long-running ones. (The platform enforces a maximum execution duration per invocation; treat it as a short, bounded window rather than memorizing an exact figure, since the durable point is "short-lived," not the specific number.)

Amazon ECS on AWS Fargate is the right home for complex or long-running MCP servers — the ones Lambda's model does not fit. Fargate runs your server as a long-lived containerized service, which buys you several things at once: warm caches and persistent streaming connections that survive across requests; the freedom to use any language or runtime and to bundle heavy native libraries or large processing pipelines; and natural integration with enterprise perimeter controls like an Application Load Balancer, AWS WAF, private subnets, and VPC endpoints. That makes it the choice for servers that are sessionful or streaming-heavy, dependency-heavy, need stable high-throughput concurrency, or must live inside a VPC alongside the private data stores they front. The technical reason it fits long-lived sessions ties back to the transport: MCP's streamable HTTP transport has a stateless mode (each call is a self-contained request, so the server scales horizontally across replicas with no session affinity) and a stateful mode (a session identifier ties a multi-step workflow to server-side context). A long-lived Fargate service is what lets you run the stateful mode well.

Amazon Bedrock AgentCore is the third option, and it sits above the compute choice entirely: rather than hosting a server on Lambda or Fargate yourself, you let AgentCore host or front it as a fully managed service, with built-in identity, memory, and tool discovery. AgentCore Runtime can host an MCP server container directly, and AgentCore Gateway can put a managed MCP endpoint in front of your tools — both covered as the managed answer below. EC2 and EKS round out the list as additional self-managed hosting locations when you need that level of control. The decision rule the exam rewards is the contrast: choose Lambda when the tools are lightweight and stateless with bursty traffic; choose ECS/Fargate when the server needs long-lived connections, streaming, server-side session state, heavy dependencies, stable concurrency, or VPC residence; and choose AgentCore when you would rather not operate MCP server infrastructure at all.

| Hosting option | Best for | Why |
|---|---|---|
| AWS Lambda | Lightweight, stateless tools with bursty traffic | Serverless, auto-scales, scales to zero, pay-per-use, short bounded execution |
| Amazon ECS on Fargate | Complex, long-running, sessionful or streaming servers | Long-lived containers, warm caches, any runtime, VPC and ALB and WAF integration |
| Amazon Bedrock AgentCore | Teams that do not want to operate MCP infrastructure | Fully managed host or endpoint with built-in identity, memory, tool discovery |
| EC2 or EKS | Maximum self-managed control | Full control of the host and networking when the managed options do not fit |

### AgentCore Gateway as a Managed MCP Endpoint

The problem with remote hosting is that someone still has to run the MCP servers — host them, scale them, secure them, authenticate to them — and that operating work is real. Gateway removes it. It gives an agent a single, unified, secure endpoint to discover and call tools; it turns APIs, AWS Lambda functions, and existing services into agent-compatible (MCP-compatible) tools; and it connects to existing MCP servers. That is precisely the framing the exam uses for Requirement-level recall: AgentCore Gateway removes the need to build and operate MCP server infrastructure directly. You point your agent at Gateway, and Gateway handles the tool plumbing behind it — including the native semantic tool search that solves the context-window problem from the client/server discussion, and multiple authentication methods (IAM, OAuth 2.1, API keys) integrated with AgentCore Identity. The one-line takeaway to carry forward: Gateway is the managed MCP endpoint, and it is the answer whenever a scenario says "we don't want to operate MCP server infrastructure."

### MCP Security

MCP's power — letting an agent reach an open-ended set of tools and data through a standard interface — is also its risk surface, so security is where this section connects forward to Section 7 (safeguarded workflows and excessive agency) and back to Guide 06 (IAM service-role boundaries). The first thing to internalize is that an agentic system has two distinct authorization hops, not one. When a user invokes an agent, the user is authenticated and authorized to the agent. The agent then orchestrates tool calls into one or more MCP servers, and each of those servers in turn accesses some downstream resource. So you must reason about authorization to the MCP servers and authorization from the servers to the downstream resources as separate problems. Collapsing them — assuming that because the user was authorized to the agent, every downstream call is fine — is the root of most MCP security mistakes.

On the question of how the agent authorizes to a server, AWS frames two patterns. Machine-to-machine (M2M) authorization involves no user consent: the agent is pre-authorized to access specified data, which fits system-wide or organizational data and background, scheduled, or batch work — a nightly agent that collects and analyzes logs, for instance. User-delegated access is the opposite: the user is present and consents to the agent accessing user-specific resources, permissions vary per user, and the interaction is real-time. User-delegated access is more complex to implement correctly and should be designed with your security team, because the agent must evaluate which tools the model selected and whether they need additional authorization, the tools must declare their auth requirements, and the client must support intermediate authorization requests. For remote MCP servers specifically, the MCP community — with AWS contributing — standardized on OAuth as the authentication mechanism, chosen so that any MCP client can connect to diverse remote servers without prior coordination. The durable fact for the exam is that OAuth is the standardized authentication approach for remote MCP servers; the specific spec revision that formalized it is a point-in-time detail and not worth memorizing.

The highest-yield security idea in this entire section is least privilege through scoped tokens, and there is a worked example that makes it unforgettable. Do not propagate the user's credentials through the agentic system — an MCP server should not reuse the very token that was used to invoke the agent to then access downstream data. Instead, downstream calls should use explicitly scoped, purpose-generated tokens. Picture a user who happens to hold full administrator permissions asking an agent to clone a production database into a pre-production environment. That task genuinely needs only two privileges: READ (to copy from source) and CREATE (to write the clone). Now suppose the model hallucinates an extra step — it decides it should also "clean up" the old database — and the MCP server is reusing the user's admin credentials, which include DELETE. The destructive delete succeeds, because the credentials allowed it. If instead the server had used an intentionally scoped-down token carrying only READ and CREATE, that same hallucinated delete simply fails, and least privilege has contained the blast radius of the model's mistake. The supporting best practices are token isolation in the same spirit: retrieve a different token per tool rather than reusing one across tools or servers, and validate the audience (`aud`) claim so a token minted for one server cannot be replayed against another. AgentCore Identity (Section 4) is AWS's managed implementation of exactly this token isolation, managing both M2M workload tokens and user-delegated tokens to prevent permission escalation.

This leads to the conceptual heart of MCP security: the tool trust boundary. With a traditional application, the code paths are deterministic — you know in advance which operations the program can perform. An agent breaks that assumption, because it reasons dynamically and chooses which tools and data to use at runtime, so you cannot predict in advance which APIs it will call. The correct design principle therefore is to assume that any permission you grant could be used, and to scope permissions to the acceptable scope of impact rather than to the intended functionality. If a tool's role can call `s3:DeleteObject`, you must assume the agent can delete any object that role can reach, whether or not you intended deletion to be part of the workflow. AWS makes this concrete with four threat scenarios that all live inside the agent's granted permissions: hallucination (the agent misreads the request and takes a wrong action, like deleting "temporary" data that was actually production); prompt injection (crafted external input steers the agent's reasoning into calling unintended APIs); logic errors (flawed reasoning leads to an unintended operation, such as concluding production data is unused and deleting it to "save costs"); and tool poisoning (a compromised MCP server or a tampered dependency performs unintended operations using the agent's own credentials — for example, a modified library that quietly exfiltrates S3 data). Tool poisoning is the MCP-specific supply-chain risk and the strongest argument both for least-privilege scoping and for vetting the MCP servers and dependencies you trust.

Two architectural recommendations follow directly. First, prefer routing an agent's access through MCP servers rather than giving it direct, unconstrained API access (for example through a raw shell or bash tool), because the MCP server provides a layer of abstraction where you can enforce least-privilege IAM, apply differentiation controls, and get monitoring through AWS CloudTrail — controls that simply do not apply when the agent bypasses MCP and hits the API directly. Second, treat the three IAM principles AWS lays out as a checklist: assume all granted permissions could be used and scope to acceptable impact; provide organizational guidance on role usage and enforce it with role governance, session policies, permission boundaries, and service or resource control policies; and differentiate AI-driven actions from human-initiated ones so you can apply different rules to each. On that last point, AWS provides IAM condition context keys that let policies tell agent-initiated calls apart from human ones (and self-managed servers can achieve the same with session tags) — a detail worth recognizing at a high level rather than memorizing for this exam. The connective tissue to keep in view: everything in this security discussion is the MCP-flavored expression of the least-privilege and excessive-agency themes that Guide 06 established for IAM service-role boundaries and that Section 7 develops as excessive agency (OWASP LLM06). MCP is the standardized interface; least privilege at the tool boundary is how you keep that interface safe.

- MCP is an open standard originated by Anthropic; its function is discover tools, execute them, process results.
- MCP standardizes three capability types: tools (callable functions), resources (context data), and prompts (templates).
- MCP is a client/server protocol — the client is the agent or host app; the server is the process exposing tools and data.
- Tool discovery has three modes: static (precise, low context cost), dynamic (loads all tools, high context cost), and semantic search (best at scale).
- Remote MCP authentication standardized on OAuth so any client can reach diverse servers without prior coordination.
- An agentic system has two auth hops: agent-to-server and server-to-downstream-resource — secure both.
- M2M authorization is pre-authorized and user-less (background, batch); user-delegated access needs user consent (interactive, per-user).

- Token isolation: a different token per tool, with `aud` validation, prevents token reuse across servers.
- Four trust-boundary threats: hallucination, prompt injection, logic errors, and tool poisoning (the MCP supply-chain risk).
- Prefer routing agents through MCP servers over direct API access for abstraction, least-privilege IAM, and CloudTrail audit.

### Knowledge Check

**Q1:** A platform team exposes an internal "currency conversion" tool to its agents. Each call takes a few hundred milliseconds, holds no session state between calls, and traffic is spiky — quiet for hours, then hundreds of calls in a burst. Where should they host this MCP server?
- A) Amazon ECS on AWS Fargate as a long-lived service
- B) AWS Lambda
- C) A local stdio subprocess on each user's machine
- D) An Amazon EC2 instance kept running 24/7

**A: B** — The tool is lightweight, stateless, and bursty, which is the canonical fit for AWS Lambda: it scales automatically to the request rate, scales to zero when idle, bills per use, and each short self-contained call needs no server-side session. Fargate (A) and an always-on EC2 instance (D) are built for long-lived, sessionful, or dependency-heavy servers and would pay for idle capacity here. A local subprocess (C) is for a single developer's machine, not a shared internal tool serving many agents.

**Q2:** True or False — Because the user who invoked the agent is a full administrator, it is fine for the MCP server to reuse that user's token when it calls downstream AWS services on the agent's behalf.

**A: False.** This is the central MCP least-privilege violation. You should not propagate the user's credentials through the agentic system; the MCP server should use an explicitly scoped, purpose-generated token rather than reusing the admin token used to invoke the agent. If the model hallucinates or is steered by prompt injection into a destructive action (say a DELETE), an inherited admin token lets it succeed, whereas a token scoped to only the privileges the task needs (for example READ and CREATE) makes that action fail. Scoped tokens contain the blast radius; reused broad credentials remove the containment.

**Q3:** A company wants its agents to reach hundreds of internal APIs and a few existing MCP servers through one secured endpoint with semantic tool selection, and explicitly does not want to build, scale, and secure MCP server infrastructure itself. What should they use?
- A) A fleet of self-managed MCP servers on EC2 behind a load balancer
- B) Local MCP servers distributed to each user
- C) Amazon Bedrock AgentCore Gateway
- D) A single Lambda function per API with no gateway

**A: C** — AgentCore Gateway is AWS's managed MCP endpoint: it gives agents a single unified secure endpoint, turns APIs and Lambda functions into MCP-compatible tools, connects to existing MCP servers, and adds native semantic tool search — removing the need to operate MCP server infrastructure directly, which is exactly the stated requirement. A self-managed EC2 fleet (A) and per-API Lambdas without a gateway (D) leave the team operating and securing the infrastructure. Local servers (B) cannot centralize access for an enterprise agent fleet.

**Q4:** Fill in the blank — Without a standard, connecting M AI applications to N tools requires building and maintaining ___ custom integrations; MCP reduces this to ___ implementations.

**A:** M × N, and M + N. The whole reason MCP exists is to collapse the multiplicative integration matrix into an additive one. With a shared open protocol you build M clients (one per application) and N servers (one per tool or data source), and any new MCP server is immediately usable by every MCP client without bespoke glue code — the same standardizing move that APIs and the Language Server Protocol made in their domains.

**Q5:** What Went Wrong — An agent is granted a tool whose IAM role includes broad S3 permissions "to be safe." A poisoned third-party dependency inside the MCP server later uses the agent's credentials to copy sensitive objects to an external bucket. Which trust-boundary failure is this, and what design principle would have limited the damage?

**A:** This is tool poisoning — a compromised MCP server or dependency performing unintended operations using the agent's own granted credentials, the MCP-specific supply-chain threat. The damage was possible because the tool's role held broad permissions that were never needed for its actual function. The design principle that limits it is to assume all granted permissions could be used and scope permissions to the acceptable scope of impact rather than to intended functionality — combined with vetting MCP servers and dependencies, and routing access through MCP servers so least-privilege IAM and CloudTrail monitoring apply. Granting broad permissions "to be safe" is precisely backwards; least privilege is what contains a poisoned tool. (This is the same excessive-agency theme developed as OWASP LLM06 in Section 7.)

### From One Agent to Many

Every section so far has lived inside the world of a single agent. Section 1 gave you the reason-act-observe loop; Section 2 gave you Amazon Bedrock Agents as the managed embodiment of that loop; Section 3 gave you the raw Converse tool-use contract underneath it; Section 4 gave you the three-layer stack — Bedrock Agents as the managed primitive, an authoring framework like Strands as the layer where you define the agent, and Amazon Bedrock AgentCore as the operating platform where you run it at scale. This section does not re-derive any of that. It asks the next question: what happens when one agent is no longer enough, and how do you coordinate several?

The honest starting point is that a single agent is the right default, and most tasks never need more. But there is a well-documented failure mode that tells you when you have outgrown it. As you keep adding responsibilities and tools to one agent — make it handle billing and shipping and technical support and account changes, each with its own set of tools — the agent slows down, loses track of context across the swelling instructions, makes inconsistent choices, and eventually, in AWS's own framing of the pattern, "collapses under its own weight." The instinctive fix is to patch it: add more prompt text to cover the new case, add more tools, add more fallback logic for when it gets confused. That makes the problem worse, because every addition enlarges the very context and tool surface that caused the confusion. The structural fix is different in kind: split the work across several specialized agents, each with one clear responsibility and a tool set tailored to it, and put a coordinator above them. Splitting work this way is what resolves the context loss, the hallucinations, and the tool overload that a single overloaded agent cannot escape.

That move — from one generalist agent to a coordinated team of specialists — is multi-agent orchestration, and it is the subject of this section. We will look at the open-source authoring side first (the Strands Agents SDK and its multi-agent primitives, then AWS Agent Squad), then the managed Bedrock side (multi-agent collaboration), then the two patterns that underlie all of them (the supervisor/orchestrator pattern and the agents-as-tools pattern), and finally the decision that the exam cares about most: when the cost, latency, and complexity of going multi-agent is actually worth it, and when a single agent is still the better answer.

### The Strands Agents SDK

Strands Agents is an open-source SDK that takes a model-driven approach to building and running AI agents in just a few lines of code, scaling from simple to complex use cases and from local development to production deployment. Section 4 already placed it in the stack as the authoring framework that runs on AgentCore; here we go deeper into what it actually is and what it gives you for multi-agent work.

The simplest definition of a Strands agent is a combination of three things: a model, a set of tools, and a prompt. That maps exactly onto the model-plus-system-prompt-plus-tools shape you have seen throughout this guide — the agent uses those three ingredients to complete a task, often autonomously. The word that distinguishes Strands from older agent frameworks is model-driven. A model-driven framework uses the model itself to dynamically direct the agent's own steps — to plan, to decide when and how to call a tool, to reflect on what happened and try again — rather than requiring the developer to hand-define a complex, fixed workflow up front. Strands explicitly contrasts itself with frameworks that make you author elaborate orchestration graphs by hand; its bet is to lean on the planning and tool-use capabilities of modern models instead. AWS offers a memorable image for this: like the two strands of DNA, Strands connects two core pieces of the agent — the model and the tools — and uses the model's reasoning to plan the next step and execute tools in a loop until the task in the prompt is done. That loop is the same reason-act-observe loop from Section 1, so there is nothing new to learn about the loop itself; what is new is that you express it in code with a model, tools, and a prompt, and the SDK runs it.

Two properties make Strands a natural fit for the themes of this guide. First, it is model-agnostic in the strong sense: it runs with any model that has reasoning and tool-use ability, naming Amazon Bedrock, Anthropic, Ollama for local models, Meta's Llama, OpenAI and others through LiteLLM, plus custom providers — which is exactly the model-flexibility theme Section 8 develops. Second, MCP is a first-class tool source. The tools a Strands agent can use can come from thousands of published MCP servers (Section 5), and Strands also ships more than twenty pre-built example tools and lets you turn any Python function into a tool with a simple `@tool` decorator. So the connection to Section 5 is concrete: an MCP server you stand up to expose a tool is immediately usable by a Strands agent.

The reason Strands belongs in a multi-agent section, though, is its set of multi-agent primitives — the structured ways it lets several agents work together. Strands models multi-agent collaboration as a natural extension of the single-agent model-tool-prompt combination, and it provides four primitives plus two supporting capabilities. The four primitives differ along two axes that the exam rewards you for understanding: how much the developer fixes the flow in advance versus how much the model decides at runtime, and whether the structure allows agents to loop back to each other.

| Primitive | What it is | Flow control | Cycles |
|---|---|---|---|
| Agents-as-Tools | Specialist agents wrapped as callable tools an orchestrator invokes; hierarchical delegation where the orchestrator keeps control | Orchestrator model decides which specialist to call | Delegation, not loops |
| Swarm | Autonomous team of agents that self-organize and hand off to peers through shared memory and context | Sequential and emergent - agents hand off to the best-suited peer | Yes |
| Graph | Developer-defined flowchart of nodes (agents) and edges (transitions); the model picks which edge at each node | Controlled but dynamic - branching, conditional logic, loops | Yes |
| Workflow | Pre-defined task graph (a DAG) executed as one reusable, non-conversational tool | Deterministic and parallel - fixed by the dependency graph | No |

The practical way to choose among them, drawn from Strands' own "when to use each pattern" guidance, is by the shape of the problem. A graph fits a structured process that needs conditional logic, branching, or loops with a deterministic overall flow — routing a customer-support request by intent, or a data-validation pipeline with an explicit error path. A swarm fits a problem you want to break into sub-tasks that benefit from different specialist perspectives coordinating through emergent handoffs — incident response that moves across specialists, or a researcher-to-architect-to-coder-to-reviewer pipeline where the path is not fixed in advance. A workflow fits a complex but repeatable process you want to encapsulate as one reliable, reusable tool — an ETL pipeline, an employee-onboarding sequence — where independent tasks can run in parallel and the dependency graph never changes. And agents-as-tools, which gets its own treatment below, fits hierarchical delegation where one orchestrator consults domain-expert agents without giving up control of the request.

Two supporting capabilities round out the set. Handoffs are provided by a built-in `handoff_to_user` tool that lets an agent explicitly transfer control to a human while preserving the full conversation context — a clean human-in-the-loop hook that Section 7 develops further. And A2A, the open Agent-to-Agent protocol, lets agents from different platforms or organizations discover and communicate with each other: any Strands agent can be wrapped as an A2A server that auto-generates an "agent card" describing its capabilities, and remote A2A agents can be used inside any of the four primitives. This is the same A2A protocol Section 4 noted AgentCore supports, so the two layers line up. Finally, the primitives compose and nest: a swarm can contain graphs, a graph can orchestrate swarms, and any pattern can use agents-as-tools. AWS frames the adoption path as a progression — start with a single agent, add specialists as tools, evolve to swarms, orchestrate with graphs as your needs grow — which is also the right mental model for the single-versus-multi-agent decision later in this section.

[FLAG — point-in-time] Strands launched as a public preview in May 2025 and reached version 1.0, described as production-ready, in mid-2025; the four multi-agent primitives and full A2A support arrived with 1.0 (in the earlier preview, A2A was listed as "coming soon"). Treat the versions, dates, and any adoption counts as snapshots. The durable, exam-relevant facts are the ones that do not move: Strands is an open-source, model-driven SDK; an agent is a model plus a prompt plus tools; it runs the reason-act-observe loop for you; MCP is a first-class tool source; and it offers four multi-agent primitives (agents-as-tools, swarm, graph, workflow) plus handoffs and A2A. Do not describe A2A as "coming soon" — it is supported.

### AWS Agent Squad

AWS Agent Squad is a flexible, lightweight, open-source framework for orchestrating multiple AI agents to handle complex conversations. Its core mechanic is an intelligent intent classifier: the classifier analyzes each user input and dynamically routes the query to the most suitable agent based on the content and the conversation context, maintaining that context as the conversation moves across agents. It ships with pre-built agents and several classifier implementations, supports both Python and TypeScript, and is built for "universal deployment" — it runs anywhere, from AWS Lambda to a local machine to any cloud. Beyond classifier-based routing, it includes a SupervisorAgent for team coordination that implements an agents-as-tools architecture: a lead agent coordinates a team of specialized agents in parallel while maintaining context, and it works with all agent types (Amazon Bedrock agents, Anthropic models, Amazon Lex bots, and more). So Agent Squad gives you two coordination models in one framework — classifier-based routing and a supervisor that treats its team as tools.

### Amazon Bedrock Multi-Agent Collaboration

This is the highest-confidence, most directly testable topic in the section, because it is a managed Amazon Bedrock feature documented in the Bedrock User Guide. Multi-agent collaboration lets multiple Amazon Bedrock Agents collaboratively plan and solve complex tasks. You designate one Bedrock Agent as the supervisor and associate one or more collaborator agents with it, producing a hierarchical collaboration model that returns real-time, synchronous responses. The supervisor uses natural-language instructions to understand each collaborator's role, and AWS's stated best practice is to designate each agent's role and responsibilities clearly and to minimize overlapping responsibilities between collaborators — overlap is what makes a supervisor's routing decisions ambiguous.

The property that ties this back to Section 2 is that every agent in the structure, including the supervisor, retains all the capabilities of an ordinary Amazon Bedrock Agent: action groups, knowledge bases, guardrails, and tools. A collaborator is not a stripped-down worker; it is a full Bedrock Agent scoped to a narrower job. When you invoke the supervisor, it automatically creates and executes a plan across its collaborators, routing requests and sub-tasks to the appropriate collaborator and pulling the pieces together. AWS's worked example is a mortgage assistant: the supervisor classifies an incoming question as being about an existing mortgage, a new mortgage, or a general topic, and routes it to the collaborator that owns that area.

The exam-critical detail is that there are two collaboration modes, and the distinction between them is exactly the kind of fine difference these questions turn on.

| Mode | Behavior | When to use |
|---|---|---|
| Supervisor | The supervisor coordinates the collaborators and synthesizes their contributions into one final response | The answer needs contributions from several collaborators combined |
| Supervisor with routing | The supervisor routes the query to the single most appropriate collaborator, which produces the final response | One specialist can fully answer; this mode reduces latency |

The memory hook is worth stating crisply because the names are so similar: plain Supervisor means synthesize — the supervisor gathers and combines what several collaborators return. Supervisor with routing means direct to one — the supervisor sends the query to a single collaborator and that collaborator answers, which AWS notes reduces latency precisely because the supervisor is not waiting on and combining multiple responses. If a scenario stresses combining specialist contributions, it is Supervisor mode; if it stresses sending a query to the one right specialist with the lowest latency, it is Supervisor-with-routing.

The setup flow and limits are worth recognizing. You create and deploy each collaborator agent scoped to a specific task, create or designate a supervisor and choose the Supervisor or Supervisor-with-routing mode, associate the collaborators' alias versions with the supervisor, prepare and test, then deploy and invoke the supervisor through the same `InvokeAgent` call you would use for any Bedrock Agent. You must save the supervisor agent before enabling multi-agent collaboration, then update it to associate collaborators. An optional "enable conversation history" setting lets the supervisor share the full context of the current session with a collaborator. AWS also surfaces an integrated trace and debug console for following the supervisor's plan and the inter-agent communication, which connects to the agent-tracing material in Section 2 and Guide 06. [FLAG — point-in-time] A supervisor can have a maximum of ten collaborator agents at this time, and the feature reached general availability in March 2025 (after a December 2024 preview). Treat the count of ten and the GA date as snapshots; the durable facts are the supervisor-and-collaborators model and the two modes.

[FLAG — conflation risk, not a contradiction] One terminology trap deserves a direct callout, because two different AWS sources use "multi-agent collaboration" in two different senses. The managed Bedrock feature described here is a supervisor-coordinated hierarchy — a central supervisor that plans and delegates, which is closer to a centralized "workflow agent" style of coordination. Separately, AWS Prescriptive Guidance uses "multi-agent collaboration" to name a broader pattern of peer-to-peer, emergent coordination among autonomous agents (through shared memory and messaging), which it explicitly contrasts with centralized workflow agents. When the exam says "Bedrock multi-agent collaboration," picture the supervisor and its collaborators.

### The Supervisor Pattern and the Agents-as-Tools Pattern

Underneath both the open-source frameworks and the managed Bedrock feature sit two patterns that are really two views of the same idea, and seeing them as one will save you from a lot of distractor confusion.

The supervisor (or orchestrator) pattern is the architectural shape: a primary coordinating agent handles user interaction and decides which specialized agent to invoke, creating a clear chain of command that mimics human team dynamics, where a manager coordinates specialists. Bedrock multi-agent collaboration is the managed embodiment of this pattern; Agent Squad's SupervisorAgent and a Strands orchestrator are the open-source embodiments. The pattern answers "who is in charge" — the supervisor — and "how is the work divided" — among role-scoped specialists.

The agents-as-tools pattern is the mechanism by which a supervisor reaches its specialists. In this pattern a specialized agent is wrapped as a callable function — a tool — that another agent can invoke. That creates a hierarchical structure: a primary orchestrator agent handles user interaction and decides which specialized agent to call, and the specialized "tool agents" perform their domain-specific tasks when called. The defining nuance is that the orchestrator delegates without giving up control of the request — it calls a specialist, gets the specialist's output back, and synthesizes the final response itself, rather than handing the whole conversation off. The benefits AWS attributes to the pattern are the same ones that motivated splitting the overloaded single agent at the start of this section: separation of concerns (each agent has a focused responsibility and is easier to understand and maintain), hierarchical delegation (the orchestrator decides which specialist to invoke, giving a clear chain of command), modularity (specialists can be added, removed, or modified independently), and improved performance (each agent gets a system prompt and tool set tuned to its narrow task).

The connection that makes this click is to Section 3. An "agent as a tool" is exposed to the orchestrator using exactly the same tool-use contract you already learned: a `toolSpec` with a name, a description, and an input schema. The orchestrator's model selects it the same way it selects any other tool, through the Converse tool-use mechanism. So "agents-as-tools" is not a new wire protocol — it is a sub-agent presented to a coordinating agent as an invokable capability, indistinguishable at the contract level from a calculator or a database-lookup tool. That is why a specialist agent and an ordinary tool can sit side by side in the same orchestrator's tool list. Strands implements the pattern in a few ways (passing agents directly in the `tools` array, an `.as_tool()` helper, or wrapping the sub-agent call in a `@tool`-decorated function), and the AWS Generative AI Application Builder's Workflow Builder uses the same "agents as tools" pattern to create a supervisor that autonomously selects which specialized agents to invoke — but the durable idea is the single sentence: a sub-agent exposed to a coordinating agent as an invokable tool.

### Single Agent Versus Multiple Agents

This is the decision the exam tests, and the discipline it rewards is restraint: a single agent is the default, and multi-agent is an escalation you justify, not a starting point you assume. The guidance AWS and Strands give is explicit — start with a single agent, add specialists as tools, evolve to swarms, orchestrate with graphs as your needs grow. Multi-agent is where you go when a single agent genuinely cannot cope, not where you begin.

The case for multiple agents rests on three observable signals. The first is the single-agent overload failure mode from the opening of this section: when one agent juggling too many responsibilities and tools starts losing context, making inconsistent choices, and hallucinating, splitting the work across specialized agents — each with one clear responsibility — resolves the overload in a way that more prompt text and more fallback logic cannot. The second is genuine specialization: when a task decomposes into sub-tasks that need different domains of expertise, or different task modalities entirely (structured API calls in one place, messy browser navigation in another), separating them into purpose-built agents keeps each agent's logic clean and reduces the hallucination risk that comes from forcing one model to switch between fundamentally different kinds of work. The third is the set of system-level benefits that fall out of specialization: parallel processing across agents, clearer audit trails because each agent's actions are scoped and traceable, and more focused analysis than a single general-purpose model can provide.

The case against multiple agents is the cost you take on the moment you add the second agent, and it has three dimensions the exam likes to pair together. Cost: each agent is its own model invocation, often running several loop iterations of its own, so coordinating several agents multiplies token consumption compared with one agent doing the work. Latency: inter-agent communication adds round-trips, and fanning a query out to multiple collaborators and waiting to synthesize their answers is slower than a single response — which is precisely why Bedrock's Supervisor-with-routing mode exists, to cut latency by sending a query to one collaborator instead of combining many. That feature is direct evidence from AWS that multi-agent fan-out has a latency price worth engineering around. Complexity: more agents means more roles to define (and the standing best practice of minimizing overlapping responsibilities to keep routing unambiguous), more failure modes, and harder debugging and monitoring. Strands' own orchestration guidance names this plainly — multi-agent introduces the architectural challenge of orchestrating the flow of information between agents, and without proper orchestration the interactions become unpredictable and the system becomes difficult to debug, monitor, and scale in production.

So the decision rule is a balance. Reach for a single agent when one model with a focused tool set can handle the task — it is the lowest cost, the lowest latency, the lowest complexity, and the right default. Reach for multiple agents when the task spans distinct specializations or modalities, benefits from parallelism, needs clear separation and auditability, or when a single agent is demonstrably overloaded — and accept the higher cost, latency, and orchestration complexity as the price of solving a problem the single agent could not. This is the same "grant the least agency and complexity the task requires" instinct from Section 1, applied one level up: do not grant your architecture more agents than the problem demands.

One last choice sits on top of the single-versus-multi decision: once you have decided to go multi-agent, do you use the managed path or a code-level framework? Prefer Amazon Bedrock multi-agent collaboration when you want the fastest managed route and your agents live inside Bedrock — supervisor plus collaborators, all Bedrock Agents, no orchestration code to operate. Prefer an open-source framework (Strands, or Agent Squad) when you need code-level control of the coordination, non-Bedrock models or agent types, or portability across environments — and remember from Section 4 that you then operate that framework somewhere, often on AgentCore. Managed native feature versus open-source framework you run is the axis, and it is the same axis that separated Bedrock Agents from AgentCore one section ago.

## Section 7: Safeguarded Agentic Workflows

### Why Agents Need Safeguarding

Everything that makes an agent valuable is also what makes it dangerous. Section 1 established the trade precisely: an agent is a foundation model placed in a loop where the model itself chooses what to do at each step, and you buy that flexibility by surrendering predictability. The model decides how many iterations to run, which tools to call, and what arguments to pass — at runtime, based on what it observes. Section 1 deferred the full treatment of how you bound that loop to this section, Section 2 flagged user confirmation as a human-in-the-loop control point it would "return to" here, and Section 5 closed by naming the least-privilege and excessive-agency themes as the connective tissue this section develops. This is where all of those promises come due.

The single named risk that ties the section together is excessive agency, which AWS and the security community track as OWASP LLM06. The intuition is simple and it is exactly the autonomy cost from Section 1 viewed through a security lens: because the model drives the control flow and "has little knowledge beyond the prompt as to what behaviors are permitted," an agent can take actions beyond its intended purpose. AWS is emphatic that this is usually not malicious — it is an unintended consequence of automation. An agent asked to clean up a dataset may decide the most thorough solution is to delete records you never meant it to touch. Every additional autonomous iteration is another opportunity for the model to take a wrong, possibly irreversible, real-world action. Safeguarding an agentic workflow is therefore the discipline of capping the blast radius of bad reasoning — bounding what the agent can do, what it can touch, how long it can run, and whether a human stands between its decisions and the consequential actions.

The rest of this section develops five layered controls that each cap a different axis of agency: bounded loops (stopping conditions, timeouts, maximum iterations, circuit breakers), least-privilege IAM boundaries, scoped tools, human-in-the-loop approval, and deterministic orchestration through AWS Step Functions, backed by an emergency shutdown. But before the controls, you need the mental model that explains why there is no single "correct" amount of safeguarding — the Agentic AI Security Scoping Matrix.

### The Agentic AI Security Scoping Matrix

The cleanest organizing frame for this section comes from the Agentic AI Security Scoping Matrix, which AWS published as the agentic successor to the Generative AI Security Scoping Matrix that Guide 06 referenced. [FLAG — point-in-time] This framework is recent (published late 2025); the four-scope model and the agency-versus-autonomy distinction are durable teaching content, but treat the specific control lists as evolving guidance rather than a fixed specification.

Its most testable contribution is a distinction the exam can exploit: agency and autonomy are not the same thing. Agency is the scope of actions an agent is permitted and enabled to take — which systems, which operations, which resources. It is about capabilities and permissions, what the agent is allowed to do, and it is governed by boundaries and permission systems, which is to say least privilege. Autonomy is the degree of independent decision-making the agent exercises without human intervention — when it acts, how it chooses, whether it needs approval. It is about independence, and it is governed by oversight mechanisms, which is to say human-in-the-loop and monitoring. The two dimensions are orthogonal: an agent can have high agency and low autonomy (it can touch many systems but must get a human's approval for each consequential action), or low agency and high autonomy (it acts entirely on its own but only within a narrow, read-only scope). You manage them with different controls, and conflating them is precisely the kind of imprecision a distractor is built on.

The matrix then arranges agentic systems into four scopes of increasing autonomy, each demanding more of the safeguards in this section. A useful way to feel the difference is a single example — a calendar assistant — carried across all four.

| Scope | Name | Autonomy and oversight | Defining control | Calendar example |
|---|---|---|---|---|
| Scope 1 | No agency | Human-initiated, read-only, no change execution | Boundary enforcement, fixed timeouts and resource limits per step | Reads calendars and suggests times; the human books manually |
| Scope 2 | Prescribed agency | Human-initiated, can change the environment but every consequential action needs human approval | Robust approval workflows, signed time-bounded approval tokens, MFA for approvers | Recommends a time and asks permission to send the invite |
| Scope 3 | Supervised agency | Human triggers, then the agent acts autonomously within bounded parameters | Real-time monitoring, automated kill switches for runaway processes, clear execution boundaries | Books the best slot itself, with no per-action approval |
| Scope 4 | Full agency | Self-initiating from environmental triggers, minimal human oversight during execution | Behavioral anomaly detection, tamper-proof human override, failsafes, graceful degradation | Another agent triggers it to auto-book a follow-up, with no human in the loop |

Two insights from the matrix are worth carrying into the controls below. The first is that oversight does not disappear as you move from Scope 1 to Scope 4 — it shifts focus. In the low scopes a human spends effort instantiating, reviewing, and approving individual actions; in the high scopes that per-action effort gives way to auditing, validating behavior against expected patterns, and running anomaly detection and kill switches. The second is the pattern AWS calls graceful degradation: a well-designed agentic system automatically reduces its own autonomy when a security event is detected — injecting more human approval, shrinking the allowed action set, or disabling the agent entirely — and you build those agentic safety mechanisms in from the start rather than bolting them on. This connects directly to circuit breakers below and to the graceful-degradation treatment in Section 8. The durable takeaway echoes Section 1's "grant the least agency the task requires": the right amount of safeguarding is not fixed, it scales with the scope you are operating in.

### Bounding the Loop: Stopping Conditions

The most fundamental safeguard answers the question Section 1 left open: how does the loop end, and what happens when the model fails to end it cleanly? The natural, clean stop is the model deciding the goal is satisfied and emitting a final answer — the end-turn stop reason from Section 1. Stopping conditions are the guards for every other case, and AWS treats them as both a cost control and a reliability control, rating the risk of omitting them as High in both pillars of the Well-Architected Generative AI Lens.

The cost argument is the more memorable one. Agentic workflows can be long-running and call external resources, which makes their cost unpredictable and, without bounds, unbounded — every extra iteration spends more tokens and more tool invocations. The durable design principle from the cost pillar is that the maximum cost of an agent's runtime should be predictable from the stopping conditions you implement. You make cost estimable by first estimating the maximum time the agent legitimately needs (accounting for model response times, tool execution times, and network latency) and then implementing conditions that let it run only up to that bound. A stopping condition can live in several places: a timeout mechanism such as the one Amazon Bedrock provides, logic in the prompt-flow layer, or logic in a software abstraction layer around the agent. The cost pillar also recommends setting token limits on model responses, which simulates timeout behavior by halting long-running generations, and evaluating asynchronous, event-orchestrated workflows, because an async design creates natural points at which you can interrupt or halt a process that has run too long.

The reliability argument explains why timeouts matter even when cost is not the concern. Agents call external systems that may themselves run long, producing idle agents that consume resources and block while they wait — and runaway execution loops are a named failure mode. Timeouts free those consumed resources. AWS describes two complementary timeout approaches. The first is runtime timeouts on the external infrastructure itself: if the agent reaches a tool through an action group, put a timeout on that function, and set it to cover the maximum allowable completion time plus a margin for edge cases like Lambda cold starts, rounding up so you do not terminate legitimate work prematurely. The second is an asynchronous, event-driven architecture in which the compute behind an action group publishes events, giving owners visibility into where an agent stalls and the ability to act on it. The reliability pillar's implementation checklist is worth holding as a pattern: define maximum runtime thresholds; set timeout controls at both the function and the workflow levels; add timeouts at the agent layer to terminate sessions waiting on user input; configure dead-letter queues for timed-out processes; monitor execution times and alert on timeout frequency; and define graceful recovery with cleanup routines. One subtle but important rule: error handling must be transparent to the user about next steps without exposing sensitive internal information.

It helps to keep three related controls distinct, because the exam tests the difference. A timeout bounds a single call or a single loop — "this one operation took too long, stop it." Maximum iterations bounds the number of reason-act-observe cycles — "the loop has run too many times, stop it" — and it is the loop-count analogue of a timeout. [FLAG — no single verified default] AWS documents the practice of bounding iterations and capping tokens, but does not publish one canonical "default maximum iterations = N" value for Bedrock Agents; treat maximum iterations as a control you configure for the task, not a fixed number. A circuit breaker is different from both: it bounds repeated failures across many calls. AWS describes the circuit-breaker pattern (implemented, for example, with Step Functions and DynamoDB) as monitoring calls to a dependency and, when failures or timeouts exceed a threshold, "opening the circuit" to stop routing requests to the failing dependency — returning immediate failures instead of letting timeouts cascade and exhaust resources, then probing periodically to detect recovery. The mental model: timeout means "this call is taking too long"; maximum iterations means "the loop has run too many times"; circuit breaker means "this tool or model keeps failing, so stop calling it and fail fast or degrade." The circuit breaker is the control that connects forward to Section 8's graceful degradation.

[FLAG — point-in-time figure] The 15-minute maximum execution duration commonly cited to motivate action-group and tool timeouts is the AWS Lambda limit; it is fine to cite it as the Lambda limit, but phrase tool timeouts qualitatively where a specific number is not the point.

Finally, hold on to why bounded loops belong in a safeguards section and not only a cost-optimization one. Because the model controls the loop, an unbounded agent does not merely waste money and add latency — every uncontrolled iteration is another chance to take a wrong, irreversible action. Bounding the loop caps the blast radius of bad reasoning, which is exactly the excessive-agency concern that frames the whole section.

From the reasoning loop, four control branches fan out — "Stopping conditions," "Least-privilege role," "Scoped tools — allow-list," and "Human approval." The picture conveys that before any action runs, it must pass through stopping conditions, a least-privilege role, scoped tools, and human approval, and that an emergency shutdown sits over the whole thing as a last resort.

### IAM Resource Boundaries and Least Privilege

If bounding the loop caps how long and how many times an agent can act, least privilege caps what its actions can ever touch — and it is the primary, AWS-named mitigation for excessive agency. The authoritative guidance is the Well-Architected best practice "implement least privilege access and permissions boundaries for agentic workflows," and it starts from a fact about the AWS object model that Guide 06 introduced: agents have execution roles, and Bedrock Flows have service roles. Those roles are where least privilege is enforced, and AWS singles out as especially important the roles that grant access to data sources (such as Amazon Kendra) and to compute (such as the AWS Lambda functions behind action groups). The guiding principle mirrors Section 5's tool trust boundary exactly: for any given request, the role should provide access only to the systems, guardrails, and data sources necessary to generate the response — and where multiple models or agents service one prompt, least privilege should be shared across them.

The agent-specific techniques layer on top of the Bedrock IAM mechanics Guide 06 already established — the control-plane versus data-plane split, the minimal inference action set, scoping a policy to a specific foundation-model ARN, validating with IAM Access Analyzer, and enforcing organization-wide guardrails with service control policies. Section 7 adds three refinements aimed at the agent's own role. First, scope the execution-role policy to the intended resource ARNs for each permitted API action, and consider IAM conditions that restrict use to trusted traffic — for example, requests originating from a specific VPC. Second, apply a permissions boundary to the role. A permissions boundary is an IAM feature that sets the maximum permissions a role can ever receive, so even if someone later attaches a broader policy, the role cannot be inadvertently granted access to additional services; boundaries can be managed at the account level. Third, enforce separation of duties by giving distinct tasks distinct roles — the prompt engineer building the agentic workflow should operate under a different IAM role than the security engineer who creates the agent's service role. That logical separation is itself an excessive-agency reduction, because no single actor both shapes the agent's behavior and grants its permissions. AWS recommends documenting these permissions-boundary and policy decisions in the organization's AI policy document with clear instructions for changing them as requirements evolve.

This is the same lesson Section 5 taught with the clone-database example, now applied to the agent's own role rather than a downstream tool: assume that any permission you grant could be used, and scope to the acceptable scope of impact rather than to the intended functionality. A worked pattern from the OWASP-mitigation guidance makes it concrete at the tool layer — apply least-privilege IAM to the action-group Lambda role, and pass a `user_id` down to the plugin layer so the tool enforces user-level, data-plane access control. AWS frames this as "wrapping deterministic access controls around a non-deterministic LLM," which is the essence of safeguarding: you do not trust the model to police itself, you constrain it with controls that hold regardless of what it reasons. For workloads that need fine-grained, per-user agent permissions evaluated at runtime, Amazon Verified Permissions can serve as an external authorization layer in front of Bedrock Agents — a useful pointer rather than a required AIP-C01 detail.

### Human-in-the-Loop

Least privilege caps what the agent can touch; human-in-the-loop caps whether a consequential action runs at all without a person's explicit consent. In the scoping matrix this is the defining control of Scope 2, prescribed agency — the agent can gather information, analyze, and prepare recommendations, but every action of consequence requires human approval. The design rule for where to place it follows from Section 1's risk-tiered deployment approaches: insert human approval at high-impact, irreversible, or sensitive actions — writes, deletes, payments, external communications, production changes — and leave read-only steps unguarded so you do not destroy the agent's value with needless friction. AWS gives you two managed mechanisms to implement it, and the exam expects you to pick the right one for the context.

The first is native Amazon Bedrock Agents user confirmation, the agent-primitive HITL that Section 2 flagged. You enable confirmation per action within an action group (through the console, CLI, or SDK). When the agent decides to call that action, the `InvokeAgent` response does not execute it — instead it returns the elicited API or function details together with the confirmation requirement in the `returnControl` / `invocationInputs` field, carrying a unique invocation identifier. Your application presents the user with a CONFIRM or DENY choice, and returns the decision in a follow-up `InvokeAgent` call via `sessionState.returnControlInvocationResults`, where each result carries a `confirmationState`. CONFIRM lets the action execute; DENY prevents it. AWS frames the purpose explicitly as a safeguard against prompt injection: if a tool or knowledge-base response between iterations contains hidden instructions, the agent may comply with them, including instructions to invoke unintended actions — requiring confirmation ensures certain actions only execute after explicit human consent. Keep user confirmation distinct from return of control (Section 2): return of control hands execution of the action to your application, whereas user confirmation gates whether the action runs in the first place. They are complementary and can compose.

The second mechanism is the AWS Step Functions human-approval pattern that Guide 06 established for content-moderation workflows, and it is the right answer whenever the requirement is that a workflow must pause, wait for a person, and then resume. Step Functions offers three service-integration patterns: Request Response (proceed as soon as the call returns), Run a Job with `.sync` (wait for a job to complete), and Wait for Callback with a task token (`.waitForTaskToken`). The third is the HITL mechanism. A task configured with `.waitForTaskToken` pauses the execution and emits a token; a human is notified out of band (an SNS email, an API Gateway approve/reject link); and the workflow resumes only when your code calls `SendTaskSuccess` or `SendTaskFailure` with that token — approve versus reject. The exam-relevant nuance is a workflow-type constraint: Standard workflows support Wait for Callback, but Express workflows support only Request Response, so a human-approval step that must wait for a callback requires a Standard workflow. This is exactly Guide 06's observation that a Standard Step Functions workflow supports the human-approval step natively. The Scope 2 security guidance rounds it out: protect the approval channel itself with time-bounded approval tokens that expire, MFA for approvers, and validation of human-provided input against injection, so the approval gate cannot be bypassed or replayed.

**Diagram (described):** This is a left-to-right flow of the Step Functions human-approval pattern. "Workflow step" leads to "Emit task token," which leads to "Pause — wait for callback," which leads to "Human reviews." From the human-review step, two branches: an "approve" branch goes to "SendTaskSuccess," and a "reject" branch goes to "SendTaskFailure." SendTaskSuccess then leads to "Resume — execute action," while SendTaskFailure leads to "Halt or compensate." The picture conveys that the workflow pauses on a token, waits for a human decision, and resumes or halts depending on approve versus reject.

[FLAG — point-in-time] `.waitForTaskToken` is supported with services such as Lambda, SNS, SQS, EventBridge, API Gateway, ECS/Fargate, and Amazon Bedrock `InvokeModel`; the optimized Step Functions integration with Amazon Bedrock AgentCore currently supports only Request Response (no `.sync`, no `.waitForTaskToken`) with a 15-minute maximum. AgentCore is fast-moving (see Section 4), so treat that integration's limits as a snapshot.

### ReAct and Orchestration via AWS Step Functions

The final control reframes the loop itself. There are two ways to "implement ReAct." The first you already know: use a managed Amazon Bedrock Agent, where ReAct is built in — AWS documents that the default orchestration strategy is ReAct (Reason and Action), which uses the foundation model's tool-use patterns, and you can override it with advanced prompt templates or a custom-orchestration Lambda for complex multi-step workflows. The second is to hand-orchestrate the reason-act-observe cycle yourself as explicit state-machine states in AWS Step Functions, trading the agent's adaptive autonomy for deterministic, fully traceable control. Step Functions is, in fact, the "deterministic orchestration" Section 1 contrasted agents against.

AWS Prescriptive Guidance gives a clean decision table for choosing between the two, and it is high-yield because it crystallizes the autonomy-versus-control tradeoff that runs through the whole guide.

| Use case | Step Functions (rule-based) | Bedrock Agents (AI-native) |
|---|---|---|
| Deterministic workflow | Ideal | Not needed |
| Unstructured user input | Rigid | Interprets and adapts |
| Complex business rules | Modeled with explicit conditions | Inferred by semantic reasoning |
| Fine-grained audit trail | Full state trace | Limited trace, mitigate with model-invocation logging |
| Latency-sensitive automation | Real-time coordination | Slightly higher from model processing |
| Goal-directed experience | Requires explicit design | Agent infers the goal and composes the flow |

The tradeoff to internalize: a managed agent gives you autonomy and flexibility at the cost of predictability and a complete audit trail, while Step Functions gives you deterministic, auditable, bounded control — built-in retries, timeouts, error handling, parallel branches, and a full state trace of every step — at the cost of having to design the flow explicitly, because it will not adapt to inputs it was not built for. The implication for safeguarding is direct: for high-risk, regulated, or audit-heavy steps, prefer the explicit control of Step Functions; for open-ended, ambiguous, natural-language goals, prefer the agent's autonomy. Hybrids are common and are exactly Section 1's LLM-augmented-workflow pattern — a deterministic Step Functions flow that calls the model to make decisions only at specific, bounded points. Encoding ReAct in Step Functions looks like a model-invoke task, a tool-invoke task, and a choice state that evaluates the result and either loops or finishes, with each step individually retryable and an optional `.waitForTaskToken` human-approval gate inserted before any consequential action.

Two concrete properties make Step Functions the safeguarded end of the agency spectrum. Standard workflows run for up to a year and can pause for callbacks, which suits long-running agentic orchestration and human approval, while Express workflows run up to five minutes and suit short, high-volume work. And the service supports recoverable, auditable patterns — AWS provides a ready prompt-chaining sample project that sequences Bedrock model calls for higher-quality output, and a saga orchestration pattern that manages retries, parallel branches, timeouts, and compensating (rollback) transactions when a step fails. [FLAG — fast-moving] A newer integration lets Step Functions invoke an Amazon Bedrock AgentCore managed harness as a workflow step, embedding agent reasoning inside a deterministic workflow with tracing, parallel execution, and human-approval capabilities; it is emerging and currently Request-Response only, so mention it lightly and do not over-anchor on specifics.

### Excessive Agency (OWASP LLM06) and the Mitigation Stack

With the five controls in hand, the named risk they all serve comes into focus. AWS's Well-Architected guidance states it directly: excessive agency is an OWASP Top 10 threat for LLMs that is typically introduced through agentic architectures, because agents are designed to take action on behalf of a user and the risk is that an agent could take actions beyond its intended purpose. OWASP decomposes the root cause into three axes, and AWS's mitigations map onto them one-for-one: excessive permissions (the agent's role can touch more than it should), excessive functionality (the agent has access to more or broader tools than the task needs), and excessive autonomy (consequential actions run without human confirmation). Naming the three axes is the key to answering LLM06 questions, because each control in this section attacks a specific axis.

AWS's agentic-AI security guidance maps a concrete control list to the LLM06 row, and the two highest-yield entries are worth depth. The first is "use deterministic execution logic unless AI is needed" — implement coded logic for anything deterministic and reserve the model for where it genuinely adds value. Minimizing the AI's scope yields faster execution, lower cost, better testability, simpler threat modeling, and a smaller attack surface that limits prompt-injection impact. Critically, where a model response drives tool selection or interfaces with another system, use an allow-list of permitted tools; unrestricted tool access can lead to data exfiltration or confused-deputy and privilege-escalation scenarios. An allow-list is the direct mitigation for excessive functionality. The second is "determine agent scoping," which is the security-lens version of Section 6's single-versus-multi-agent decision. A super-agent (one agent with many tools) simplifies development and entitlements but concentrates risk; a disparate-worker approach (several narrowly scoped agents) reduces the chance of a sub-agent doing something the orchestrator did not intend, shrinks the blast radius if one agent is compromised, narrows the exposure point for access control, and improves explainability — which is why organizations with lower risk tolerance favor it.

The mitigation list also includes an incident-response control the exam likes to probe under the phrase "stop a runaway agent": establish emergency shutdown capabilities for high-risk scenarios. This is the kill switch — an emergency response process that can immediately shut a system down, roll back to a stable version, disable functionality, or move to a safe mode when a threat is detected. AWS pairs it with operational observability to detect unauthorized actions, business-continuity planning, and automated recovery, and it flags an agent-specific hazard: if the AI fails, human operators may lack the skills to run the workload manually, so you must keep fallback staff and processes ready.

Putting it together gives the section's thesis — the five controls are a layered defense, each capping a different axis of excessive agency:

- Least privilege and permissions boundaries cap excessive permissions — what the agent's role can touch.
- Scoped tools, allow-lists, agent scoping, and deterministic logic cap excessive functionality — which tools the model can reach.
- Human-in-the-loop and user confirmation cap excessive autonomy — whether consequential actions run without a human.
- Bounded loops (stopping conditions, timeouts, maximum iterations, circuit breakers) cap runaway execution.
- Deterministic orchestration through Step Functions, plus emergency shutdown and observability, caps overall agency and gives an auditable, recoverable control plane.

This is also where the section closes the loop with Section 5. Section 5 taught the tool trust boundary — assume all granted permissions could be used; scope to acceptable impact, not intended functionality — and its four threat scenarios: hallucination, prompt injection, logic errors, and tool poisoning. Excessive agency (LLM06) is the named OWASP umbrella for exactly those risks, and least-privilege scoping at the tool boundary is the shared mitigation. Section 5 secured the interface to the tools; this section secures the agent that drives them. Together they express one idea: never trust the model to constrain itself — wrap deterministic controls around it so that no matter what it reasons, the damage it can do is bounded in advance.

### Exam-Relevant Distinctions

| If the exam says... | The answer is... | Why |
|---|---|---|
| "Agent keeps looping, never terminates, runaway cost" | Stopping conditions - max iterations, timeout, token limits | Bounds cost, latency, and safety; makes max cost predictable |
| "A dependency keeps failing and calls are piling up" | Circuit breaker - open the circuit, fail fast | Bounds repeated failures, distinct from a per-call timeout |
| "One call is taking too long, end it" | Timeout | A timeout bounds a single call or loop, not repeated failures |
| "Constrain what an agent's actions can touch" | Least-privilege execution role plus permissions boundary | Caps excessive permissions, scope to resource ARNs |
| "Cap the maximum permissions a role can ever receive" | Permissions boundary | It sets the ceiling even if a broader policy is attached |
| "Limit which tools the model may select" | Allow-list plus agent scoping plus deterministic logic | Caps excessive functionality, prevents confused-deputy |
| "Require sign-off before an action inside a Bedrock Agent" | Native user confirmation - CONFIRM or DENY per action | Managed agent-primitive HITL, also defends against injection |
| "Workflow must pause for a human, then resume" | Step Functions Standard plus wait-for-callback task token | Express cannot wait for a callback |
| "Deterministic, fully auditable, retryable control flow" | AWS Step Functions | Trade agent autonomy for predictability and a full state trace |
| "Open-ended, ambiguous, natural-language goal" | Bedrock Agent - ReAct, AI-native | Interprets and adapts; Step Functions would be rigid |
| "Name the risk of an agent acting beyond its purpose" | Excessive agency - OWASP LLM06 | Mitigate with least privilege, scoped tools, HITL, bounded loops |
| "Immediately stop a misbehaving production agent" | Emergency shutdown / kill switch | Roll back, disable, or move to safe mode (incident response) |
| "Agent can do many things but needs approval for each" | High agency, low autonomy - Scope 2 prescribed agency | Agency is permissions, autonomy is independence - distinct |

- Excessive agency has three axes: excessive permissions, excessive functionality, and excessive autonomy; the controls map one-to-one.
- Agency is the scope of permitted actions (governed by least privilege); autonomy is independence from human intervention (governed by oversight).
- Bedrock Agents' default orchestration strategy is ReAct; override it with advanced prompt templates or a custom-orchestration Lambda.
- Native user confirmation returns the decision through `confirmationState`; CONFIRM executes the action, DENY blocks it.
- A permissions boundary sets the maximum permissions a role can receive — it does not grant permissions, it caps them.
- Standard Step Functions workflows support wait-for-callback (and run up to a year); Express workflows support only Request Response (up to five minutes).
- Separation of duties: a different IAM role for the prompt engineer than for the security engineer who creates the agent's service role.
- Disparate narrowly-scoped worker agents reduce blast radius versus a single super-agent with many tools.
- Maximum iterations is a control you configure for the task, not a fixed AWS default.
- Use deterministic logic unless AI is needed, and an allow-list wherever a model response selects a tool.

### Knowledge Check

**Q1:** An autonomous agent in production occasionally gets stuck reasoning in circles, calling the same tools repeatedly and never returning a final answer. Cost spikes unpredictably and some sessions never end. Which combination of controls most directly addresses this?
- A) A larger foundation model and a longer system prompt
- B) Stopping conditions — maximum iterations, timeouts, and token limits
- C) A Bedrock Guardrail with denied topics
- D) Cross-Region Inference for more capacity

**A: B** — This is the unbounded-loop failure mode, and the direct remedy is the set of stopping conditions: cap the number of reason-act-observe cycles (maximum iterations), put timeouts on the agent layer and the tools it calls, and set token limits on responses. AWS frames these as both a cost control (the maximum cost of a run should be predictable from the stopping conditions) and a reliability control (timeouts free resources and stop runaway execution loops). A bigger model or longer prompt (A) does not bound the loop. A Guardrail (C) filters content, not loop length. Cross-Region Inference (D) adds capacity, which would make the runaway cost worse, not better.

**Q2:** True or False — Because agency and autonomy mean the same thing, granting an agent fewer permissions is the same control as requiring human approval for its actions.

**A: False.** Agency and autonomy are distinct dimensions, and that is a likely exam trap. Agency is the scope of actions an agent is permitted and enabled to take — its capabilities and permissions — and you constrain it with least privilege and permissions boundaries. Autonomy is the degree to which the agent acts without human intervention — its independence — and you constrain it with oversight mechanisms like human-in-the-loop and monitoring. Reducing permissions (lowering agency) does not require a human to approve actions, and requiring approval (lowering autonomy) does not narrow what the agent's role can touch. A real design manages both independently — an agent can have high agency with low autonomy (Scope 2: it can do many things but needs approval for each).

**Q3:** A claims-processing workflow must pause when a payout exceeds a threshold, wait for a human adjuster to approve or reject, and only then continue. The team is using Express Step Functions workflows for everything. What must change, and which integration pattern implements the pause?
- A) Nothing — Express workflows support callbacks natively
- B) Switch the approval workflow to Standard and use Wait for Callback with a task token
- C) Use Request Response so the workflow proceeds immediately
- D) Replace Step Functions with a single Lambda timer

**A: B** — Pausing indefinitely until a human responds is the Wait for Callback with Task Token pattern (`.waitForTaskToken`): the task emits a token and pauses, the human is notified, and the workflow resumes only when `SendTaskSuccess` or `SendTaskFailure` is called with that token. The catch is the workflow type — Standard workflows support Wait for Callback, but Express workflows support only Request Response, so the approval workflow must be a Standard workflow. Request Response (C) would not wait for the human at all, and a Lambda timer (D) cannot model an approve/reject decision with an audit trail.

**Q4:** Match the excessive-agency axis to its mitigation. (a) Excessive permissions, (b) Excessive functionality, (c) Excessive autonomy.

**A:** (a) Excessive permissions is mitigated by least-privilege IAM and permissions boundaries on the agent's execution role — scope what the role can touch to specific resource ARNs and cap it with a boundary. (b) Excessive functionality is mitigated by scoped tools and allow-lists (and agent scoping plus deterministic logic) — restrict which tools the model may select so it cannot reach capabilities the task does not need. (c) Excessive autonomy is mitigated by human-in-the-loop — native Bedrock Agents user confirmation or a Step Functions wait-for-callback gate — so consequential actions do not run without a person's approval. OWASP LLM06 excessive agency is the umbrella, and these three axes plus bounded loops and an emergency shutdown are the layered defense.

**Q5:** What Went Wrong — To move fast, a team gave their agent's execution role broad permissions "so it never gets blocked," gave it every internal tool, and let it act with no approvals. During a demo the agent, steered by text hidden in a retrieved document, deleted a production table. Identify the OWASP risk and the controls that would have prevented it.

**A:** This is excessive agency (OWASP LLM06), and the incident hit all three axes at once. Excessive permissions: the broad execution role could reach and delete production data — a least-privilege policy scoped to the needed resource ARNs, plus a permissions boundary, would have made the delete fail (the same lesson as Section 5's clone-database example). Excessive functionality: handing the agent every tool widened the attack surface — an allow-list of only the tools the task needs would have removed the destructive capability. Excessive autonomy: no human stood between the model's decision and a high-impact action — native user confirmation (which AWS specifically positions as a defense against prompt injection driving unintended actions) or a Step Functions approval gate would have required a CONFIRM first. The trigger itself — hidden instructions in retrieved content — is the prompt-injection threat from Section 5's tool trust boundary, and an emergency shutdown capability would have let the team stop the agent the moment it misbehaved.

## Section 8: Model Flexibility & Resilience

### Two Halves of One Exam Pattern

This section is the agentic face of exam Pattern 5, and Pattern 5 has two halves that the exam tends to test as a pair. The first half is "switch the underlying foundation model without changing code," and the second is "keep the workload running when a model, a tool, or a Region has a bad day." Both halves rest on the same instinct you have been building since Section 1 — never hardwire a decision the system might need to change at runtime — and both are best understood as questions about where, in your architecture, the brittle assumptions live.

A deliberate scoping note before we go deep. The non-agentic, infrastructure-level treatment of these topics lives in two neighboring guides, and this section cross-references rather than duplicates them. Guide 01 owns the full mechanics of Cross-Region Inference, the Converse API, and application inference profiles — the source-and-destination-Region routing, the real-time capacity checks, the cost accounting, and the SCP and IAM rules. Guide 05 (Enterprise Integration & Deployment, file 08) owns the broader deployment and enterprise-integration story. What belongs here, and only here, is the agentic angle: how model flexibility and resilience apply inside an agent's reasoning loop, across its tools and MCP servers, and over a multi-agent topology where latency and failure compound. Read this section as "Pattern 5, but for the reasoning core and the tools the agent drives."

### Switching Models Without Code Changes: A Ladder of Abstraction

Start with the trap, because the exam almost always frames Pattern 5 around it. The trap is hardcoding a single model ID — or a single Region — directly in your application or agent code. It looks harmless in a prototype: you pick the best model available, paste its identifier into the call, ship it. The cost arrives later. You are now coupled to that one model's availability, its pricing, and its capacity, and the only way to move to a different model — because a cheaper one launched, because a faster one fits your latency budget, because the one you chose is throttling under load — is to edit code, test it, and redeploy. Every model switch becomes a deployment with deployment risk, which in practice means switches happen slowly or not at all. The whole point of the techniques below is to make changing the model a configuration decision, not a code change.

The cleanest way to hold the solution is as a ladder of abstraction, where each rung removes a different kind of hardcoding. There are three rungs for the model choice and a fourth for the Region.

The first rung is the model-agnostic Converse API at the code layer. Section 3 already taught Converse's defining property in the context of tool use: it "provides a consistent interface that works with all models that support messages," so you write your integration once and run it against different models by changing a model ID rather than rewriting the call. Section 8 simply cashes that property in for model selection overall. You select the model at invocation time by passing a `modelId` in the request, and that `modelId` can name either a foundation model or an inference profile. Switching models is therefore, at the code layer, a single changed value — the request shape, the message format, the way you read the response all stay identical. That is what "model-agnostic" buys you: the code stops caring which model answers.

The second rung is an indirection layer at the configuration layer, and this is the architecture the exam most wants you to recognize for "switch models without redeploying code." Put an abstraction in front of Amazon Bedrock — an AWS Lambda function fronted by Amazon API Gateway — so that callers never address a model directly; they call your endpoint, and the Lambda decides which model to invoke through Converse. Then store the active model choice as external configuration in AWS AppConfig rather than as a constant in the Lambda's code. To change models, you update the value in AppConfig; the Lambda reads the new configuration and routes to the new model on its next invocation, with no code deployment at all. Guide 01 carries the full version of this pattern with its diagram; here the point is why AppConfig specifically, rather than a hardcoded constant or even a plain environment variable. AWS AppConfig is a feature of AWS Systems Manager built for exactly this — using feature flags and dynamic configuration to change application behavior in production without a full code deployment — and it adds three things a bare constant cannot: validators that reject a malformed or disallowed configuration before it ships, deployment strategies that roll the change out gradually instead of all at once, and automatic rollback driven by a CloudWatch alarm if the new configuration starts causing errors. When the "configuration" in question is which foundation model serves production traffic, those safety rails matter: a bad model switch can be caught and reverted automatically rather than taking down every request. (The Lambda needs only modest permissions to read the configuration — the AppConfig data-plane actions for starting a configuration session and getting the latest configuration.)

The third rung is provider abstraction at the framework layer, which Section 6 already established for Strands. The Strands Agents SDK is model-agnostic in the strong sense: it runs with any model that offers reasoning and tool-use ability — Amazon Bedrock, Anthropic, Ollama for local models, Meta Llama, and OpenAI and others through LiteLLM, plus custom providers. Swapping the model behind a Strands agent is a provider-and-configuration change in the framework, not an agent rewrite. This is the framework-layer counterpart to Converse's code-layer portability: where Converse makes one Bedrock call model-agnostic, a model-agnostic framework makes the whole agent portable across providers.

The fourth rung concerns the Region rather than the model, and it is Cross-Region Inference, covered next. Naming a system-defined inference profile as your `modelId` instead of a single-Region model is the move that stops you from hardcoding one Region the same way Converse stops you from hardcoding one model.

**Diagram (described):** This diagram contrasts the trap with the fix. On one side, "Hardcoded model and Region" leads to "Trap — redeploy to switch." On the other side, "Externalize the choice" branches into four rungs of abstraction: "Converse modelId — code" (the code layer), "Strands provider — framework" (the framework layer), "AppConfig — config layer" (the configuration layer), and "Inference profile — Region" (the Region layer). The picture conveys that hardcoding forces redeploys, while externalizing the model and Region choice across these four layers turns a switch into a configuration change.

There is a governance counterpart worth naming because the exam sometimes phrases model flexibility as a management problem rather than a code problem. The Well-Architected Generative AI Lens, in its Reliability pillar, recommends standardizing resource management through centralized catalogs for prompts and models — a single source of truth that enables version control and clean updates or rollbacks, reducing the risk of an application quietly running an outdated or inappropriate model. That is the same idea as the AppConfig rung viewed from the governance side: do not scatter model choices through your code, manage them centrally so you can see, version, and revert them.

### Cross-Region Inference: Redundancy for the Agent's Reasoning Core

Guide 01 treats Cross-Region Inference in full, so this section states only the minimum needed and then adds the agentic framing the exam expects here. The mechanism: Cross-Region Inference routes a model request across multiple AWS Regions to increase throughput and absorb traffic bursts, working through an inference profile that defines a foundation model plus the set of Regions a request can route to. On-demand requests that would otherwise be throttled by a single Region's quotas or peak load get spread across compute in several Regions. Bedrock performs the routing for you — it does a real-time capacity check, first in the source Region and then in other Regions in the profile if needed — which means you do not write any client-side load-balancing logic. The price is based on the source Region you call from, with no additional routing cost, and data stays on the AWS network and is encrypted in transit between Regions. Two scopes exist: a geographic profile routes only within a defined geography (US, EU, APAC) and is the right answer when data residency or compliance constrains where inference may run, while a global profile lets Bedrock pick the optimal commercial Region worldwide for the highest throughput. One operational rule the exam likes: because requests can land in any destination Region in the profile, your SCPs and IAM policies must permit the model-invocation actions in all of those destination Regions — if a single destination Region is blocked by an SCP, the request fails even when other Regions would have worked.

Now the agentic framing, which is the net-new contribution here. For an agent, the foundation model is the reasoning core — it is what decides every step of the loop — and that makes it a potential single point of failure. If the agent can only reach one model in one Region and that endpoint has trouble, the whole agent stops thinking, not just one feature. AWS positions Cross-Region Inference precisely as the managed answer to this: when you use managed models, Bedrock's Cross-Region Inference provides built-in redundancy for supported models, automatically routing requests to alternative Regions when the primary endpoint has issues. So in the agentic context Cross-Region Inference is doing double duty. It is a capacity control — agents make many model calls per task, and a recursive or multi-agent workload can burst hard, exactly the bursty pattern cross-Region routing was built to absorb. And it is an availability control — it gives the agent's reasoning core redundancy across Regions and mitigates the single-point-of-failure risk, without you building or operating a multi-Region failover system yourself. (If instead you self-manage models — for example hosting an open-weights model on your own infrastructure — you take on building that multi-Region deployment and automated failover yourself; the managed path hands it to Bedrock.)

Treat the specific numbers around Cross-Region Inference as point-in-time snapshots rather than durable facts. The roughly ten-percent cost difference sometimes quoted for global profiles, and the question of whether a given profile's destination Region list is fixed or grows as AWS adds Regions, are both the kind of detail that shifts over time and varies by profile. What is durable, and what the exam actually rewards, is the shape: geographic equals fixed-geography routing for residency, global equals route-to-any-commercial-Region for maximum throughput and availability, no extra routing cost, source-Region pricing, and SCP and IAM must allow every destination Region. Anchor on the shape, not the snapshot.

### Circuit Breakers and Graceful Degradation as Model and Tool Fallback

Section 7 already taught the circuit-breaker mechanics in depth, so there is nothing new to learn about the mechanism — there is something new to do with it. Recall the mental model: a timeout bounds a single call ("this call is taking too long"), a maximum-iterations limit bounds the loop ("the loop has run too many times"), and a circuit breaker bounds repeated failures across calls ("this tool or model keeps failing, so stop calling it"). When failures or timeouts against a dependency cross a threshold, the breaker opens, the system fails fast instead of letting timeouts pile up and exhaust resources, and it probes periodically to detect recovery. Section 7 left a deliberate forward reference here, because the circuit breaker is not the end of the story — it is the trigger. What happens when the breaker opens is graceful degradation, and graceful degradation in an agentic workload means falling back to a model or a tool that still works.

Graceful degradation has a precise definition worth memorizing: maintain core functionality while advanced features become unavailable. The system does not fail entirely when something breaks; it does a simpler or less capable version of the job. In an agent there are three natural fallback directions, and the exam draws scenarios from all three.

The first is tool fallback. AWS frames this as routing from a failed tool or MCP server to an alternative that provides similar functionality while maintaining consistent schemas — what it calls contract-compatible fallbacks. The orchestration layer should coordinate tool redundancy so that when a primary tool becomes unavailable the agent automatically routes to an alternative that does a similar job, even if that alternative is less sophisticated, rather than failing the whole task. The canonical example: an internal chat assistant whose knowledge base is down can fall back to a plain search tool — a worse answer than a curated retrieval, but a useful answer instead of an error. There is a security rule attached to this that the exam can test obliquely: the end-user permissions and tool-level access rights must be identical between the primary and the failover component, so the failover path does not quietly open a privilege-escalation gap. The fallback tool must be scoped just as tightly as the one it replaces — the least-privilege lesson from Sections 5 and 7 follows the agent down the degraded path.

The second is model fallback, and its rationale is the latency-versus-predictive-power tradeoff. Within a model family, the most capable model usually carries the highest latency, while a lower-capability model in the same family responds faster — Claude Opus versus Claude Haiku is the standard illustration. AWS makes the counterintuitive point directly: the best model for a workload is often not the one with the best predictive power, but the one that trades some predictive power for lower latency. When the primary model is unavailable or overloaded, falling back to a smaller, faster model in the same family keeps the agent answering, at some cost in answer quality. This matters most in multi-agent designs, because, as Section 6 established, multi-agent latency is the sum of every agent's latency — a slow model on each hop compounds, so a faster fallback on each hop compounds in your favor too.

The third direction is Region fallback, which is just Cross-Region Inference reappearing as a degradation path: when the model in the primary Region has trouble, route the reasoning to another Region. Because Cross-Region Inference does this automatically through the inference profile, it is the lowest-effort fallback of the three — you get it by naming a profile instead of a single-Region model.

**Diagram (described):** This diagram shows the fallback decision. "Agent reasoning loop" leads to a decision node "Primary model or tool ok." If healthy, flow goes to "Use primary model and tool." If failing (circuit open), flow goes to "Fall back," which branches into three fallback directions: "Smaller faster model," "Alternative tool — degraded," and "Other Region — CRIS" (Cross-Region Inference). All three fallback paths, plus the healthy "Use primary" path, converge on "Return best available answer." The picture conveys that when the circuit opens, the agent degrades gracefully along one of three fallback directions rather than failing outright.

Two more agentic-resilience details round out the picture. The first is capacity-driven degradation. Agents can consume capacity in ways a request-response service cannot, so AWS recommends setting a recursion limit to prevent redundant loops and runaway resource consumption — the capacity-facing analogue of Section 7's maximum-iterations safety control — and managing maximum completion tokens, because an agent's output feeds the next recursion and unbounded output compounds. Under sustained high load, the same graceful-degradation instinct applies: use intelligent request queuing with priority-based allocation so essential work keeps flowing, and temporarily shed non-essential features while preserving the core. The second is retry discipline, which has a sharp edge the exam likes. Retrying with exponential backoff and jitter is correct for transient failures, but when failures are caused by overload, naive retries make the overload worse — each retry adds load to a system that is already saturated. The disciplined answer is a token bucket that caps the retry rate, with tokens refilling slowly and in step with successful completions, so the system backs off instead of hammering a struggling dependency.

It helps to see all of this as one framework rather than a grab-bag of tactics. AWS organizes generative-AI resilience around five properties — redundancy, sufficient capacity, timely output, correct output, and fault isolation — and every control in this section maps onto one of them. Redundancy is Cross-Region Inference for the model and contract-compatible alternatives for the tools. Sufficient capacity is cross-Region burst absorption plus recursion limits and queuing. Timely output is the smaller-faster-model fallback and disciplined retries. Correct output is the Guardrails, input validation, and evaluation work from Guides 02 and 06. Fault isolation is the circuit breaker and per-tool containment with schema validation at the boundaries. The through-line back to Section 1 and Section 7 is exact: resilience and safeguarding share the same instinct — when something goes wrong, degrade rather than fail, and reduce what the agent attempts rather than letting it spin. Pattern 5's second half is that instinct applied to outages.

### Knowledge Check

**Q1:** A team hardcoded a single model ID directly in their agent's Lambda. They now want to A/B different models and switch quickly if one starts throttling, without a code deployment each time. Which architecture best delivers this?
- A) Add a second hardcoded model ID and an if/else on a build flag
- B) Front Bedrock with a Lambda that calls Converse, and store the active model ID in AWS AppConfig
- C) Provision Throughput for the current model so it never throttles
- D) Move the model ID into a Lambda environment variable

**A: B** — This is the canonical "switch models without redeploying code" pattern. A Lambda calling the model-agnostic Converse API makes the invocation independent of which model answers (Converse needs only a different `modelId`), and storing the active model in AWS AppConfig externalizes the choice so a switch is a configuration update — with validation, staged rollout, and automatic CloudWatch-alarm rollback — rather than a code deployment. A build flag (A) still requires a redeploy to flip. Provisioned Throughput (C) addresses capacity for one model, not flexibility across models. An environment variable (D) is better than a literal but still requires a deployment to change and gives you none of AppConfig's validation or rollback safety.

**Q2:** True or False — Because Cross-Region Inference routes requests to other Regions automatically, you can leave your SCPs allowing model invocation in only your primary Region.

**A: False.** Cross-Region Inference can route a request to any destination Region defined in the inference profile, so your SCPs and IAM policies must allow the model-invocation actions in every one of those destination Regions. If even a single destination Region is blocked by an SCP, the request fails — even when other Regions in the profile would have served it. This is a frequent trap: the routing is automatic, but the permissions are not, and a too-narrow Region condition silently breaks the very redundancy you added.

**Q3:** An internal assistant agent normally answers from a Bedrock Knowledge Base. During an incident the Knowledge Base is unavailable and calls to it keep timing out, stalling the agent. Which pair of techniques keeps the assistant useful, and what is the security caveat?
- A) A larger model and a longer timeout
- B) A circuit breaker on the Knowledge Base plus graceful degradation to an alternative tool (for example a search tool)
- C) Cross-Region Inference for the Knowledge Base
- D) Retrying the Knowledge Base with no backoff until it responds

**A: B** — The circuit breaker detects the repeated failures and opens, so the agent fails fast instead of piling up timeouts, and graceful degradation routes to a contract-compatible alternative — a plain search tool that returns a less curated but still useful answer, preserving core functionality while the advanced retrieval feature is offline. The security caveat: the failover tool's end-user permissions and access rights must be identical to the primary's, so degrading does not open a privilege-escalation gap. A bigger model and longer timeout (A) make the stall worse. Cross-Region Inference (C) is for the model, not for an unavailable tool. Retrying with no backoff (D) is exactly the wrong move when a dependency is struggling.

**Q4:** A multi-agent workflow is breaching its latency budget. Accuracy is comfortably above the requirement, and profiling shows the foundation model calls dominate the time. What is the most direct lever, and why does it help more here than in a single-agent design?

**A:** Fall back to a smaller, faster model in the same family (the Opus-to-Haiku move) — trade some predictive power, which you have to spare, for lower latency. The reason it helps disproportionately in a multi-agent design is that multi-agent latency is the sum of every agent's latency: shaving time off each hop compounds across all the hops, whereas in a single-agent design you only save on one call. AWS makes the underlying point explicitly — the best model for a workload is often not the most capable one, but the one that trades some predictive power for lower latency.

**Q5:** What Went Wrong — Under a traffic spike, an agent's model calls start failing. The team's retry logic immediately retries each failure with exponential backoff and jitter, but the situation gets worse, not better, and the model endpoint stays saturated. Diagnose the flaw and the fix.

**A:** Backoff with jitter is the correct discipline for transient failures, but these failures are caused by overload, and under overload naive retries amplify the problem — every retry adds more load to an already-saturated endpoint, so the system retries itself deeper into the hole. The fix is to cap the retry rate with a token bucket: retries draw from a slowly refilling bucket whose tokens replenish in step with successful completions, so when success rates fall the system automatically backs off instead of hammering the dependency. Pairing that with a circuit breaker (fail fast once failures cross a threshold) and graceful degradation (fall back to a smaller model or another Region via the inference profile) keeps the workload serving instead of collapsing.

Two exam patterns live here in concentrated form. Pattern 3 — "build an autonomous agent that calls tools" — is the spine of this entire guide, and its two-sided trap (choosing an autonomous agent when something simpler fits, or choosing an agent correctly and then hand-rolling orchestration a managed primitive already provides) is the single most common way agentic questions are written to mislead. ### The Agentic Decision Tree

The fastest way to get an agentic question right is to answer four nested decisions in order. Resist the temptation to jump to a service name; the early decisions frequently eliminate the agent entirely, and the exam rewards the simplest construct that satisfies the requirement.

The first decision — taught in Section 1 — is whether you need an agent at all. Characterize the task before you reach for any tool. A self-contained transformation such as summarizing, classifying, or rewriting needs no external actions and no multi-step reasoning, so a single prompt is the right answer. A question that must be answered from a private corpus through one grounded generation is a fixed retrieve-then-generate flow, which is a RAG pipeline (Guide 02), not an agent. A highly standardized, rule-based, high-volume process belongs to traditional automation or robotic process automation, because agentic reasoning is wasted on a flow that never needs to adapt. Only a multi-step task whose sequence depends on intermediate results — one that must decide at runtime what to do next and call tools to do it — actually warrants an agent. AWS frames the gate as three questions: assess the task (is it complex and non-standardized enough?), weigh the risk and impact, and check the return on investment, because an agent is the most expensive option per task.

**Diagram (described — the build decision tree):** This diagram begins at "Characterize the task," which branches into four task shapes: "One-shot transform," "Answer from a corpus," "Fixed rule-based flow," and "Multi-step and adaptive." Each shape maps to a construct: one-shot transform leads to "Single prompt"; answer from a corpus leads to "RAG pipeline — Guide 02"; fixed rule-based flow leads to "Automation or RPA"; and multi-step and adaptive leads to "Build an agent." The "Build an agent" branch then splits into four build paths: "Managed inside Bedrock" leads to "Bedrock Agents"; "Any framework at scale" leads to "AgentCore platform"; "Code-level loop control" leads to "Strands framework"; and "Raw self-managed loop" leads to "Converse tool use." The picture conveys that only the adaptive multi-step task warrants an agent, and once you need one, the build path depends on whether you want a managed Bedrock primitive, an at-scale operating platform, a code-level framework, or the raw Converse loop.

The second decision — drawing on Sections 2, 3, 4, and 6 — is how to build the agent once you have decided you need one. If you want the fastest managed agent that lives entirely inside Bedrock, with action groups, an orchestration loop, and native memory, and you accept the managed orchestration, the answer is Amazon Bedrock Agents, the managed primitive. If you already have an agent built on an open-source or custom framework (LangGraph, CrewAI, Strands) and any model, and you need to run it in production at scale with session isolation, managed memory, secure tool access, and tracing without a rewrite, the answer is Amazon Bedrock AgentCore, the operating platform. If you want code-level control of the loop and portability across providers, the answer is an open-source authoring framework such as Strands, which you then operate somewhere (often on AgentCore). And if you want the raw self-managed loop with no framework at all, the answer is client-side Converse tool use, where your application owns the detect-run-return-reinvoke cycle. The critical framing — and a frequent trap — is that these are three layers of one stack (managed primitive, authoring framework, operating platform), not three competing products: "Strands versus AgentCore" and "should we migrate off Bedrock Agents to AgentCore" are false choices.

The third decision — from Section 6 — is whether one agent is enough. The default is always a single agent, because it carries the lowest cost, latency, and complexity; multi-agent is an escalation you justify, never a starting point. You go multi-agent only when a single agent collapses under its own weight (context loss, inconsistent choices, tool overload), when there is genuine specialization across domains or modalities, when parallelism gives a real benefit, or when separation buys auditability. If you do go multi-agent, the managed path is Bedrock multi-agent collaboration (a supervisor coordinating collaborator agents, all Bedrock Agents), and the code-level path is Strands or Agent Squad (for non-Bedrock models and mixed agent types, with portability). Within Bedrock multi-agent collaboration, Supervisor mode synthesizes answers from several collaborators, while Supervisor-with-routing sends the request to a single best-fit collaborator for lower latency.

**Diagram (described — single versus multi-agent):** This diagram starts at "Agent chosen" and asks "Single agent overloaded." Two branches: "No — context fits" leads to "Single agent — default"; "Yes — context loss" leads to "Need specialists." The "Need specialists" node splits by deployment style: "All Bedrock Agents" leads to "Multi-agent collaboration," and "Mixed or portable" leads to "Strands or Agent Squad." The "Multi-agent collaboration" node then splits into the two modes: "Supervisor — synthesize" and "Routing — one specialist." The picture conveys that a single agent is the default unless it is overloaded, and that going multi-agent forks first by whether everything is Bedrock-native versus mixed/portable, then by Supervisor mode versus Supervisor-with-routing.

The fourth decision — synthesizing Sections 1 and 7 — is orthogonal to the others and often the deciding factor in a question: how much autonomy should the control flow have? When the goal is open-ended, ambiguous, expressed in natural language, and the path must be discovered at runtime, a Bedrock Agent running the ReAct loop is the AI-native answer, because it interprets and adapts. When the workflow is deterministic, governed by complex explicit business rules, latency-sensitive, regulated, or needs a fine-grained audit trail, AWS Step Functions is the better answer, because it gives deterministic, retryable, fully traceable control flow. The hybrid — an LLM-augmented workflow — is a deterministic Step Functions flow that calls the model only at specific bounded points, with a wait-for-callback task token inserted as a human-approval gate before any consequential action. The trade is autonomy and flexibility (the agent) against predictability and auditability (Step Functions).

### Common Exam Traps and Distractors

Distractors in this domain are rarely random. They are constructed from the handful of confusions below, and once you can name the confusion you can usually eliminate two options on sight. These are the traps worth rehearsing.

**The Pattern 3 two-sided trap (over-agency and reinvention).** This is the most important trap in the guide because it cuts both ways. The first form is choosing a fully autonomous agent when a single prompt, a RAG pipeline, or a deterministic workflow would do the job more cheaply and more predictably — "agentic" sounds advanced, so it tempts you. The second form is the mirror image: you correctly decide an agent is warranted, then hand-roll the orchestration loop, tool wiring, memory, and safeguards on raw model calls, when Amazon Bedrock Agents already provides all of it. The exam rewards the simplest construct that fits and the managed primitive over a bespoke build, so when you see "agent," check first whether you even need one, and second whether you are being steered into rebuilding something managed.

**AWS Agent Squad is not a managed AWS service.** The name pairs "AWS" with the word "Squad" and questions often add "managed" nearby, but Agent Squad is a lightweight open-source framework (formerly the Multi-Agent Orchestrator) that has moved out of AWS Labs to the `2fastlabs/agent-squad` home, and you operate it yourself. The managed, native multi-agent feature on AWS is Amazon Bedrock multi-agent collaboration. If a scenario wants a managed supervisor-and-collaborator capability, that is Bedrock multi-agent collaboration, not Agent Squad.

**AgentCore Gateway versus the AgentCore MCP Server.** The two share the "AgentCore" and "MCP" words and sound interchangeable, but they operate at different times. AgentCore Gateway is the runtime managed MCP endpoint that an agent calls to reach tools. The AgentCore MCP Server is a build-time open-source developer aid that coding assistants (Kiro, Q Developer, Claude Code) use to scaffold and deploy agents. A scenario about an agent reaching tools at runtime points to Gateway; a scenario about a coding assistant helping you build an agent points to the MCP Server.

**Multi-strategy memory is AgentCore, not native Bedrock Agents.** Both describe "long-term memory," which is the bait. Semantic, summarization, and episodic memory strategies belong to AgentCore Memory, a separate service. Native Bedrock Agents long-term memory is SESSION_SUMMARY only. The strategy names are the tell — when you see them, the answer is AgentCore.

**Agency and autonomy are different dimensions.** Everyday speech blurs them, and the exam exploits that. Agency is the scope of actions an agent is permitted to take, and it is governed by least privilege and permissions boundaries. Autonomy is the degree of independence from human intervention, and it is governed by human-in-the-loop oversight. An agent can be high-agency and low-autonomy at the same time. Reducing what an agent is allowed to do (permissions) is not the same control as requiring a human to approve it (oversight), so a question asking how to stop an agent from taking a damaging action has a different answer (scope the role) than one asking how to keep a human in command of a consequential decision (add an approval gate).

**Hardcoding one model ID or one Region (the Pattern 5 trap).** Embedding a single model identifier or a single Region in the agent couples you to that model's availability, price, and capacity, and forces a redeployment to change anything. The fix has two halves: the model-agnostic Converse API addressed by `modelId`, externalized through a Lambda plus AppConfig indirection layer so a switch is a configuration change rather than a code deploy; and Cross-Region Inference so the reasoning model has built-in redundancy and burst capacity. Note the related sub-trap: Cross-Region Inference only works if your SCPs and IAM permit model invocation in every destination Region — one blocked Region fails the request.

**The OpenAPI and post-processing gotchas in Bedrock Agents.** Two small but heavily tested facts. Action-group OpenAPI schemas must be exactly version 3.0.0 — not 3.0.1 and not 3.1 — and you constrain allowed values in a field's `description` because `enum` is unsupported. Separately, post-processing is disabled by default, so any question that assumes the agent always reformats its final answer is wrong unless post-processing was explicitly enabled.

**Reusing the user's credentials downstream, or registering every tool.** In MCP security, propagating the user's (often broad) token through the agent to downstream calls means a hallucinated destructive action succeeds because the credentials allowed it; the fix is scoped, purpose-generated tokens carrying only the privileges the task needs (for example READ and CREATE, never DELETE). The parallel scaling trap is registering every tool from every server, which drains the context window a few hundred tokens at a time; the fix is semantic tool selection, which AgentCore Gateway provides natively. And in client-side tool use, remember the model never executes the tool — on a tool-use stop reason your application runs it and returns the result.

### Service-Selection Quick Reference

The five constructs below are the ones the exam most often asks you to tell apart. The discriminator is purpose, not capability — several of them can be made to do overlapping things, so the question is which one is built for the job the scenario describes. The memory hook ties them together: Bedrock Agents is the managed primitive, AgentCore is the operating platform, Strands and Agent Squad are open-source authoring and orchestration that you run yourself, and Step Functions is the deterministic control plane.

| Service or construct | What it is | Primary purpose | Reach for it when |
|---|---|---|---|
| Amazon Bedrock Agents | Managed agent primitive inside Bedrock - action groups, orchestration loop, native memory | The fastest managed agent fully inside Bedrock | You accept managed orchestration and the agent lives in Bedrock |
| Amazon Bedrock AgentCore | Framework- and model-agnostic operating platform - Runtime, Gateway, Memory, Identity, Observability, Code Interpreter, Browser | Deploy and operate any-framework, any-model agents at scale | Moving an open-source or custom-framework agent from POC to production securely |
| Strands Agents | Open-source, model-driven SDK - model plus prompt plus tools, with multi-agent primitives | Author the agent in code with provider portability | You want code-level loop control and portability; it runs on AgentCore |
| AWS Agent Squad | Open-source, not managed, multi-agent orchestrator - intent classifier plus supervisor | Code-level multi-agent routing across mixed agent types | You need classifier-based routing or mixed agent types and operate it yourself |
| AWS Step Functions | Deterministic state-machine orchestration | Explicit, auditable, retryable control flow and human gates | Deterministic workflows, regulated steps, or hybrid LLM-augmented flows |

### Agentic Troubleshooting Quick Reference

Domain 2 owns building agents, but when an agent misbehaves in production the question crosses into Domain 5 (Monitoring, Optimization, and Troubleshooting). The four symptoms below are the ones the exam names, each mapped to its likely root cause, the fix developed earlier in this guide, and where you would look to diagnose it. The diagnostic column is the Domain 5 connection: the agent trace (OrchestrationTrace and FailureTrace events), CloudWatch Logs and Logs Insights, X-Ray spans surfaced through AgentCore Observability or CloudWatch Transaction Search, and the Converse `stopReason` are the instruments that turn a vague "the agent is broken" into a specific, fixable cause.

| Symptom | Likely root cause | Fix | Diagnostic - where to look |
|---|---|---|---|
| Agent loops without terminating, sessions never end, runaway cost | Unbounded reasoning loop with no stopping condition | Stopping conditions - maximum iterations, timeouts at the function, workflow, and agent layers, token limits, a recursion limit | Agent trace OrchestrationTrace rationale; CloudWatch execution-time and timeout metrics; DLQ for timed-out runs |
| Tool invocation fails or the agent will not advance | Malformed tool call, missing Lambda resource-based policy, bad action-group schema, wrong OpenAPI version or use of enum | Fix the schema - OpenAPI 3.0.0, no enum, description-driven; add a resource-based policy for the Bedrock principal; inspect the stop reason | Agent trace FailureTrace and InvocationInput; Converse stopReason; CloudWatch Logs |
| Excessive permissions - the agent does more than intended | Excessive agency (OWASP LLM06) - broad execution role, every tool granted, no approval gate | Least-privilege role scoped to resource ARNs plus a permissions boundary; tool allow-list and agent scoping; human-in-the-loop confirmation; a kill switch | IAM Access Analyzer; CloudTrail with AI-versus-human differentiation; agent trace; scoped-token audience checks |
| Model unavailable or throttling | Hardcoded single model or Region, or an FM capacity burst | Cross-Region Inference profile for redundancy and burst; circuit breaker into graceful degradation; token-bucket retry under overload | CloudWatch model-invocation metrics and alarms; circuit-breaker state; Bedrock Model Invocation Logs |

### Knowledge Check

**Q1:** A retail team is asked to add a feature that classifies inbound support emails into one of five categories and rewrites them into a standard internal format. There are no external lookups and no branching on results. A colleague proposes "an agent with tools." What is the best response, and which exam pattern is in play?
- A) Build an Amazon Bedrock Agent with an action group per category
- B) Use a single prompt - it is a self-contained transform, not an agentic task
- C) Build a multi-agent system with one classifier agent and one rewriter agent
- D) Use AgentCore to operate the classifier at scale

**A: B** — Classifying and rewriting with no external actions and no runtime branching is a self-contained transformation, which a single prompt handles. Proposing an agent here is the over-agency half of the Pattern 3 trap: "agent" sounds advanced, but the task never needs to adapt or call tools, so the agent adds cost, latency, and risk for no benefit. A and C build agentic machinery the task does not need; D operates an agent you should not have built. The first decision in the tree — do you need an agent at all — eliminates the agent before service selection even begins.

**Q2:** True or False — A team running Amazon Bedrock Agents in production should migrate to Amazon Bedrock AgentCore because AgentCore is the newer and more capable service.

**A: False.** Bedrock Agents and AgentCore are complementary layers of one stack, not competitors, and there is no forced migration. Bedrock Agents is the managed agent primitive inside Bedrock (action groups, orchestration loop, native memory); AgentCore is the framework- and model-agnostic platform for operating agents of any framework and model at scale. AgentCore is the right move when you need to run an open-source or custom-framework agent in production without a rewrite — not as a mandatory upgrade from a working Bedrock Agent. "Newer means you must switch" is the trap.

**Q3:** Scenario — A bank needs a workflow that gathers loan documents, runs explicit eligibility rules, and must pause for a human underwriter's approval before disbursing funds, with a complete, retryable audit trail of every step. Which architecture fits, and why not a fully autonomous agent?
- A) A fully autonomous Bedrock Agent with a disbursement action group
- B) AWS Step Functions Standard with a wait-for-callback task token for the approval gate
- C) A Strands multi-agent system with a supervisor
- D) A single prompt with a long system message describing the rules

**A: B** — The requirements are the signature of deterministic orchestration: explicit business rules, a mandatory human-approval gate, and a fine-grained retryable audit trail. AWS Step Functions Standard provides deterministic, traceable control flow, and its wait-for-callback service-integration pattern (the `.waitForTaskToken` token) pauses the workflow until the underwriter responds, then resumes. A fully autonomous agent (A) trades away the predictability and auditability a regulated, high-impact flow demands and could disburse funds without the gate. A multi-agent system (C) adds nondeterminism and complexity. A single prompt (D) cannot orchestrate a multi-step approval workflow at all. Note that Express Step Functions cannot do wait-for-callback — it must be Standard.

**Q4:** Fill in the blanks — To switch foundation models without a code deployment, call the model-agnostic ___ API addressed by ___, and externalize the active model through a Lambda plus AWS ___ layer; to give the reasoning model built-in redundancy and burst capacity, use ___.

**A:** the Converse API, addressed by `modelId`, externalized through a Lambda plus AWS AppConfig layer, with Cross-Region Inference for redundancy and burst capacity. This is the full fix for the Pattern 5 trap of hardcoding one model or Region: Converse makes the invocation independent of which model answers, AppConfig turns a model switch into a validated, roll-back-able configuration change rather than a redeploy, and Cross-Region Inference adds managed multi-Region capacity for the foundation-model core. Remember the sub-trap: SCPs and IAM must permit model invocation in every destination Region, or a single blocked Region fails the request.

**Q5:** What Went Wrong — An internal "database operations" agent was given an execution role with broad permissions for convenience, and the MCP server reuses the invoking user's admin token for downstream calls. Asked to clone a production database into pre-production, the agent also deleted the source database, citing "cleanup." Diagnose the failure and the fix, and name the distinction the question turns on.

**A:** This is excessive agency (OWASP LLM06) compounded by credential propagation. The clone task needed only READ and CREATE, but because the MCP server reused the user's admin token (which includes DELETE), the model's hallucinated "cleanup" step succeeded and destroyed production data — the credentials allowed it. The fix is least privilege through scoped, purpose-generated tokens carrying only READ and CREATE, so the same hallucinated delete simply fails, plus a tool allow-list and a human-in-the-loop confirmation before any destructive action. The distinction the question turns on is agency versus autonomy: scoping the token reduces the agent's agency (what it can do), which is the right primary control here; an approval gate would reduce its autonomy. Reducing permissions and requiring approval are different levers, and the durable lesson is to scope to the acceptable scope of impact, not to the intended functionality.

### Multiple-Response Knowledge Check

**Q1 (Select TWO):** A regulated lender is building a loan-disbursement workflow. It must apply explicit eligibility rules, pause for a human underwriter's approval before moving money, and produce a complete, retryable audit trail of every step. Which TWO design choices satisfy these requirements?
- A) AWS Step Functions Standard workflows
- B) AWS Step Functions Express workflows
- C) A `.waitForTaskToken` (Wait for Callback) task for the approval gate
- D) A fully autonomous Bedrock Agent with a disbursement action group and SESSION_SUMMARY memory
- E) A single prompt with the eligibility rules in the system message

**A: A, C** — Explicit rules, a mandatory human gate, and a fine-grained retryable trace are the signature of deterministic orchestration, which AWS Step Functions provides. The approval gate specifically requires the Wait for Callback service-integration pattern (`.waitForTaskToken`), which pauses the execution, emits a token, and resumes only on `SendTaskSuccess`/`SendTaskFailure`. The trap is B: Express workflows support only Request Response, so they cannot do Wait for Callback — a callback gate requires a **Standard** workflow, which is why A and C must be chosen together. D trades away the predictability and full audit trail a regulated, money-moving flow demands and could disburse without the gate. E cannot orchestrate a multi-step approval workflow at all.

**Q2 (Select TWO):** An "infrastructure operations" agent reaches AWS resources through an MCP server. During a clone-database task it hallucinated a "cleanup" step and deleted the source production database. Which TWO controls would have prevented the destructive action while still allowing the clone to succeed?
- A) Pass the invoking user's admin token through to the MCP server's downstream calls
- B) Issue a scoped, purpose-generated token carrying only READ and CREATE for the downstream calls
- C) Apply a tool allow-list so the agent can reach only the clone-related operations
- D) Increase the agent's maximum-iterations stopping condition
- E) Add Cross-Region Inference so the model has burst capacity

**A: B, C** — This is excessive agency (OWASP LLM06) compounded by credential propagation. A scoped token carrying only READ and CREATE (never DELETE) makes the same hallucinated delete simply fail because the credentials no longer allow it — this caps excessive permissions. A tool allow-list of only the clone operations caps excessive functionality, removing any destructive tool from what the model can even select. A is the named trap and is the cause of the incident, not a fix — reusing the user's broad token is exactly why the delete succeeded. D bounds runaway loops (cost/latency) but does nothing to stop a single authorized destructive call. E addresses model availability, an unrelated axis.

**Q3 (Select TWO):** An architect is mapping out an agent platform and keeps confusing similarly named AWS capabilities. Which TWO statements are accurate as written?
- A) AWS Agent Squad is the managed, native multi-agent supervisor service on AWS
- B) Amazon Bedrock multi-agent collaboration is the managed, native multi-agent capability on AWS
- C) The AgentCore Gateway is the runtime managed MCP endpoint an agent calls to reach tools
- D) The AgentCore MCP Server is the runtime managed MCP endpoint an agent calls to reach tools
- E) Semantic, summarization, and episodic memory strategies are a native Bedrock Agents feature

**A: B, C** — The managed, native multi-agent feature on AWS is Amazon Bedrock multi-agent collaboration (B). AgentCore Gateway is the runtime managed MCP endpoint that turns APIs, Lambda, and existing MCP servers into agent-reachable tools behind one secured endpoint (C). The distractors are the guide's named traps: A is wrong because Agent Squad is an open-source framework (formerly the Multi-Agent Orchestrator, now at `2fastlabs/agent-squad`) that you operate yourself, not a managed service. D swaps the two AgentCore artifacts — the AgentCore MCP Server is a build-time open-source developer aid for coding assistants (Kiro, Q Developer, Claude Code), not the runtime endpoint. E is the memory trap: multi-strategy memory belongs to AgentCore Memory; native Bedrock Agents long-term memory is SESSION_SUMMARY only.

**Q4 (Select THREE):** A team wants to switch foundation models without redeploying code and to survive a single-Region model outage. Which THREE elements make up the correct fix for this Pattern 5 scenario?
- A) Invoke the model through the model-agnostic Converse API addressed by `modelId`
- B) Externalize the active model behind a Lambda plus AWS AppConfig indirection layer
- C) Use Cross-Region Inference for redundancy and burst capacity
- D) Hardcode a single model ID and Region in the agent for predictability
- E) Replace the agent with a single prompt to remove model coupling

**A: A, B, C** — The Pattern 5 fix has exactly these halves: Converse makes the invocation independent of which model answers (A); a Lambda plus AppConfig layer turns a model switch into a validated, roll-back-able configuration change rather than a redeploy (B); and Cross-Region Inference gives the reasoning core managed multi-Region redundancy and burst capacity (C). D is the trap itself — hardcoding one model ID or Region couples you to that model's availability and forces a redeploy to change anything. E does not address model switching or resilience and would discard needed agentic behavior. Remember the sub-trap that pairs with C: SCPs and IAM must permit model invocation in **every** destination Region, or one blocked Region fails the request.
